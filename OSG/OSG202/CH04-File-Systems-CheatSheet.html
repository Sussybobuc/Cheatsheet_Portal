File Operations

The OS provides a system call interface for file I/O. All operations go through the kernel.

  • open(path, flags, mode) — Returns a file descriptor; OS checks permissions & places entry in open-file table
  • close(fd) — Releases kernel resources; flushes kernel buffers if O_SYNC not set
  • read(fd, buf, n) — Copies up to n bytes from current file position into user buffer
  • write(fd, buf, n) — Copies n bytes from user buffer to file; may extend file size
  • lseek(fd, offset, whence) — Repositions file pointer; no actual I/O performed
  • unlink(path) — Removes directory entry; file data freed only when link count = 0
  • rename(old, new) — Atomic on same FS; cross-FS requires copy + delete
  • truncate(path, len) — Shrinks or extends file to exactly len bytes
  • fcntl(fd, cmd, arg) — Miscellaneous: get/set flags, advisory locking (F_SETLK)
  • mmap(addr, len, prot, flags, fd, off) — Maps file into process address space
Open File Table
Kernel maintains a per-process FD table → system-wide open file table (position, flags, ref-count) → vnode/inode table. Multiple FDs can share one open-file entry via dup() / fork().

Access Methods

Sequential Access

Sequential access reads/writes bytes in order from the beginning. The file pointer advances automatically after each operation. Efficient for streaming; used by compilers, grep, sort.

read(fd, buf, n) /* advances ptr by n */ write(fd, buf, n) /* advances ptr by n */ rewind(file) /* ptr = 0 */
Random Access

Random access (direct access) allows reading/writing at any byte position without scanning preceding bytes. Essential for databases.

lseek(fd, offset, SEEK_SET) /* absolute */ lseek(fd, offset, SEEK_CUR) /* relative */ lseek(fd, -offset, SEEK_END) /* from end */ New position = base + offset SEEK_SET → base = 0 SEEK_CUR → base = current position SEEK_END → base = file size
Indexed Access
Some systems add an index layer: an index file maps keys → byte offsets; main file holds variable-length records. Databases implement this via B-trees on top of the OS file.
Sparse Files
lseek past EOF then write creates a sparse file. Gaps are never allocated on disk but appear as NUL bytes. cp may un-sparse them; use cp --sparse=always.

Directory Structures

Single-Level (Flat)

All files in one directory. Simple; name collisions inevitable with multiple users. Used in early MS-DOS and embedded systems.

Two-Level

One master directory (MFD) → per-user directories (UFD). Isolates users; still no subdirectories. Naming like user/file.

Tree-Structured (Hierarchical)

Arbitrary-depth subdirectories form a tree rooted at /. Each directory entry maps name → inode. Used by all modern UNIX/Windows systems.

  • Each process has a current working directory (cwd) stored in its PCB
  • chdir(path) changes the cwd for the calling process
  • Leaf nodes = files; internal nodes = directories
DAG (Directed Acyclic Graph)

Allows shared subdirectories — multiple parents point to the same directory inode. More flexible than a pure tree.

⚠ DAG Cycle Risk
Hard links to directories can introduce cycles, breaking tools like find and du. UNIX forbids hard links to directories (except . and ..) for this reason. Symbolic links can still create cycles; find -L uses a depth limit to guard against them.

File System Layout on Disk

A disk is divided into partitions. Each partition can hold an independent file system. The overall disk structure (simplified UNIX ext2-style):

Sector 0: [ MBR — Master Boot Record ] Contains: bootstrap code (446 B) Partition table (4×16 B) Magic signature 0x55AA ───────────────────────────────────── Partition: [ Boot Block ] ← sector 0 of partition [ Superblock ] ← FS metadata [ Free Bitmap ] ← 1 bit per block [ I-node Table ] ← array of inodes [ Root Dir ] ← inode #2 on ext2 [ Data Blocks ] ← file/dir content
MBR vs GPT
MBR (Master Boot Record) supports max 4 primary partitions and 2 TB disks. GPT (GUID Partition Table) supports 128 partitions and 9.4 ZB disks. UEFI systems use GPT with a protective MBR for compatibility.
MBR — Master Boot Record

Located at sector 0 of the disk. BIOS loads MBR into RAM at address 0x7C00 and jumps to it. The bootstrap code reads the partition table to find the active partition and loads its boot block.

ComponentSizePurpose
Bootstrap code446 bytesFinds & loads boot sector of active partition
Partition table64 bytesFour 16-byte entries: start, size, type, flags
Magic number2 bytes0x55 0xAA — BIOS validity signature
Superblock

The superblock is the most critical metadata structure. Corruption = FS unmountable. Multiple backup copies kept at fixed offsets.

  • s_magic — identifies FS type (ext2: 0xEF53)
  • s_blocks_count — total blocks in FS
  • s_inodes_count — total inodes allocated
  • s_block_size — bytes per block (1K, 2K, 4K)
  • s_free_blocks_count — free data blocks
  • s_free_inodes_count — free inodes
  • s_mtime — last mount time (used by fsck)
Block Groups (ext2/3/4)
ext2 divides the disk into block groups. Each group has its own superblock copy, block bitmap, inode bitmap, inode table, and data blocks. This localises related data and improves cache hit rate.

Contiguous Allocation

Each file occupies a contiguous run of disk blocks. The directory entry stores only the start block and length.

/* Directory entry */ struct cont_entry { char name[14]; int start; /* first block number */ int length; /* number of blocks */ }; /* Access block i of file */ block_addr = start + i;
Advantages
Sequential read is maximally fast — one seek, then streaming reads. Random access is O(1): compute start + i, seek once, read.
External Fragmentation
As files are created and deleted, free space is split into small holes. A large file cannot fit even if total free space is sufficient. Compaction fixes this but requires copying the entire disk — prohibitively expensive. Used on CD-ROMs (read-only, no deletion) and some DVDs where fragmentation is impossible.
Rule: Contiguous allocation requires knowing file size at creation time. Not practical for files that grow dynamically.

Linked List Allocation

Each disk block contains a pointer to the next block. Directory entry stores only the first block number. No external fragmentation — any free block can be appended.

/* Each block on disk */ struct linked_block { uint32_t next; /* next block# or -1 */ uint8_t data[BSIZE - 4]; /* payload */ }; /* Directory entry */ struct dir_entry { char name[14]; uint32_t first_block; };
Random Access is O(n)
To read block i, you must follow i pointers sequentially from the start — each requiring a disk seek. Databases that need random access cannot tolerate this.
Space Overhead
The pointer steals bytes from each block. With a 512-byte block and 4-byte pointer, only 508 bytes of actual data per block. If a record spans two blocks, two seeks are needed to read it — breaking the concept of a "block".
Rule: Linked allocation works well for sequential access only. Reliability is poor — one corrupt pointer breaks the chain for all subsequent data.

FAT — File Allocation Table

FAT is linked-list allocation moved into a separate table kept entirely in RAM. Each FAT entry stores the next block number for that block. The entire chain for any file can be traversed in memory without disk seeks.

FAT (in RAM — indexed by block number): Block: 0 1 2 3 4 5 6 FAT: [ - | 3 | - | 6 | - | 2 | EOF ] ↑ ↑ file A start file A continues File A: block 1 → block 3 → block 6 → EOF File B: block 5 → block 2 → EOF
  • Directory entry stores starting block only — FAT handles the chain
  • Free blocks marked with 0; end-of-file with special value (0xFFFF, 0x0FFFFFFF)
  • Random access: traverse chain in RAM (fast), then one disk seek for actual data
  • FAT12 (12-bit) → max 4,096 clusters; FAT16 → 65,536; FAT32 → 268 million
RAM Requirement
A 200 GB FAT32 disk with 512-byte clusters needs a 200 MB FAT resident in RAM at all times. This is why large drives use larger cluster sizes (4–64 KB), reducing FAT size at the cost of internal fragmentation.
Variants
FAT12 — floppy disks · FAT16 — small hard disks · FAT32 — USB drives, SD cards · exFAT — large flash storage (no 4 GB file limit)

I-node Structure

An i-node (index node) stores all file metadata plus block addresses. Only the inodes of open files need to be in memory simultaneously — far more scalable than FAT.

/* Simplified UNIX inode (on disk) */
struct inode {
  uint16_t mode;        /* type + rwx bits     */
  uint16_t nlinks;      /* hard link count      */
  uint16_t uid, gid;    /* owner IDs            */
  uint32_t size;        /* bytes in file        */
  uint32_t atime;       /* last access          */
  uint32_t mtime;       /* last modification    */
  uint32_t ctime;       /* last inode change    */
  uint32_t direct[12];  /* 12 direct block ptrs */
  uint32_t single_ind;  /* → block of ptrs      */
  uint32_t double_ind;  /* → block of blk ptrs  */
  uint32_t triple_ind;  /* → 3 levels of ptrs   */
};
Key Design Insight
Small files (the vast majority) need only the 12 direct pointers — no indirection overhead. Large files pay the indirection cost only when needed. The inode itself is typically 128 or 256 bytes.
Block Pointer Levels
Assume: block_size = 1 KB ptr_size = 4 bytes ptrs/block = 1024/4 = 256 Direct (×12): 12 × 1 KB = 12 KB Single Indirect: 256 × 1 KB = 256 KB Double Indirect: 256² × 1 KB = 64 MB Triple Indirect: 256³ × 1 KB = 16 GB ───────────────────────────────────────────── Max file size ≈ 16 GB (1 KB blocks) With 4 KB blocks (ptrs/block = 1024): Direct: 12 × 4 KB = 48 KB Single: 1024 × 4 KB = 4 MB Double: 1024² × 4 KB = 4 GB Triple: 1024³× 4 KB = 4 TB
Inode Number vs File

A file is identified by its inode number, not its name. The directory only maps names → inode numbers. Delached/renamed files retain their inode.

Inode count is fixed at mkfs time. Running out of inodes (df -i) prevents new files even with free disk space.
In-Memory Inode
When a file is opened, the on-disk inode is loaded into a kernel in-memory inode which adds: reference count, dirty flag, lock, device ID, and a pointer to associated page-cache pages.

Fixed-Size Directory Entries

Classic UNIX v7 used fixed 16-byte directory entries: 14 bytes for filename + 2 bytes for inode number. Simple to implement, easy to binary-search.

/* UNIX v7 directory entry (16 bytes) */ struct dir_v7 { uint16_t inode; /* 2 bytes: inode # */ char name[14]; /* 14 bytes: filename */ }; /* inode==0 means entry is deleted/free */

Max Directory Entries
With 1 KB blocks: 1024 / 16 = 64 entries per block. A directory file is just a list of these 16-byte records; it grows by allocating new blocks like any other file.

Variable-Length Directory Entries

Modern FS (ext2+, NTFS) use variable-length entries to support long filenames efficiently. Each entry has a length field so the next entry can be found.

/* ext2-style directory entry */ struct ext2_dir_entry { uint32_t inode; /* inode number */ uint16_t rec_len; /* total entry length */ uint8_t name_len; /* length of name */ uint8_t file_type; /* regular/dir/symlink… */ char name[]; /* name (not NUL-term'd)*/ }; /* Entries are 4-byte aligned; padding fills gap */
FieldSizeNotes

Deletion Creates Holes
When an entry is deleted, its rec_len is merged into the preceding entry's rec_len, absorbing the hole. The next create may split a hole if the name fits.
Hash Trees (dir_index)
ext3/4 optionally organise directory entries as an HTree (half-B-tree), turning filename lookup from O(n) linear scan into O(log n). Enabled with tune2fs -E hash_alg.

Hard Links vs Symbolic Links

Hard Links

A hard link is simply a new directory entry pointing to an existing inode. The inode's nlinks counter increments. The file's data is not duplicated.

ln original.txt hardlink.txt /* Both entries → same inode number */ /* inode.nlinks == 2 now */ /* rm original.txt → nlinks == 1 */ /* Data freed only when nlinks == 0 */
  • Hard links cannot span different file systems (inode numbers are per-FS)
  • Cannot hard-link directories (would create cycles — only . and .. are exceptions)
  • Deleting the "original" leaves all other hard links fully valid
Symbolic (Soft) Links

A symlink is a special file whose data is a path string. The kernel follows it transparently on every access.

ln -s /home/alice/file.txt link.txt /* Symlink file contains: /home/alice/file.txt */ /* Crossing FS boundaries: YES */ /* Linking directories: YES (with care) */
Dangling Symlink
If the target path is deleted or moved, the symlink dangles — it exists but every access returns ENOENT. Hard links cannot dangle because they hold a reference to the inode directly.
Fast Symlinks
If the target path is ≤ 60 characters, many FS (ext2+) store it inline in the inode itself (in the block pointer array space), avoiding a separate disk block entirely.

Log-Structured File System (LFS)

LFS treats the entire disk as a circular log. All writes (data AND metadata) are buffered in a memory segment and written as one large sequential operation — eliminating random write I/O.

  1. Writes accumulate in a write buffer (segment, typically 512 KB–1 MB)
  2. Full segment flushed to disk as one sequential write at the log tail
  3. Each segment has a segment summary at its start listing what's inside
  4. New inode written into segment; inode map updated to record new location
  5. Cleaner thread runs in background: finds old segments with live data, copies live data to new segment, marks old segment as free
Disk layout (circular): [ Seg 0 ][ Seg 1 ][ Seg 2 ][ ... ][ Seg N ] ↑ Clean point ↑ Log tail (write here) Inode Map: inode# → current disk address Checkpoint: periodic flush of inode map to fixed location
Why LFS Wins on Writes
Traditional FS: create a file = 5+ seeks (inode, bitmap, data, directory, updated inode). LFS: all of those go into the same segment = 1 sequential write. Disk write throughput approaches raw disk bandwidth.
Read Performance
Fresh files read sequentially — fast. After cleaning, old file data may be scattered around the log, causing seeks. LFS works best for write-heavy workloads (web logs, databases).

Journaling File Systems

A journal (write-ahead log) records pending operations before performing them. On crash, the journal is replayed to reach a consistent state — no need for full fsck.

  1. Write a begin-transaction record to the journal
  2. Write all changed blocks (metadata, optionally data) to the journal
  3. Write a commit record to the journal (operation is durable once this lands)
  4. Write blocks to their actual locations on disk (the "real" write)
  5. Mark the journal space as free (transaction complete)
Journal modes (ext3/4): journal — data + metadata logged (safest, slowest) ordered — data written before metadata logged (default) writeback — metadata only; data may predate metadata (fastest)
File SystemJournalNotes
ext3Yes (3 modes)Backward compatible with ext2

NTFSYes (MFT log)Windows default since NT 3.1

Rule: Operations must be idempotent — replaying a committed transaction twice must leave the FS in the same state as replaying it once. This is why journal entries describe what the final state should be, not just the delta.

Virtual File System (VFS)

VFS is an abstraction layer in the kernel that presents a uniform POSIX interface to user processes regardless of the underlying concrete file system (ext4, NTFS, NFS, tmpfs…).

Dentry Cache
The dcache maps (parent inode, name) → child inode in RAM, avoiding repeated directory lookups. It is one of the most effective kernel caches — a cache hit means no disk I/O for path resolution.
Adding a New FS
Implement the file_operations, inode_operations, and super_operations function tables and register with register_filesystem(). The VFS dispatches all calls through these tables.

Block Size Trade-off

The block size chosen at mkfs time fundamentally affects both performance and space efficiency. It cannot be changed without reformatting.

Efficiency(b) = file_size / ceil(file_size / b) × b Small blocks (512 B): + Low internal fragmentation - Many seeks for large files - Large inode indirect chain Large blocks (64 KB): + Fast streaming I/O, few seeks - Severe waste for small files (a 1-byte file uses 64 KB on disk!)
Median File Size Stat
Studies of real-world FS show the median file size ≈ 2,475 bytes (Tanenbaum, 4th Ed.). With 4 KB blocks (typical default), the average waste per file is nearly 50% of the last block. Yet 4 KB is the near-universal default because disk bandwidth benefits outweigh wasted space for most workloads.

Free Space Management

The OS must track which disk blocks are free for allocation. Two primary methods:

Bitmap (Bit Vector)

One bit per block: 0 = free, 1 = allocated. Very compact — a 100 GB disk with 4 KB blocks needs only ~3 MB bitmap.

/* Find first free block */ for (i = 0; i < num_blocks/32; i++) { if (bitmap[i] != 0xFFFFFFFF) { bit = ffs(~bitmap[i]) - 1; /* find first 0 bit */ return i*32 + bit; } } /* Hardware popcount/BSF instructions make this fast */
Bitmap Advantage
Supports contiguous block search efficiently using bit manipulation. Modern CPUs can scan 64 bits at once. The bitmap for an entire disk fits in RAM, enabling fast kernel-side allocation decisions.
Free List (Linked Blocks)

Free blocks are chained together — each free block stores the address of the next free block. Only one block of the list needs to be in RAM.

  • Low memory overhead — only one block in RAM
  • Poor performance: allocating n blocks requires reading n disk blocks
  • Cannot easily find contiguous free regions
  • Used when RAM is severely constrained
Rule: Bitmap preferred for general-purpose FS. Free list used only when memory is extremely limited (early Unix, embedded systems).

Disk Quotas

Disk quotas limit how much disk space and how many files each user (or group) can consume, preventing any single user from filling the disk.

Limit TypeBehaviourEnforcement
Soft limitWarning issued when exceededGrace period (e.g., 7 days); then hard-enforced

Backup Types

Physical Dump

Copies the disk block by block, starting from block 0, regardless of whether the block belongs to a file. Simple, fast, but cannot skip free blocks or selected directories.

Physical dump sequence: for block = 0 to disk_size/block_size: read block → write to backup medium (includes free blocks, bad blocks, everything)
Physical Dump Limitations
Cannot back up a live (mounted) FS safely — the FS may be mid-write. Usually done from rescue media with the FS unmounted. Cannot easily restore individual files — must restore the whole volume.
Logical Dump

Traverses the directory tree and backs up files selectively. The Unix dump / restore utilities implement this. Skips free blocks and optionally excludes directories.

Full BackupIncremental Backup
What is savedEvery file, alwaysOnly files changed since last backup
Storage usedHighLow
Backup speedSlowFast
Restore speedFast (one tape)Slow (multiple tapes)
Restore complexitySimpleMust apply in order
Recommendation
Use a full backup weekly + incremental daily. To restore, apply the last full backup first, then each daily incremental in order. Store backups off-site or on a separate device — a fire that destroys the server also destroys on-site backups.

Backup Considerations

Effective backup requires careful thought about what to back up, how to detect changes, and how to restore.

Detecting Changed Files
  • mtime comparison — back up if mtime > last_backup_time. Simple but timezone/clock drift causes issues
  • Archive bit — a flag in each file's metadata, cleared on backup, set on modification (Windows / FAT approach)
  • Checksums — cryptographic hash of file content; detect changes and corruption but expensive to compute
  • Snapshot (CoW) — LVM / ZFS / Btrfs take an instant filesystem snapshot; back up the snapshot for a consistent view
Special Files to Handle Carefully
Problem Files
Hard links: multiple names for same inode — dump must not duplicate the data. Dump tracks all backed-up inodes and writes data only once.
Special files (/dev/*): should never be backed up as data — restore the device node with correct major/minor numbers.
Holes in sparse files: must be restored as holes, not as actual NUL bytes, or the restored file will be much larger.
Incremental dump levels (traditional UNIX): Level 0: Full dump of entire FS Level 1: All changes since Level 0 Level 2: All changes since Level 1 ... Level 9: All changes since Level 8 Restore: apply 0, then 1, 2, ... in sequence
Modern Tools
rsync -avz --link-dest=PREV src/ backup/ — incremental with hard-link dedup
tar -czf backup.tar.gz --newer=TIMESTAMP dir/
zfs send / btrfs send — send FS snapshots efficiently

UNIX v7 File System Layout

The original UNIX v7 FS (1979) established the disk layout pattern that influenced virtually all subsequent Unix-derived file systems.

Disk partition layout: ┌─────────────┬────────────┬────────────────┬──────────────────────┐ │ Boot Block │ Superblock │ I-node Table │ Data Blocks │ │ (1 block) │ (1 block) │ (fixed # inod) │ (rest of partition) │ └─────────────┴────────────┴────────────────┴──────────────────────┘ Boot Block: may contain bootstrap loader or be empty Superblock: FS size, inode count, free block list head, magic number, mount state I-node table: fixed array; inode 1 = bad blocks, 2 = root dir Data Blocks: actual file content + directory content
ComponentSizePurpose
Boot block512 BOS bootstrap code (or unused)
Superblock512 BFS global metadata + free list
Inode tableN × 64 BFixed at mkfs; inode #1 = bad, #2 = root

UNIX v7 Inode Size
The original v7 inode was 64 bytes with 13 block pointers: 10 direct + 1 single indirect + 1 double indirect + 1 triple indirect. With 512-byte blocks and 16-bit addresses, max file size ≈ 1 GB.
Evolution
BSD FFS (1984) added block groups, larger blocks (up to 8 KB), fragments for tail packing. ext2 (1993) added 32-bit block numbers. ext4 (2008) added extents, 48-bit block numbers, and delayed allocation.

File Descriptor Table Chain

Each open() call threads through three kernel tables. Understanding this chain is essential for fork(), dup(), and close() semantics.

Per-process FD table (in task_struct.files): fd[0] → [open file entry A] ← stdin fd[1] → [open file entry B] ← stdout fd[2] → [open file entry C] ← stderr fd[3] → [open file entry D] ← user file System-wide Open File Table: entry A: f_pos=0, f_flags=O_RDONLY, f_count=1, *vnode→inode X entry D: f_pos=512, f_flags=O_RDWR, f_count=1, *vnode→inode Y Vnode / In-memory Inode Table: inode X: disk inode 42, ref_count=1, dirty=0, ... inode Y: disk inode 87, ref_count=1, dirty=1, ...
  1. fork() — child gets a copy of parent's FD table; both FDs point to the same open file entry → shared file position. f_count incremented.
  2. dup(fd) — creates new FD in same process pointing to same open file entry → same position. f_count++.
  3. open() — always creates a new open file entry with its own position, even if same file opened twice.
  4. close(fd) — decrements f_count; entry freed when 0. Inode ref_count decremented; inode freed when 0 AND nlinks==0.
POSIX File Locking Quirk
fcntl() advisory locks are per-process, not per-FD. If a process opens the same file twice and holds a lock via fd[3], closing fd[4] releases the lock on fd[3] too — a well-known POSIX "surprise".
