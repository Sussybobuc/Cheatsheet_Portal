What Is an Operating System?

OS = Extended Machine + Resource Manager

The OS sits between the hardware and user applications, providing two key abstractions:

Extended Machine View
  • Hides messy hardware details behind clean abstractions
  • Provides processes, files, sockets instead of raw disk sectors and registers
  • Top-down view: OS as convenience provider
Resource Manager View
  • Manages CPU time, memory, I/O devices, storage
  • Multiplexes resources: time multiplexing (CPU scheduling) and space multiplexing (memory partitioning)
  • Bottom-up view: OS as traffic cop
Key Insight
The OS runs in kernel mode (supervisor mode) with full hardware access. User programs run in user mode — restricted. Switching between them is done via system calls (traps).

Kernel Mode vs User Mode

PropertyKernel ModeUser Mode
Access to hardwareFull, unrestrictedNone (via OS only)
Execute privileged instructionsYesNo (trap/fault)
Access all memoryYesOwn address space only
Who runs hereOS kernelApplications, shells
Error consequenceSystem crashProcess killed
System Call Mechanism
User program places args in registers → executes TRAP instruction → CPU switches to kernel mode → OS handles request → returns result → CPU switches back to user mode.
User → TRAP → Kernel → execute → return → User

Why Abstractions Matter
Without abstractions, every application would need to know disk geometry, CPU scheduling policies, and interrupt handling — making software portability impossible.

History of Operating Systems — Five Generations

Gen 1: 1945–55
Vacuum Tubes
  • No OS — machine programmed in machine language or plug boards
  • One job at a time, operator is programmer
  • ENIAC, UNIVAC era
Gen 2: 1955–65
Transistors & Batch
  • Batch systems: jobs collected on cards, run sequentially
  • FMS (Fortran Monitor System), IBSYS
  • Operator loads job decks
Gen 3: 1965–80
ICs & Multiprogramming
  • Multiprogramming: multiple jobs in memory; CPU switches to keep busy during I/O
  • Timesharing: multiple interactive users (CTSS, MULTICS)
  • IBM OS/360 — one OS for all models
Gen 4: 1980–present
Personal Computers
  • PC revolution: CP/M, DOS, Windows, macOS
  • GUI interfaces (Xerox Alto → Apple Mac → Windows)
  • UNIX → Linux → Android/iOS
Gen 5: Now
Mobile & Cloud
  • Distributed OS, cloud computing, VMs
  • Android, iOS dominate mobile
  • Containers: Docker, Kubernetes
Key Insight: Why Multiprogramming?
CPU is fast, I/O is slow. In batch, CPU sat idle waiting for tape/disk. Multiprogramming keeps CPU busy by switching to another job while the first waits for I/O — crucial efficiency gain.

CPU Architecture

Key CPU Registers
RegisterPurpose
PCProgram Counter — next instruction address
SPStack Pointer — top of current stack
PSWProgram Status Word — flags, mode bit, condition codes
MARMemory Address Register — address for memory op
MDRMemory Data Register — data being read/written
IRInstruction Register — current instruction
Pipeline
Modern CPUs use pipelining: fetch, decode, execute overlap. Superscalar CPUs execute multiple instructions per cycle. Hyperthreading presents 2 logical CPUs per physical core.
Cache Hierarchy
L1 (fastest, ~1ns, 32-64KB) → L2 (~5ns, 256KB-1MB) → L3 (~20ns, 4-32MB) → RAM (~60ns) → Disk (~10ms)

I/O, Interrupts & DMA

Three I/O Methods
  1. Programmed I/O (Busy Wait) — CPU polls device constantly, wastes cycles. while(!done) check();
  2. Interrupt-Driven I/O — CPU starts I/O, does other work. Device fires interrupt when done. OS runs interrupt handler via interrupt vector.
  3. DMA (Direct Memory Access) — DMA controller transfers data between device and RAM without CPU. CPU sets up DMA chip (count, address, direction) and is interrupted only when done.
⚠ Interrupt Handling Steps
1. Device asserts interrupt line → 2. CPU finishes current instruction → 3. Saves PC + PSW on stack → 4. Looks up handler in interrupt vector → 5. Handler runs → 6. Returns, restores PC/PSW
Interrupt Vector[device_number] → handler_address

Memory Hierarchy & Buses

Memory Hierarchy (speed ↑, size ↓)
Registers → Cache (L1/L2/L3) → RAM → SSD → HDD → Tape
Modern Bus Architecture
  • PCIe — main bus; serial, point-to-point, replaces parallel PCI; 16x PCIe 3.0 = 128 Gbps
  • DMI — CPU to Platform Controller Hub (south bridge)
  • USB — slow peripherals; USB 3.0 = 5 Gbps; hot-pluggable
  • SATA — hard disks & SSDs via hub
  • DDR3/4 — CPU to RAM; dedicated high-speed bus
Boot Process
  1. Power on → BIOS in Flash ROM runs, checks RAM, scans PCIe/PCI devices
  2. BIOS reads MBR from boot device; executes boot loader
  3. Boot loader loads OS kernel into RAM
  4. Kernel initializes tables, loads device drivers, starts processes
  5. Login program / GUI starts

Monolithic & Layered Kernels

Monolithic Kernel

Monolithic kernels run the entire OS as a single large program in kernel space. All OS services share the same address space.

  • Pros: fast — no user/kernel switches for internal calls
  • Cons: one bug can crash entire system; hard to maintain
  • Examples: Linux, early UNIX, MS-DOS
  • Can use loadable kernel modules (LKMs) for extensibility
Layered Kernel (THE System)

Layered design: OS divided into N layers, each built on top of the one below. Layer 0 = hardware, Layer N = user interface.

  • Pros: clean separation; easier to debug layer by layer
  • Cons: defining clean layer boundaries is hard; performance overhead
  • THE system (Dijkstra, 1968) — first layered OS

Microkernel & Client-Server

Microkernel = minimal kernel; everything else as user-space servers
What lives in the microkernel?
  • IPC (message passing between processes)
  • Basic memory management
  • Basic CPU scheduling
What moves to user space?
  • File systems, device drivers, network stacks, window managers
  • Run as normal user-space processes
Microkernel Pros & Cons
✅ Reliability: driver crash doesn't kill OS
✅ Security: smaller attack surface
❌ Performance: user↔kernel switches for every message
Examples: MINIX 3, QNX, L4, Mach
Client-Server Model

Variant of microkernel: OS services are servers; apps are clients sending requests via message passing. Works across networks for distributed OS.

Virtual Machines & Hypervisors

VM = exact copy of underlying hardware exposed to guest OS
Type 1 Hypervisor (Bare Metal)
  • Runs directly on hardware; no host OS
  • Examples: VMware ESXi, Xen, Hyper-V
  • Each guest OS thinks it owns the machine
  • Better performance than Type 2
Type 2 Hypervisor (Hosted)
  • Runs on top of a host OS as an application
  • Examples: VirtualBox, VMware Workstation
  • Slower but easier to install/use
Java Virtual Machine
JVM is a different kind of VM — compiles Java bytecode; provides portability, not hardware emulation. "Write once, run anywhere."
Containers vs VMs

Containers (Docker) share the host OS kernel; much lighter than VMs. VMs each have their own OS.

System Call Execution in C
/* fork() + exec() pattern — how shell runs commands */
pid_t pid = fork();          /* create child */
if (pid == 0) {
    /* child process */
    execve("/bin/ls", argv, envp);   /* replace with ls */
    exit(1);                          /* exec failed */
} else {
    /* parent */
    wait(NULL);              /* wait for child to finish */
}

⚠ System calls are slow
Each syscall involves a mode switch (user→kernel→user), saving/restoring registers, and looking up handlers. Minimize unnecessary syscalls in tight loops.

Process Concepts Overview

Process = Program in Execution = Program + State + Resources
What a Process Contains
  • Address space: code (text), data, heap, stack
  • Process table entry (PCB): registers, open files, signals, memory maps
  • Resources: open file descriptors, pending alarms, child processes
Process Hierarchy (UNIX)

UNIX processes form a tree. init (PID 1) is the root — ancestor of all user processes. Parent-child relationship via fork(). Child inherits parent's UID.

Signals

Signals are software interrupts sent to processes. Handler runs briefly, then process resumes. Common signals:

SignalCause
SIGKILLUnconditional kill (cannot catch)
SIGSEGVSegmentation fault (invalid memory)
SIGALRMTimer expired (from alarm())
SIGCHLDChild process terminated
SIGINTCtrl+C from terminal

Files & File System Concepts

File = named sequence of bytes — OS provides uniform I/O interface
File Descriptor

An integer file descriptor (fd) identifies an open file. Inherited by child processes via fork().

fdNameDefault
0stdinKeyboard
1stdoutTerminal
2stderrTerminal
Directory Structure
  • Hierarchical: root / → subdirectories → files
  • Path names: absolute (/usr/bin/ls) or relative (../lib)
  • . = current dir, .. = parent dir
Special Files
UNIX unifies I/O: block special files (disks), character special files (keyboard, printer), pipes, and sockets — all accessed via read()/write().
Pipe Example
ls -l | grep ".c" | wc -l
# Pipeline: stdout of ls → stdin of grep → stdin of wc

Security — UIDs, GIDs & Protection

Every process has a UID and GID — determines access rights
User & Group IDs
  • UID (User ID): integer assigned by sysadmin. Process inherits parent's UID.
  • GID (Group ID): users belong to groups for shared file access
  • Superuser (root, UID=0): bypasses all protection checks — full system access
File Permissions (9 bits)
rwxrwxrwx = owner|group|others
BitOctalMeaning on file
r (read)4Read file contents
w (write)2Modify file
x (execute)1Run as program
setuid Bit

When setuid bit is set on executable, it runs with the file owner's UID, not the caller's. Used by passwd to modify /etc/shadow as root.

⚠ Security Risk
Setuid programs are prime targets for exploits — a bug in a setuid-root program can give attacker root access.

The Shell & I/O Redirection

Shell as Command Interpreter

The shell is a regular user-space program (not part of OS). It reads commands, forks child processes, and waits for them. Shell types: sh, bash, zsh, csh.

I/O Redirection Examples
ls > output.txt        # redirect stdout to file
cat < input.txt        # redirect stdin from file
ls 2> errors.txt       # redirect stderr
ls > out.txt 2>&1     # both stdout and stderr to file
sort < words | uniq    # pipe with input redirect
Background Processes
./longprogram &        # run in background (shell doesn't wait)
jobs                   # list background jobs
fg %1                  # bring job 1 to foreground
How Shell Runs a Command
1. Read line from terminal → 2. fork() → 3. Child: exec(command) → 4. Parent: wait() → 5. Return to prompt

OS Design Principles & Trade-offs

Key Design Principles
  • Policy vs Mechanism — separate what to do (policy) from how to do it (mechanism). Enables flexibility without changing implementation.
  • Portability — write OS in C, not assembly; isolate machine-dependent code
  • Performance — avoid unnecessary work; batch similar operations; use caching
  • Simplicity — simple interfaces even if complex implementation
Common OS Trade-offs
Trade-offOption AOption B
SafetyUser mode (isolation)Kernel mode (performance)
ComplexityMonolithic (fast)Microkernel (reliable)
FairnessRound-robin schedulingPriority scheduling
SpaceKeep in memorySwap to disk
POSIX Standard
POSIX (Portable Operating System Interface) defines a standard API for UNIX-like OS. Programs written to POSIX run on Linux, macOS, BSDs without changes.
