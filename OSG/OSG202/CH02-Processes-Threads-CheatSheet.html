Process Model & PCB

Process = Program in Execution. One CPU → many processes via multiprogramming illusion.
Process Control Block (PCB)

The PCB (process table entry) stores all info needed to restart a suspended process:

PCB FieldContents
Process stateRunning / Ready / Blocked
Program CounterAddress of next instruction
CPU registersAll register values at suspension
Memory infoBase/limit regs, page table ptr
Scheduling infoPriority, quantum used, arrival time
I/O statusOpen files, pending I/O operations
AccountingCPU time used, wall time
Core Image
A suspended process = its core image (address space snapshot) + its PCB. The OS saves PCB in the process table (array of PCBs, one per process).

Process States & Transitions

Three Core States
StateMeaning
RunningCurrently using the CPU
ReadyRunnable, waiting for CPU
BlockedWaiting for I/O or event — cannot run even if CPU free
State Transitions (4 possible)
  1. Running → Blocked: process requests I/O or waits for event (read(), wait())
  2. Running → Ready: scheduler preempts process (quantum expired)
  3. Ready → Running: scheduler picks this process (dispatched)
  4. Blocked → Ready: I/O completes, event occurs — process becomes runnable again
⚠ Note
A blocked process can never go directly Running → it must go Blocked → Ready → Running.
Running ↔ Ready ↔ Blocked (Blocked→Ready only when event fires)

Process Creation & Termination

Process Creation — fork() + exec()
pid_t pid = fork();
/* fork() returns TWICE:
   - in parent: returns child's PID (>0)
   - in child:  returns 0
   Both share identical copy of parent's address space (copy-on-write) */

if (pid == 0) {
    /* child */
    execve("/usr/bin/gcc", args, env);
} else {
    /* parent */
    int status;
    waitpid(pid, &status, 0);
}
Termination Causes
  • Normal exit: exit(0) — voluntary, task complete
  • Error exit: exit(1) — voluntary, error detected
  • Fatal error: divide by zero, segfault → OS kills process
  • Killed by another: kill(pid, SIGKILL)
Zombie & Orphan
Zombie: child exited but parent hasn't called wait(). Orphan: parent died first — orphan adopted by init (PID 1).

Pipe Example
int fd[2];
pipe(fd);       /* fd[0]=read end, fd[1]=write end */
if (fork() == 0) {
    /* child writes */
    close(fd[0]);
    write(fd[1], "hello", 5);
    exit(0);
} else {
    /* parent reads */
    close(fd[1]);
    char buf[10];
    read(fd[0], buf, 5);
    printf("%s\n", buf);
}
Message Passing Semantics
  • Blocking send: sender waits until message delivered
  • Non-blocking send: sender continues immediately
  • Blocking receive: receiver waits for message (mailbox)
  • Rendezvous: both sender and receiver must be ready

Race Conditions

Race Condition: final result depends on exact timing/order of concurrent processes.
Classic Example — Print Spooler
  1. Process A reads in pointer = 7 (next free slot)
  2. Clock interrupt → scheduler switches to Process B
  3. B also reads in = 7, stores its file name in slot 7, sets in = 8
  4. A resumes, writes its filename to slot 7 (overwrites B!), sets in = 8
  5. B's file is lost — B never gets printed
⚠ Why Races Are Insidious
Race conditions are nondeterministic: they appear rarely, are hard to reproduce, and almost impossible to debug by inspection. They become more common as CPU core counts increase.
Four Conditions for a Good Solution
  1. No two processes simultaneously in their critical region
  2. No assumptions about CPU speed or number of CPUs
  3. No process outside critical region may block another
  4. No process waits forever to enter critical region (no starvation)

Busy Waiting Solutions

Strict Alternation (Fails condition 3)
/* Process 0: */
while (TRUE) {
    while (turn != 0);  /* spin wait */
    critical_region();
    turn = 1;
    noncritical_region();
}
/* Problem: if P0 is fast and P1 slow,
   P0 must wait even though P1 is in
   NON-critical region. Violates #3. */
Peterson's Algorithm (Correct!)
#define N 2
int turn, interested[N];  /* all init 0 */

void enter_region(int process) {
    int other = 1 - process;
    interested[process] = TRUE;
    turn = process;
    /* wait if other wants in AND it's their turn */
    while (turn == process &&
           interested[other] == TRUE);
}
void leave_region(int process) {
    interested[process] = FALSE;
}
Why Peterson Works
If both call enter_region simultaneously, last write to turn wins — that process waits, giving the other access. Satisfies all 4 conditions.

Hardware Solutions: TSL & XCHG

TSL = Test and Set Lock — atomic read-modify-write instruction
TSL Instruction
/* TSL RX, LOCK:
   Atomically: RX = LOCK; LOCK = 1
   Locks memory bus during execution — no other
   CPU can access LOCK between read and write */

enter_region:
    TSL REGISTER, LOCK  ; copy lock to reg, set lock=1
    CMP REGISTER, #0    ; was lock zero?
    JNE enter_region    ; if not, loop (spin)
    RET                 ; in critical region

leave_region:
    MOVE LOCK, #0       ; set lock to 0
    RET
XCHG Alternative (x86)
/* XCHG atomically swaps register with memory */
enter_region:
    MOVE REGISTER, #1   ; set register to 1
    XCHG REGISTER, LOCK ; swap (atomic)
    CMP REGISTER, #0    ; old value = 0?
    JNE enter_region    ; no: spin
    RET
⚠ Busy Waiting Wastes CPU
Spin locks waste CPU time. Only acceptable when: (1) expected wait is very short, (2) on multiprocessor where another CPU holds the lock. Use sleep/wakeup for longer waits.

Sleep/Wakeup & Semaphores

Sleep & Wakeup Primitives

sleep() blocks the calling process until another calls wakeup() on it.

⚠ Lost Wakeup Problem
If wakeup() arrives before sleep(), the signal is lost and the process sleeps forever. This race condition motivated semaphores.
Semaphore (Dijkstra, 1965)
Semaphore = integer counter + sleeping queue. Two atomic ops: P() (down/wait) and V() (up/signal)
/* Semaphore operations — MUST be atomic */
void down(semaphore *s) {   /* P() / wait() */
    s->count--;
    if (s->count < 0) {
        /* add to sleep queue, block */
        sleep();
    }
}
void up(semaphore *s) {     /* V() / signal() */
    s->count++;
    if (s->count <= 0) {
        /* wake one sleeping process */
        wakeup(s->queue.front());
    }
}
Semaphore Types
TypeInit ValueUse
Binary (Mutex)1Mutual exclusion
CountingNCount available resources

Mutex & Monitor

Mutex (Binary Semaphore Simplified)
/* POSIX mutex */
pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&mtx);   /* acquire — blocks if held */
    /* critical section */
pthread_mutex_unlock(&mtx); /* release */

/* Condition variable (used WITH mutex) */
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
pthread_cond_wait(&cv, &mtx);  /* atomically release mtx + sleep */
pthread_cond_signal(&cv);       /* wake one waiter */
pthread_cond_broadcast(&cv);    /* wake all waiters */
Monitor (High-Level Construct)

A monitor is a programming language construct that bundles data + procedures + automatic mutual exclusion. Only one process can be active inside the monitor at a time.

  • Used in Java (synchronized methods) and C# (lock)
  • Condition variables: wait() releases monitor + sleeps; signal() wakes one waiter
  • Automatic unlock eliminates many semaphore bugs (forgotten V)
Monitor vs Semaphore
Semaphores are error-prone (easy to forget V, call P twice). Monitors make synchronization automatic and harder to misuse.

Classical Synchronization Problems

1. Producer-Consumer (Bounded Buffer)
/* semaphore full=0, empty=N, mutex=1 */
/* Producer: */
down(&empty);   /* wait for free slot */
down(&mutex);   /* lock buffer */
insert_item();
up(&mutex);
up(&full);      /* signal item available */

/* Consumer: */
down(&full);    /* wait for item */
down(&mutex);
remove_item();
up(&mutex);
up(&empty);     /* signal slot freed */
2. Dining Philosophers (Deadlock Avoidance)

5 philosophers sit at round table. Each needs 2 forks to eat. Between each pair: 1 fork (semaphore). Naive solution deadlocks when all pick up left fork simultaneously.

Solution: at most 4 philosophers allowed to sit simultaneously
OR: pick up both forks atomically (monitor)
OR: odd philosophers pick left first, even pick right first
3. Readers-Writers Problem
  • Multiple readers can read simultaneously (no conflict)
  • A writer needs exclusive access (blocks all others)
  • Reader preference: writers may starve (new readers keep arriving)
  • Writer preference: readers may starve
  • Fair solution: use queue to alternate access

Preemptive vs Non-Preemptive
Preemptive: OS can forcibly take CPU from running process (timer interrupt). Non-preemptive: process runs until it voluntarily yields or blocks.
Round Robin Details

Each process gets a fixed time quantum (10–100ms). At end of quantum, process preempted and moved to back of ready queue.

Quantum too short → too many context switches (overhead)
Quantum too long → poor response for interactive users
Optimal: slightly longer than typical CPU burst
Multilevel Feedback Queue
  • Multiple queues with different priorities and quantum sizes
  • New processes start in highest-priority queue
  • If process uses full quantum → demoted to lower queue
  • If process blocks before quantum → stays or promoted
  • Effectively separates CPU-bound from I/O-bound processes

Turnaround Time = Completion Time − Arrival Time
Waiting Time = Turnaround Time − Burst Time
Response Time = Time of first CPU use − Arrival Time
CPU Utilization = (CPU busy time / Total time) × 100%
Throughput = Number of processes completed per unit time

Thread Model

Thread = Lightweight Process. Threads share address space; each has own stack and registers.

Why Threads?
Web server: one thread per request — all share connection pool and cache. Without threads, need multiple processes (expensive) or complex event-driven code.

⚠ User-Level Thread Problem
If one user-level thread makes a blocking syscall (e.g., read()), the kernel blocks the entire process, starving all other threads. Solution: jacketing — wrap blocking calls with non-blocking versions.
Hybrid Threading (M:N Model)
Map M user threads onto N kernel threads (N < M). Gets benefits of both: fast switching for most threads, true parallelism when needed. Used in Solaris.

Thread Creation Example
#include <pthread.h>

void *worker(void *arg) {
    int id = *(int*)arg;
    printf("Thread %d running\n", id);
    return NULL;
}

int main() {
    pthread_t tid;
    int arg = 42;
    pthread_create(&tid, NULL, worker, &arg);
    pthread_join(tid, NULL);  /* wait */
    return 0;
}

Thread Safety & Common Pitfalls

Shared Global State Problem
/* DANGER: errno is global in old C libraries */
/* Thread 1 calls access() → errno = EACCES */
/* Context switch! */
/* Thread 2 calls open() → errno = ENOENT (overwrites!) */
/* Thread 1 reads wrong errno */

/* SOLUTION: Thread-Local Storage (TLS) */
__thread int errno;  /* each thread has own copy */
Non-Reentrant Library Functions

Functions like strtok(), gethostbyname() use internal static buffers — calling from multiple threads corrupts state.

Reentrant-Safe Functions
Use strtok_r(), gethostbyname_r() (reentrant variants). Always check if a library function is thread-safe before using in multithreaded code.
Stack Overflow Risk

Each thread has its own stack (typically 1-8MB). Kernel auto-grows process stack; does NOT auto-grow thread stacks. Stack overflow in a thread = crash.

Pop-Up Threads & IPC Summary

Pop-Up Threads

When a message arrives, the system automatically creates a new thread to handle it — called a pop-up thread. Key advantages:

  • No history to restore (no saved registers/stack)
  • Very fast creation → low message-handling latency
  • Ideal for network servers handling bursts of requests
