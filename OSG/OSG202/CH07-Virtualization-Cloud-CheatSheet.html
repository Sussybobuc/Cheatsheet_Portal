History & Motivation

Origins

In the 1960s, IBM mainframes introduced virtual machines to allow multiple users to share a single expensive physical machine, each believing they had exclusive access to hardware.

Popek & Goldberg (1974)

Popek & Goldberg formally defined the requirements a computer architecture must satisfy to support efficient virtualization in their landmark 1974 paper.

Three Requirements: Compatibility · Performance · Isolation
  • Compatibility: A VM must behave identically to real hardware — software cannot distinguish VM from real machine.
  • Performance: The majority of guest instructions must execute directly on hardware without VMM intervention (efficiency).
  • Isolation: The VMM must have full control of system resources; no guest program can access resources not explicitly granted.
Modern Revival Drivers
  • Server consolidation: Run many VMs on one server → reduce hardware and operational costs significantly.
  • Isolation: Fault in one VM does not crash others; strong security boundary between tenants.
  • Encapsulation: Entire OS state captured in a file → easy backup, cloning, and migration.
  • Cloud computing: IaaS providers (AWS, Azure, GCP) rent virtualized hardware on demand.
  • Development/testing: Spin up clean environments instantly, snapshot before risky changes.
Key Term
The Virtual Machine Monitor (VMM) — also called the hypervisor — is the software layer that creates and manages VMs. It controls access to all physical hardware resources.

Hypervisor Types

Type 1 — Bare-Metal Hypervisor

Runs directly on physical hardware. No host OS layer between the hypervisor and hardware. The hypervisor itself is the privileged OS kernel. Offers best performance and isolation.

  • VMware ESX / ESXi — enterprise datacenter standard
  • Xen — open-source, used by AWS historically; supports paravirt
  • Microsoft Hyper-V — integrated into Windows Server
  • KVM (Kernel-based VM) — Linux kernel module, turns Linux into Type 1
Type 1: Hypervisor → Hardware (no host OS)
Paravirtualization

Paravirtualization (Xen PV): guest OS is modified to replace non-virtualizable instructions with explicit hypercalls into the VMM. Avoids binary translation overhead at cost of guest OS changes.

Guest OS → hypercall(VMM) → VMM executes privileged op → returns result
Type 2 — Hosted Hypervisor

Runs atop an existing host OS. The host OS manages hardware; the hypervisor runs as an application. Easier to install, more overhead than Type 1.

  • VirtualBox — free, cross-platform, popular for development
  • VMware Workstation — commercial, powerful snapshot/clone features
  • QEMU — software emulator (can also pair with KVM for acceleration)
Type 2: Application → Host OS → Hardware

Popek & Goldberg Criteria (Formal)
A VMM can be constructed if and only if the set of sensitive instructions is a subset of the set of privileged instructions. Sensitive instructions are those that behave differently in user mode vs kernel mode, or that reveal/modify the hardware state. Privileged instructions trap to the OS when executed in user mode.

CPU Virtualization

Three Core Approaches
1. Trap-and-Emulate  |  2. Binary Translation  |  3. Hardware-Assisted (VT-x / AMD-V)
1 — Trap-and-Emulate

Guest executes most instructions natively. When a privileged instruction is encountered in user mode, the CPU traps to the VMM. The VMM inspects the instruction, emulates its effect on the virtual machine state, and returns control to the guest.

Guest instr → privileged? → CPU trap → VMM emulates → resume guest
x86 Problem: 17 Sensitive Non-Privileged Instructions
Classic x86 has 17 instructions (e.g., POPF, PUSHF, SGDT, SIDT, STR) that are sensitive (reveal/modify privileged state) but do not trap in user mode — they silently misbehave or return host values. Pure trap-and-emulate cannot work on classic x86 without additional mechanisms.
2 — Binary Translation (VMware approach)

The VMM scans guest kernel code at runtime, detects problematic instructions, and replaces them with safe sequences that call back into the VMM. Translated code is cached in a translation cache for reuse.

  • Guest user-mode code: runs native (fast path)
  • Guest kernel-mode code: translated by VMM before execution
  • POPF → replaced with VMM-safe flag manipulation stub
Performance
Binary translation adds overhead but modern caching makes it near-native for many workloads. VMware pioneered this for x86 before hardware support existed.
3 — Hardware-Assisted: Intel VT-x / AMD-V

Modern CPUs add a new VMX (Virtual Machine eXtension) mode with two privilege rings: VMX root (hypervisor) and VMX non-root (guest). Guest runs with apparent full hardware access; sensitive operations cause VM Exit to VMM automatically.

InstructionDescription
VMXONEnter VMX operation mode (hypervisor init)
VMXOFFExit VMX operation mode
VMLAUNCHLaunch a new VM (first time)
VMRESUMEResume a VM after VM Exit
VMPTRLDLoad pointer to VMCS structure
VMREAD/VMWRITERead/write VM Control Structure fields
Guest runs (VMX non-root) → sensitive instr / interrupt → VM Exit (hardware saves guest state to VMCS) → VMM handles event (VMX root mode) → VMRESUME → Guest continues (VMX non-root)
VT-x Operation Steps
  1. VMM executes VMXON to enable VMX operation; CPU enters VMX root mode.
  2. VMM allocates a VMCS (VM Control Structure) per vCPU — stores guest/host state and exit controls.
  3. VMM populates VMCS with guest initial register state, entry/exit controls, I/O bitmaps.
  4. VMLAUNCH transfers control; CPU enters VMX non-root mode, guest runs natively.
  5. On sensitive event (privileged instr, interrupt, I/O, EPT fault): hardware VM Exit, saves guest state to VMCS, loads VMM host state.
  6. VMM reads exit reason from VMCS, handles it (emulate device, inject interrupt, etc.).
  7. VMRESUME — CPU restores guest state from VMCS, continues guest execution.
AMD-V (SVM)
AMD's equivalent uses VMRUN / VMEXIT / VMLOAD / VMSAVE instructions. The control structure is the VMCB (VM Control Block). Functionally equivalent to Intel VT-x.

Memory Virtualization

Address Space Hierarchy

With virtualization there are three levels of addresses:

AbbreviationFull NameMeaning
GVAGuest Virtual AddressVirtual address used by guest processes
GPAGuest Physical AddressWhat the guest thinks is physical RAM
HPAHost Physical AddressActual physical machine memory
GVA ──(guest page table)──▶ GPA ──(shadow/EPT)──▶ HPA
Shadow Page Tables (Software)

The VMM maintains a shadow page table that directly maps GVA → HPA. The hardware MMU uses the shadow PT, not the guest's PT. The VMM intercepts all guest modifications to its page table.

Shadow PT Overhead
Every guest CR3 load (page table switch = context switch) causes a VM Exit. Every guest page table modification must be intercepted. Write-protecting guest page tables causes frequent faults. This overhead can be significant for workloads with heavy memory management.
  • VMM write-protects guest PTs → any write triggers fault → VMM updates shadow PT
  • On guest CR3 load → VM Exit → VMM switches to corresponding shadow PT
  • Must maintain one shadow PT per guest address space (per process in guest)
Extended Page Tables — EPT (Intel) / NPT (AMD)

Hardware-assisted 2-level page walk: CPU MMU understands both levels. Guest manages GVA→GPA mappings normally; hardware automatically walks an additional EPT table for GPA→HPA. No VM Exit on CR3 load.

EPT Advantage
Eliminates most VM Exits due to memory management. Guest can modify its own page tables freely. VMM only needs to manage the EPT (GPA→HPA), which changes much less frequently. Dramatically reduces overhead vs shadow PTs.
EPT page walk cost: up to 24 memory accesses (4-level×4-level) vs native: 4 memory accesses → EPT TLB (EPTTLB) caches GPA→HPA translations
TLB Management — VPID

Without tagging, every VM switch requires a full TLB flush to prevent one VM from using another VM's cached translations. VPID (Virtual Processor ID) tags TLB entries with a VM identifier, allowing TLB entries to persist across VM switches.

VPID = 0 → VMM entries | VPID = N → VM N entries (no flush needed on VM switch)
Memory Balloon Driver
VMMs use a balloon driver inside the guest OS to reclaim physical memory. The balloon driver allocates memory inside the guest (inflates), causing the guest OS to swap out other pages. The VMM then reclaims that physical memory for other VMs.
Memory Overcommit Techniques
  • Transparent Page Sharing (TPS): VMM detects identical memory pages across VMs, maps them to single physical page (copy-on-write). Effective for VMs running same OS.
  • Swapping: VMM swaps VM memory pages to disk when physical memory pressure is high.
  • Ballooning: Balloon driver cooperates with VMM to gracefully reclaim memory.
  • NUMA-aware: Modern VMMs place vCPU and memory on same NUMA node for optimal memory bandwidth.

I/O Virtualization

Three Approaches
Device Emulation → Paravirtual (virtio) → Direct Assignment (passthrough)
1 — Device Emulation

VMM software emulates a complete hardware device (e.g., e1000 NIC, IDE disk). Guest uses standard unmodified drivers. VMM intercepts all I/O port accesses and MMIO reads/writes and translates to real device operations.

  • Highest compatibility — no guest OS changes required
  • Lowest performance — every I/O op causes VM Exit + VMM processing
  • Example: QEMU emulates Intel e1000, RTL8139, IDE controller
2 — Paravirtual I/O (virtio)

virtio is a standardized paravirtual I/O framework. The guest OS runs a virtio frontend driver; the VMM implements a virtio backend. Communication via shared memory ring buffers eliminates many VM Exits.

Guest virtio-net driver → virtqueue (shared ring) → VMM backend picks up descriptor → sends to real NIC → interrupt back to guest (batched)
  • virtio-net — paravirtual network card
  • virtio-blk — paravirtual block (disk) device
  • virtio-scsi — paravirtual SCSI controller
  • virtio-balloon — memory balloon driver
  • vhost — moves virtio backend into kernel for even higher throughput
Performance Comparison
Emulated NIC: ~300 Mbps. virtio-net: ~8 Gbps+. Direct assignment: near line-rate (10/25/100 Gbps possible). Always use virtio drivers in production VMs.
3 — Direct Assignment (Passthrough)

A physical device is assigned exclusively to a single VM using the IOMMU (I/O Memory Management Unit). The IOMMU provides DMA remapping so the device uses GPA addresses but the IOMMU translates to HPA transparently.

SR-IOV — Single Root I/O Virtualization
A PCIe device (e.g., a NIC) that supports SR-IOV exposes one Physical Function (PF) and up to hundreds of Virtual Functions (VFs). Each VF appears as a separate PCIe device and can be assigned to a different VM — near-native performance with hardware-level isolation.
  • Intel VT-d / AMD-Vi — IOMMU implementations that enable passthrough
  • DMA remapping prevents device from accessing memory outside its VM's GPA range
  • Interrupt remapping routes device interrupts to the correct VM's vCPU
  • Limitation: device is dedicated to one VM — no sharing between VMs (except SR-IOV VFs)
Live Migration Limitation
Direct passthrough devices generally cannot be live-migrated — the device state is hardware-specific and cannot easily be captured and transferred. Use virtio for migrateable VMs.

Live Migration

Goal & Challenge

Move a running VM from source host (A) to destination host (B) with minimal service disruption. Key challenge: VM continues executing and dirtying memory pages during the transfer.

Typical downtime target: < 100ms for most production workloads
Pre-Copy Migration Algorithm
Phase 1 (Pre-copy): Copy all memory pages to destination Phase 2 (Iterative): Resend pages dirtied during Phase 1 Phase 3 (Stop-and-copy): Pause VM, send remaining dirty pages + CPU state Phase 4: Resume VM on destination, redirect network traffic
  1. Setup: Establish connection between source VMM (A) and destination VMM (B). Allocate memory on B for the VM.
  2. Initial copy: Copy all VM memory pages from A to B over the network. VM continues running on A during this transfer.
  3. Iterative dirty page copy: Track pages written by VM since last transfer (dirty bitmap). Resend those dirty pages. Repeat until dirty set is small enough (convergence criterion met).
  4. Stop VM on source: Briefly pause VM on host A. Copy remaining dirty pages + full CPU register state (vCPU context) + device state to B.
  5. Resume on destination: Start VM on host B using transferred state. VM resumes exactly where it left off.
  6. Network cutover: Send a RARP/gratuitous ARP to update network switches — VM's MAC address now reachable via host B. Redirect storage I/O if using local storage.
  7. Cleanup: Release memory and VM resources on source host A.
Convergence Problem
If the VM writes memory faster than the network can transfer it (high write-rate workloads), the dirty set never shrinks. The VMM may fall back to a fixed iteration limit and then force stop-and-copy regardless of dirty set size — potentially causing longer downtime.
Requirements for Live Migration
  • Shared storage (SAN/NFS) OR block-device replication — so disk data is identical on both hosts
  • Same (or compatible) CPU architecture between source and destination
  • Network connectivity between source VMM and destination VMM (typically on management VLAN)
  • VM's MAC address reachable from destination host after cutover (L2 adjacency or SDN)
Post-Copy Migration (Alternative)
Post-copy transfers only the CPU state first, then pages on demand as the VM faults on them. Eliminates multiple copy iterations — achieves very low initial pause time. Risk: if destination host fails mid-transfer, VM is lost. Pre-copy is generally preferred for reliability.
Use Cases
  • Load balancing: Move VMs from overloaded hosts to underutilized ones automatically (VMware DRS)
  • Hardware maintenance: Evacuate VMs before taking a server offline for maintenance
  • Power management: Consolidate VMs onto fewer hosts during off-peak hours, power off idle servers
  • Disaster avoidance: Move VMs away from failing hardware before actual failure

Containers vs Virtual Machines

Virtual Machines

Full hardware virtualization. Each VM has its own complete OS kernel, device drivers, and system libraries. Strong isolation boundary enforced in hardware.

  • Boot time: 30 seconds to several minutes
  • Image size: GBs (full OS disk image)
  • Isolation: hardware-enforced (VMM boundary)
  • Resource overhead: full OS memory footprint per VM
  • Portability: limited (architecture-specific images)
  • Use case: untrusted tenants, full OS freedom, legacy apps

Containers

OS-level virtualization: all containers share the host kernel. Isolation via Linux namespaces and resource limits via cgroups. Much lighter weight than VMs.

  • Boot time: milliseconds (no OS boot)
  • Image size: MBs (application layers only)
  • Isolation: kernel namespace + cgroup (software only)
  • Resource overhead: near-zero (no duplicate kernel)
  • Portability: high (OCI image standard)
  • Use case: microservices, CI/CD, cloud-native apps
cgroups (Control Groups)

cgroups limit, account, and isolate resource usage of process groups:

  • cpu — CPU time shares and quotas (CFS bandwidth control)
  • memory — RAM and swap limits; OOM killer scope
  • blkio — Block I/O bandwidth limits and scheduling
  • net_cls / net_prio — Network traffic classification and priority
  • pids — Limit number of processes in group (fork bomb protection)
Docker Layered Filesystem

Docker uses overlayfs (overlay filesystem) to stack read-only image layers with a single read-write container layer on top. Layers are content-addressed and shared across containers using the same base image.

VMware Architecture

Product Positioning

VMware Workstation is a Type 2 hypervisor running atop a host OS (Windows or Linux). Despite running as an application, it achieves near-native performance via binary translation and later hardware assist (VT-x).

VMware = Type 2 | Runs on Host OS | Binary Translation + Shadow PTs
Three Main Components
ComponentModeResponsibility
VMXUser-spaceVM configuration UI, virtual device emulation (disk, NIC, USB), user-space I/O handling, snapshot management
VMM DriverHost kernel moduleKernel-mode component of VMware, manages world switch, allocates host memory, interacts with host kernel
VMMVMM world (own address space)Runs in its own context with full hardware control, performs binary translation, manages shadow PTs, handles VM exits
World Switch

VMware uses a concept called a world switch to transition between the host OS world and the VMM world. A world switch is more complete than an OS context switch:

World Switch vs Context Switch: Context switch: saves/restores registers, page table (CR3) World switch: saves/restores entire address space, exception handlers (IDT), segment descriptors (GDT), all debug registers, LDTR, TR — full machine state
  • The VMM runs in its own isolated address space — not in host OS address space
  • World switch is triggered by VMM driver via a special far jump
  • Host OS is completely suspended during VMM execution (appears as if time stopped)
  • On world switch back to host: host OS resumes exactly where it was interrupted
Binary Translation Details

VMware's binary translator operates at basic block granularity. It scans guest kernel code one basic block at a time, translates problematic instructions, and caches the result.

  • POPF → stub that reads EFLAGS from virtual CPU state, sets safe bits
  • PUSHF → stub that constructs EFLAGS from virtual CPU state
  • SGDT/SIDT → stub returning virtual GDT/IDT address, not host values
  • MOV to CR0/CR3/CR4 → translated to world switch or shadow PT update
  • Guest user-mode code: never translated (runs natively, cannot execute privileged instructions)
Translation Cache
Translated basic blocks are stored in a translation cache (TC) indexed by guest code address. On repeated execution of the same code (loops, syscall handlers), the cache avoids re-translation — achieving near-native throughput for common code paths.
Shadow Page Tables in VMware

VMware maintains shadow page tables per guest process address space. The host hardware MMU walks the shadow PT (which maps GVA → HPA directly). The VMM intercepts all guest CR3 loads and guest page table modifications.

Consistency Problem
When the guest modifies its page tables, VMware must update the corresponding shadow PT entry. VMware write-protects guest page tables. Any write fault triggers the VMM to update the shadow entry. This requires careful locking to maintain consistency across vCPUs in SMP VMs.
VMware ESXi (Type 1) — Brief

VMware ESXi is VMware's bare-metal hypervisor for production datacenters. A privileged VM called dom0 (Service Console in older ESX) runs management agents. All other VMs share hardware resources managed by the hypervisor kernel.

  • vSphere — management layer: vCenter Server orchestrates many ESXi hosts
  • DRS (Distributed Resource Scheduler) — automatic load balancing via live migration
  • HA (High Availability) — restart VMs automatically on host failure
  • vMotion — VMware's branded live migration technology
  • VMFS — VMware clustered filesystem for shared VM disk images

Virtualization Security

Attack Surfaces
  • VM escape: Guest exploits vulnerability in hypervisor/emulated device to gain host privileges. Historically in QEMU's Cirrus VGA, Virtio implementations.
  • VM hopping: Attacker moves from one VM to another via shared resource (memory side-channel, network misconfiguration).
  • Hyperjacking: Malicious hypervisor installed beneath legitimate OS (rootkit-level attack).
  • Side-channel attacks: Spectre/Meltdown, Rowhammer — exploit shared CPU caches/DRAM between co-resident VMs.
Spectre / Meltdown in Virtualized Environments
Co-resident VMs on the same physical CPU can potentially leak each other's data via CPU speculation side-channels. Mitigations (KPTI, Retpoline, microcode updates) impose significant performance overhead — up to 30% on I/O-intensive workloads.
Defense Mechanisms
  • IOMMU — prevents DMA attacks from passthrough devices; device cannot access memory outside its allowed GPA range
  • Secure Boot — ensures hypervisor binary is signed and unmodified before loading
  • vTPM — virtual Trusted Platform Module provides key storage and attestation per VM
  • AMD SEV / Intel TDX — hardware-encrypted VM memory so even the hypervisor cannot read guest RAM plaintext
  • Minimal hypervisor attack surface: VMware uses VMM Driver in kernel only for world switch; QEMU runs privileged emulation in separate process
AMD SEV / Intel TDX (Confidential Computing)
SEV (Secure Encrypted Virtualization) encrypts guest memory with a per-VM AES key managed by the AMD Secure Processor. The hypervisor sees only ciphertext. TDX (Trust Domain Extensions) is Intel's equivalent, creating "Trust Domains" with hardware attestation.

Xen & KVM Architecture

Xen Hypervisor

Xen is a Type 1, open-source bare-metal hypervisor originally from Cambridge University (2003). Used historically by AWS EC2 (now transitioning to KVM).

Physical Hardware ──── Xen Hypervisor (ring 0) ──── Dom0 (privileged) | DomU (guest VMs) - Device drivers | - No direct HW access - Toolstack | - Communicates via Xen buses - Management | - PV or HVM mode
DomainRole
dom0Privileged management domain; runs device drivers; controls DomU lifecycle via xl/libxl toolstack
domUUnprivileged guest VMs; no direct hardware access; communicate with dom0 via split-driver model (Xen bus)
Xen Virtualization Modes
  • PV (Paravirtualization): Guest OS modified to use hypercalls instead of privileged instructions. Fast, but requires PV-aware OS (Linux, NetBSD). Runs all in ring 1 (not ring 0).
  • HVM (Hardware Virtual Machine): Unmodified guest using Intel VT-x / AMD-V. Slower than PV historically, but now preferred with PVHVM drivers.
  • PVH: Hybrid — minimal paravirt for boot, then hardware assist. Best of both worlds.
Xen PV hypercall (like system call): Guest (ring 1) → HYPERCALL → Xen (ring 0) e.g.: HYPERVISOR_mmu_update(), HYPERVISOR_sched_op()
KVM (Kernel-based Virtual Machine)

KVM is a Linux kernel module (kvm.ko + kvm-intel.ko or kvm-amd.ko) that turns the Linux kernel itself into a Type 1 hypervisor. Requires Intel VT-x or AMD-V.

  • Each VM appears as a regular Linux process (managed by QEMU process)
  • QEMU handles device emulation in user-space; KVM module handles VT-x VM entries/exits in kernel
  • /dev/kvm — character device; QEMU uses ioctl() to create and manage VMs
  • Supports live migration, EPT, VPID, all modern CPU virtualization features
  • Now the dominant hypervisor for public cloud (AWS Nitro, GCP, Azure)
AWS Nitro System
AWS's custom KVM-based hypervisor offloads I/O (NVMe, VPC networking) to dedicated Nitro Cards (custom ASICs). The result: near bare-metal performance, hypervisor overhead < 1% for most workloads. The Nitro hypervisor itself has minimal attack surface (purpose-built, not general-purpose Linux).
Scheduler: Xen Credit Scheduler

Xen's Credit Scheduler assigns credit points to vCPUs based on configured weights. A vCPU can run until it exhausts its credits (UNDER state), then other vCPUs with credits run. Credits are replenished periodically. Prevents any single VM from monopolizing CPU time.

Credit earned = weight / sum(all weights) × physical_CPU_capacity vCPU states: OVER (negative credits) → UNDER → BOOST (high priority event)

Address Translation Chain
With EPT: GVA → [guest PML4→PDPT→PD→PT] → GPA → [EPT PML4→PDPT→PD→PT] → HPA Total: up to 5×4 = 20+ memory accesses worst case (EPTTLB caches GVA→HPA to avoid full walk)
Pre-Copy Migration
Convergence: dirty_rate < network_bandwidth Downtime ≈ (remaining_dirty_pages × page_size) / bandwidth Iteration i: dirty_pages(i) ≈ dirty_pages(i-1) × (dirty_rate / bw)
Memory Rules
Shadow PT: VMM intercepts every CR3 load + PT write → overhead
EPT: hardware walks 2-level PT → no VM Exit on CR3 load → faster
VPID 0 = hypervisor | VPID n = VM n → tag TLB, avoid flush
Container Isolation Stack
Application Process ──── Linux Namespaces ──── (what it sees) ──── cgroups ──── (resource limits) ──── Host Linux Kernel ──── (shared!) ──── Hardware ────
Cloud Model Memory Hook
IaaS = Infra | PaaS = Platform | SaaS = Software
Pizza-as-a-Service analogy: IaaS: raw ingredients (VMs, storage) — you cook PaaS: semi-prepared (runtime, DB) — you write code SaaS: delivered pizza — just eat (configure & use)
