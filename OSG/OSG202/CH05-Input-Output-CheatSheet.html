I/O Device Categories

Block Devices

Block devices store information in fixed-size blocks, each with its own address. Transfers are always in whole blocks; each block is independently readable and writable regardless of all others.

  • Hard disk drives (HDDs) — typical block 512 B or 4096 B
  • Solid-state drives (SSDs) — page-based internally, block interface externally
  • USB flash drives — NAND flash, block-addressable
  • CD-ROM / Blu-Ray — read-only optical, block-addressed
  • Magnetic tape drives — sequential block access only
Character Devices

Character devices deliver or accept a stream of characters with no block structure; they are not addressable and support no seek operation.

  • Keyboards, mice — input character streams
  • Serial / parallel printers — output character streams
  • Network interface cards — deliver packet streams
  • Audio input/output — sample streams
  • Terminals, PTYs — bidirectional character streams
Other / Special Categories
  • Clocks — generate periodic interrupts; do not fit block/char model
  • Frame buffers — special memory-mapped display devices; byte-addressable
  • Memory-mapped files — OS abstraction layered over block devices
Design Impact
Block devices use buffer caches for performance; character devices use raw I/O streams or line disciplines (termios).

I/O Hardware Concepts

Ports

A port is a connection point — a specific address — through which the CPU communicates with a device controller. Each device family has a unique port address in the I/O address space (e.g., 0x3F8 for COM1 on x86).

Buses

A bus is a shared set of wires carrying address, data, and control signals. Multiple devices share the bus; arbitration resolves simultaneous requests.

Device Controllers

A device controller (adapter) is the chip or board that physically controls the device. It presents a simple register interface to the CPU and hides all hardware complexity (e.g., a SATA controller hides disk head mechanics).

Interrupt Controller

The Programmable Interrupt Controller (PIC / APIC on x86) arbitrates multiple IRQ lines and forwards the highest-priority pending interrupt to the CPU's INTR pin. The OS programs the interrupt vector table (IDT) mapping IRQ numbers to Interrupt Service Routines.

Each controller has: Control · Status · Data-In · Data-Out registers
IRQ_line → PIC/APIC → CPU INTR pin → IDT lookup → ISR runs

The Four I/O Software Layers

I/O software is organized in four hierarchical layers. Requests flow top-down from user program to hardware; completion notifications flow bottom-up from interrupt handler back to user space. Each layer provides a clean interface to the layer above it.

Layer 4 — User-Space Libraries
printf() · scanf() · fread() · fwrite()

Library routines that buffer I/O internally (stdio), format data, and eventually issue write() / read() system calls. Spooling daemons (e.g., cupsd) also live here. Runs in user mode — no privilege needed.

Layer 3 — Device-Independent OS Software
Uniform naming · Buffering · Error reporting · Allocation

Maps device names (/dev/sda) to drivers. Enforces uniform block sizes regardless of hardware. Manages kernel buffers, quota, and device allocation. Reports errors consistently. Protects devices from unauthorized access.

Layer 2 — Device Drivers
Set registers · Check status · Service interrupts

Kernel-mode code, one per device class. Translates abstract requests into device-specific register operations. Registers interrupt handlers. Has a top half (called by kernel for requests) and bottom half (interrupt handler, deferred work via tasklet / workqueue).

Layer 1 — Interrupt Handlers (bottom)
Save state → acknowledge → handle → wake driver

Runs in response to hardware IRQ. Performs the minimum work needed: acknowledge interrupt controller, copy data if needed, wake the blocked driver process, restore context. Runs with interrupts disabled for critical sections.

Request Flow — Top Down
  1. User calls printf(); stdio buffers in user space
  2. Buffer full → library calls write() syscall
  3. Kernel switches to kernel mode
  4. Device-independent layer checks permissions, buffers
  5. Driver writes command to device control register
  6. CPU releases and runs another process
Reply Flow — Bottom Up
  1. Device completes operation, raises IRQ
  2. Interrupt handler runs, acknowledges controller
  3. Handler signals (wakes) the waiting driver
  4. Driver reads status, passes result up
  5. Process unblocked; write() returns

Goals of I/O Software

  • Device independence — programs work with any device; sort </dev/sda same as sort <file.txt
  • Uniform naming — devices mounted into filesystem namespace; no magic numbers in programs
  • Error handling — hardware errors caught as close to hardware as possible; upper layers see clean interface
  • Synchronous vs Async — physical I/O is async (interrupt-driven); OS presents blocking (synchronous) read() to user
  • Buffering — data arriving from network cannot be stored directly in final destination; kernel buffers absorb rate mismatches
  • Sharable vs Dedicated — disk is sharable (multi-process); printer is dedicated (must spool)

Programmed I/O (Polling)

Programmed I/O (also called busy-waiting or polling) has the CPU do all the work of transferring data. The CPU continuously reads the device's status register until it reports ready, then copies data one word at a time.

Classic Polling Loop (C)
/* Print a string to a character device */
void print_string(char *str) {
    while (*str != '\0') {
        /* Busy-wait: spin until device not busy */
        while (status_reg & BUSY)
            ;  /* do nothing */
        data_reg = *str++;
        status_reg = START;
    }
}

/* Example read with timeout */
int poll_read(volatile uint8_t *status, uint8_t *data) {
    int timeout = 100000;
    while (!(*status & READY) && timeout--)
        ;
    if (timeout < 0) return -1; /* error */
    *data = *data_reg;
    return 0;
}
CPU Waste
Polling wastes CPU cycles proportional to device latency. A 1 Gbps NIC polled on a 1 GHz CPU consumes 100% of a core just checking status — no other work done.
When Polling Is Acceptable
  • Very fast devices where wait time is tiny (µs range)
  • Embedded / real-time systems needing deterministic latency
  • Kernel poll loops for < ~10 µs expected wait (avoids interrupt overhead)
  • DPDK / SPDK high-performance networking (deliberate polling)
Programmed I/O: simple · predictable · CPU-intensive

Interrupt-Driven I/O

With interrupt-driven I/O, the CPU initiates an I/O operation, then does other useful work. When the device finishes, it asserts its IRQ line; the CPU is interrupted and runs an Interrupt Service Routine (ISR) to handle the completion.

Interrupt I/O Sequence
  1. Driver writes command to device control register (WRITE | START)
  2. CPU switches to another ready process (does useful work)
  3. Device completes I/O operation asynchronously
  4. Device asserts IRQ line to PIC/APIC
  5. CPU finishes current instruction; checks INTR pin
  6. CPU saves registers, switches to kernel mode (privilege escalation)
  7. IDT lookup: calls registered ISR for this IRQ number
  8. ISR acknowledges interrupt controller (EOI to PIC)
  9. ISR copies data, wakes blocked driver process, returns
  10. Scheduler decides: resume interrupted process or switch
Interrupt Types
TypeSourceExample
Hardware IRQExternal deviceDisk, NIC, keyboard, timer
Software INTUser programint 0x80 (Linux syscall)
ExceptionCPU fault/trapPage fault, divide-by-zero, GPF
Interrupt Overhead
Saving/restoring context + cache misses = ~1–5 µs overhead per interrupt. High-rate devices (>100K irq/s) may use interrupt coalescing or switch to polling (NAPI in Linux).
CPU utilization = 1 − (I/O_wait / total_time)

Direct Memory Access (DMA)

DMA allows a specialized controller to transfer data directly between a device and main memory without CPU involvement. The CPU only sets up the transfer and is interrupted only on completion — not once per byte/word.

DMA Transfer Setup (C)
/* Set up DMA for disk read */
void dma_read_disk(void *buf, size_t n, uint32_t lba) {
    dma_ctrl.src_addr  = disk_sector_addr(lba); /* device addr */
    dma_ctrl.dst_addr  = (uint32_t)(uintptr_t)buf; /* memory */
    dma_ctrl.count     = n;          /* bytes to transfer */
    dma_ctrl.direction = DMA_FROM_DEV;
    dma_ctrl.control   = DMA_START | DMA_IRQ_ON_DONE;
    /* CPU is now free — DMA runs autonomously */
}

/* ISR called when DMA completes */
void dma_isr(void) {
    ack_pic(DMA_IRQ);
    if (dma_ctrl.status & DMA_ERROR) handle_error();
    wakeup(waiting_process);
}
DMA Operation Sequence
  1. CPU programs DMA: source address, destination, byte count, direction
  2. DMA controller takes ownership of the memory bus (cycle stealing)
  3. DMA transfers one word at a time: device → memory buffer
  4. CPU may continue executing (if bus not stolen in that cycle)
  5. DMA interrupts CPU when full transfer complete
  6. ISR wakes blocked process; driver signals completion
T_cpu_busy = T_setup_only ≪ N × T_per_byte (no per-byte interrupt)
Cycle Stealing
DMA steals memory bus cycles from the CPU when transferring, slightly slowing CPU. This is still far faster than CPU doing the transfer itself. Modern systems use separate memory channels to minimize contention.
IOMMU
Modern systems use an IOMMU (Input-Output Memory Management Unit) to translate DMA addresses, preventing rogue devices from accessing arbitrary physical memory — critical for security in virtualized environments.

Device Driver Structure

A device driver is kernel-mode code that provides a standard interface between the OS and a specific class of hardware. One driver per device type (e.g., one driver for all Intel e1000 NICs).

Two Halves of a Driver
HalfWhen CalledResponsibilities
Top HalfCalled by kernel on behalf of user process (synchronous)Validate request, set up DMA/registers, enqueue request, block process
Bottom HalfCalled by interrupt handler (async, after device done)Copy data, update state, wake sleeping processes, schedule deferred work
Driver Responsibilities
  • Initialization — detect device, allocate resources (IRQ, I/O ports, DMA channels), register with kernel
  • Read / Write — translate abstract block/byte requests into device register sequences
  • Interrupt processing — ISR bottom half; handle completion, errors
  • Error recovery — retry failed commands, reset controller, report to kernel
  • Power management — respond to suspend/resume, implement sleep states
  • ioctl / control — device-specific configuration commands beyond read/write
Kernel Privilege
Drivers run in ring 0 (kernel mode) with full hardware access. A bug in a driver can crash the entire OS — hence Linux's KUnit / kselftest driver testing frameworks.
User_read() → VFS → block layer → driver top-half → IRQ → driver bottom-half

Interrupt Handling Steps

When a device raises an IRQ, the CPU and OS must execute a precise sequence before and after running the ISR. This sequence must be transparent — the interrupted program must resume exactly as if nothing happened.

Complete Interrupt Sequence
  1. Device asserts IRQ line; PIC/APIC notes and prioritizes
  2. CPU finishes current instruction (atomicity guarantee)
  3. CPU reads interrupt vector from PIC/APIC (INTA cycle)
  4. CPU saves current PC, PSW/EFLAGS, and CS onto kernel stack
  5. CPU switches to kernel mode (ring 0) and kernel stack
  6. Assembly interrupt stub saves all general-purpose registers
  7. OS sets up interrupt stack frame; loads kernel data segment
  8. OS sends End-of-Interrupt (EOI) to PIC to re-enable further IRQs
  9. C interrupt handler (ISR) runs: copies data, clears status flags
  10. ISR wakes any blocked process waiting for this device
  11. Scheduler checks: should we preempt? Run higher-priority process?
  12. Restore all registers; return from interrupt (IRET / ERET)
  13. CPU resumes interrupted process (or newly scheduled one)
ISR Rules
ISRs run with interrupts partially disabled — keep them as short as possible. Defer slow work to tasklets, workqueues, or softirqs (Linux bottom-half mechanisms).
ISR latency = interrupt detect + context save + handler + context restore

Disk Geometry & Hardware

Physical Structure
  • Platter — circular magnetic disk; modern drives have 1–10 platters stacked on a spindle
  • Track — concentric circle on one platter surface; tens of thousands per surface
  • Cylinder — same track number across all platter surfaces simultaneously; no seek needed between cylinder surfaces
  • Sector — smallest addressable unit; typically 512 B (legacy) or 4096 B (Advanced Format); includes sync, header, data, ECC fields
  • Read/Write Head — one per surface; all heads move together on the same actuator arm
Access Time Components
T_access = T_seek + T_rotation + T_transfer
  • T_seek — time to move actuator arm to target cylinder (mechanical; 1–20 ms)
  • T_rotation — time for target sector to rotate under head (0–full rotation; avg = half rotation)
  • T_transfer — time for data to pass under head at disk RPM × track density

Disk Scheduling & SSDs
Traditional disk scheduling algorithms (FCFS, SSTF, SCAN) are designed for HDDs. For SSDs, simple FCFS or deadline schedulers work best since there is no seek overhead to optimize.

FCFS — First Come, First Served

FCFS services disk requests in the exact order they arrive in the queue. No reordering, no optimization — pure FIFO discipline.

Characteristics
  • Implementation — trivially simple; just a FIFO queue
  • Fairness — perfectly fair; no starvation possible
  • Throughput — poor; head may thrash back and forth across disk
  • Predictability — bounded worst-case; maximum wait = queue length × max seek
Example (queue: 98, 183, 37, 122, 14, 124, 65, 67 from head=53)
53→98→183→37→122→14→124→65→67
Total seek = 45+85+146+85+108+110+59+2 = 640 cylinders
When to Use
FCFS is suitable only when the disk request queue is almost always short (load < 1 request at a time), or when fairness is the overriding concern (real-time systems with deadlines).

SSTF — Shortest Seek Time First

SSTF always services the pending request closest to the current head position, regardless of arrival order. Greedy local optimization.

Characteristics
  • Throughput — much better than FCFS; minimizes each individual seek
  • Average seek — significantly reduced vs FCFS
  • Implementation — moderate: sort by distance from current head
  • Response time — good for middle tracks; poor for outer/inner tracks
Example (same queue, head=53)
53→65→67→37→14→98→122→124→183
Total seek = 12+2+30+23+84+24+2+59 = 236 cylinders
Starvation Risk
If requests keep arriving near the current head position, requests to far tracks can be starved indefinitely. Under heavy load, outer tracks may never be serviced. Use aging to prevent starvation.
SSTF ≈ 63% fewer seeks than FCFS on this example (236 vs 640)

SCAN / Elevator Algorithm

The SCAN algorithm (also called the elevator algorithm) moves the disk arm in one direction servicing all requests in its path, then reverses direction and sweeps back. Like an elevator serving floors.

C-SCAN provides more uniform wait times than SCAN because both ends of the disk are serviced equally — SCAN gives shorter waits near the center of the disk.
SCAN: head sweeps like elevator → fair, no starvation, good throughput

RAID Concepts

RAID (Redundant Array of Independent Disks, originally Inexpensive) presents multiple physical disks as a single logical disk to the operating system. A RAID controller intercepts all read/write operations and distributes them across the array.

RAID Goals
  • Performance — parallel I/O across multiple disks; multiple read/write heads active simultaneously
  • Reliability — redundancy allows continued operation after one or more disk failures
  • Capacity — aggregate storage of multiple drives (minus redundancy overhead)
RAID Implementation Types
TypeDescriptionPerformance
Hardware RAIDDedicated RAID controller with own CPU/cache; transparent to OSBest; battery-backed write cache
Software RAIDOS handles striping/parity (Linux mdadm, Windows Storage Spaces)Good; uses main CPU
Fake RAIDMotherboard "RAID" managed by driver, not real hardwareAvoid; poorest option
RAID Is Not Backup
RAID protects against hardware failure, not against accidental deletion, ransomware, or data corruption. A proper backup strategy is separate from RAID.
Effective capacity (RAID 5) = (N − 1) × smallest_disk
Effective capacity (RAID 1) = 1 × smallest_disk

Striping, Mirroring & Parity

Striping (RAID 0 basis)

Striping splits data into fixed-size strips and writes them round-robin across all N drives. Drives work in parallel, multiplying throughput. Strip size is a key tuning parameter.

Strip size k sectors → stripe = k × N total sectors across all disks
  • Small strips → better parallelism for large I/O; more spindle contention for small I/O
  • Large strips → each request likely served by one drive; less parallelism
  • Typical: 64 KB – 256 KB strip size
Mirroring (RAID 1 basis)

Mirroring writes identical data to N disks simultaneously. Any single disk serves reads; any single disk can fail without data loss. Reads can be distributed for performance.

Parity (RAID 4/5/6 basis)

Parity uses XOR of data strips to generate a redundancy block. If one drive fails, its data can be reconstructed from the remaining data + parity.

P = D0 XOR D1 XOR D2 XOR ... XOR D(n-1)
D_lost = P XOR D0 XOR D1 XOR ... (XOR all surviving drives)
/* XOR parity reconstruction */
uint8_t parity_reconstruct(uint8_t *drives[], int n_drives, int lost_idx) {
    uint8_t result = 0;
    for (int i = 0; i < n_drives; i++) {
        if (i != lost_idx)
            result ^= drives[i][block];
    }
    return result; /* = lost drive's data */
}
RAID 5 Write Penalty
Each write to RAID 5 requires 4 disk operations: read old data, read old parity, write new data, write new parity. This write penalty reduces write performance significantly under heavy random writes.

During RAID 5 rebuild after a failure, all surviving drives are read in full. URE (Unrecoverable Read Error) rate on large drives makes a second failure during rebuild likely. RAID 6 mitigates this.
Choosing RAID Level
RAID 0: max speed, no safety. RAID 1: max safety, 50% overhead. RAID 5: balanced (1 disk overhead). RAID 6: safe for large arrays. RAID 10: performance + safety, expensive.

Hardware Clock

The hardware clock (system timer) generates periodic interrupts that drive all time-related OS functions. Without a clock, the OS cannot implement scheduling, timeouts, or process accounting.

Crystal Oscillator

A crystal oscillator generates a precise, stable frequency (e.g., 14.318 MHz on classic PC). The PIT (or HPET) divides this down to produce a programmable interrupt rate.

Programmable Interval Timer (PIT / HPET)
ComponentDescription
Holding registerLoaded by OS with desired tick interval count
Decrementing counterCounts down from holding register value at crystal frequency
IRQ generationWhen counter hits zero: assert IRQ0, reload from holding register
Hardware Timer Devices (x86)
  • PIT 8253/8254 — legacy; 1.19 MHz base, 18.2 Hz or 1000 Hz tick
  • APIC Timer — per-CPU local timer; used for scheduling quantum
  • HPET — High Precision Event Timer; ≥10 MHz; 3+ comparators
  • TSC — Time Stamp Counter; CPU cycle counter (RDTSC); nanosecond precision
  • RTC — Real Time Clock; battery-backed; keeps wall time across reboots
Tick frequency = crystal_freq / divisor (Linux HZ = 100/250/1000)
Tickless Kernel
Modern Linux uses CONFIG_NO_HZ (dynamic ticks) — the timer is stopped when the CPU is idle and reprogrammed for the next scheduled event, saving power on idle systems.

Soft Timers

Soft timers (virtual timers) avoid the overhead of hardware interrupts for high-frequency timer events. Instead of using hardware IRQs, the OS checks for timer expirations at natural kernel entry points.

Motivation
Interrupt Overhead
Each hardware interrupt costs ~1–5 µs (context save/restore, cache invalidation). At 100K interrupts/sec, timer IRQs alone consume 10–50% of CPU time. Soft timers avoid this cost.
How Soft Timers Work
  1. Application registers timer expiry callback and delay
  2. Kernel records timer in sorted timer queue (min-heap by expiry)
  3. On every kernel entry (syscall, page fault, interrupt), kernel checks timer queue head
  4. If current time ≥ expiry time, run callback immediately (no interrupt needed)
  5. If no kernel entry occurs: hardware timer fires as fallback
Linux Timer API
/* Kernel timer (struct timer_list) */
struct timer_list my_timer;

void my_callback(struct timer_list *t) {
    /* timer expired — do work here */
    mod_timer(t, jiffies + msecs_to_jiffies(100)); /* rearm */
}

void setup(void) {
    timer_setup(&my_timer, my_callback, 0);
    mod_timer(&my_timer, jiffies + HZ); /* fire in 1 second */
}
del_timer_sync(&my_timer); /* cancel timer safely */
hrtimer
Linux high-resolution timers (hrtimer) use the HPET or APIC for nanosecond precision. Used by nanosleep(), clock_nanosleep(), and POSIX realtime timers.

Clock Uses in the OS

Core OS Functions Using Clock
  • Time of Day — maintain wall-clock time; increment jiffies / xtime on each tick; serve gettimeofday()
  • CPU Accounting — charge time to current process on each tick; used by getrusage(), times(), top, scheduler statistics
  • Scheduling Quantum — tick decrements remaining quantum; on expiry, need_resched flag set → context switch at next opportunity
  • Alarm Signals — alarm(n) system call; OS delivers SIGALRM to process after n seconds using timer queue
  • Watchdog Timers — detect hung kernel; if not reset within timeout, trigger panic or NMI handler
  • Network Timeouts — TCP retransmit timer, TIME_WAIT expiry, keepalive intervals
  • Profiling — statistical sampling of PC on each tick; gprof / perf use this to estimate hot code paths
  • Power Management — track idle time; CPU frequency scaling; disk spindown timers

User-Space I/O Libraries

Most user programs never call the kernel directly for I/O — they use standard library wrappers that buffer, format, and eventually issue system calls on their behalf. This user-space layer is crucial for performance.

  • Fully buffered — flush when buffer full; default for files; controlled by setvbuf(fp, NULL, _IOFBF, size)
  • Line buffered — flush on newline or buffer full; default for terminals; _IOLBF
  • Unbuffered — every write is a syscall; default for stderr; _IONBF; use for error/debug output
printf("hello\n") → libc buffer → [when flushed] → write(1, "hello\n", 6) → kernel
Performance Impact
The stdio buffer amortizes syscall overhead. Writing 1 byte at a time without buffering would call write() once per byte — buffering to 4 KB reduces syscall count by 4096×.

Spooling

Spooling (Simultaneous Peripheral Operations On-Line) is a technique to handle dedicated I/O devices in a multiprogramming environment. A dedicated device (like a printer) can only be used by one process at a time, but multiple processes may want to print simultaneously.

Spooling Mechanism
  1. Process wants to print file report.pdf
  2. Process copies file into spooling directory (e.g., /var/spool/cups/)
  3. Process adds job entry to spool queue; continues running immediately
  4. Printer daemon (cupsd) is the only process holding printer open
  5. Daemon takes next job from queue; sends data to printer sequentially
  6. Daemon signals completion; cleans up spool file
Why Spooling Works
  • Disk is a sharable device (concurrent processes can write spool files simultaneously)
  • Printer is a dedicated device (only one process — the daemon — holds it open)
  • User processes never wait for the slow printer; they finish quickly after spooling
  • Daemon serializes access — prints one job at a time in order

Spool Disk Usage
If spool disk fills up, the spooler rejects new jobs. Monitor /var/spool usage. A stuck daemon can cause spool queue to grow unboundedly — always set disk quotas on spool partitions.

OS Power-Saving Techniques
  • Display — DPMS: dim → blank → standby → off after idle timeout; OLED pixel shutdown
  • CPU governor — powersave (min freq), performance (max), ondemand (dynamic), schedutil (scheduler-driven)
  • Disk spindown — hdparm -S n spins disk down after n×5 seconds of idle
  • PCIe ASPM — Active State Power Management; L0s/L1 link states reduce PCIe power
  • USB autosuspend — idle USB devices suspended; /sys/bus/usb/devices/*/power/autosuspend
  • WiFi power save — NIC sleeps between beacons; wake on scheduled DTIM intervals
Linux: /sys/bus/cpu/cpufreq/scaling_governor = "schedutil"
Hibernate vs Sleep
S3 (sleep) keeps RAM powered — fast resume but uses battery. S4 (hibernate) writes RAM to disk — zero battery drain, but slow resume. Modern laptops use Modern Standby (S0ix) — CPU in deep sleep but NIC active for push notifications.
