No Memory Abstraction → Address Spaces

No Memory Abstraction (Early Systems)

Earliest systems: programs access physical memory directly. Address in instruction = real RAM address.

⚠ Problems with No Abstraction
If two programs run simultaneously, they can overwrite each other's memory. Program 1 jumps to address 28 — it might land inside Program 2's code!
Base & Limit Registers

First solution: each process gets two hardware registers:

  • Base register: physical start address of process in RAM
  • Limit register: size of process's address space
  • Every memory access: physical = virtual + base
  • If virtual ≥ limit: hardware trap → OS kills process
Physical Address = Virtual Address + Base
Check: Virtual Address < Limit (else → SIGSEGV)
Base/Limit Drawback
Requires copying entire process to contiguous memory block. As processes come and go, memory becomes fragmented with unusable holes.

Swapping

Swapping = move entire process between RAM and disk to share limited memory.
Swapping Process
  1. OS needs memory for new/runnable process
  2. Select victim process (blocked, lowest priority)
  3. Write victim's entire address space to swap partition on disk
  4. Load target process into freed RAM
  5. Update base/limit registers for loaded process
Memory Compaction

After many swap-ins/outs, RAM has many holes between processes. Compaction moves all processes down to close holes — but it's expensive (copying gigabytes).

Memory Allocation Strategies
StrategyMethodResult
First FitFirst hole big enoughFast; creates small holes at start
Best FitSmallest hole that fitsSlow search; many tiny unusable holes
Worst FitLargest holeLeaves largest remaining hole; rarely used
Next FitFirst fit from last positionFaster than first fit; spreads allocation

Fragmentation

Fragmentation = wasted memory that cannot be used for new allocations.
External Fragmentation

Free memory exists in many small holes scattered across RAM. Total free space is enough, but no single contiguous block is available for the new process.

Total Free = Σ(all holes) ≥ Request → but fragmented!
Internal Fragmentation

Memory allocated is larger than what was requested (e.g., allocate in fixed 64-byte chunks, process needs 50 bytes → 14 bytes wasted inside the block).

TypeLocationCauseSolution
ExternalBetween allocated blocksVariable-size allocationCompaction, paging
InternalInside allocated blockFixed-size allocationSmaller block size
Paging Solves External Fragmentation
By dividing memory into fixed-size pages, there's never external fragmentation — any free page frame fits any needed page. Only internal fragmentation remains (last page of process may be partially used).

Virtual Memory & Paging Mechanics

Core Concept
Virtual Memory = process sees large contiguous address space; OS/MMU maps pages to physical frames on demand.

The MMU (Memory Management Unit) is hardware that translates virtual addresses to physical addresses on every memory access, using the page table.

Address Translation
Virtual Address: [page number (p) | offset (d)]
Physical Address: [frame number (f) | offset (d)]
page_table[p] → f (if present bit = 1)
Physical = f × page_size + d
Typical Sizes (32-bit)
ParameterTypical
Page size4 KB (2¹² = 4096 bytes)
Virtual address32 bits → 4 GB address space
Page number bits20 bits → 2²⁰ = 1M pages
Offset bits12 bits (log₂(4096))
Page table size1M entries × 4 bytes = 4 MB
Page Table Entry (PTE) Fields
BitNameMeaning
PPresent1=in RAM, 0=on disk (page fault)
RReferencedSet on any access; cleared by OS periodically
MModified/DirtySet on write; page must be saved before eviction
CachingCache disableFor memory-mapped I/O registers
ProtProtectionread/write/execute permission bits
Frame #Physical framePoints to physical page frame
Page Fault Handling — Step by Step
  1. CPU references virtual address; MMU checks PTE — P=0 → hardware trap
  2. OS saves faulting process state (registers, PC)
  3. OS finds physical address on disk (swap space or file)
  4. OS selects victim frame (using replacement algorithm)
  5. If victim is dirty (M=1): write victim page to disk
  6. Read faulting page from disk into frame
  7. Update page table: set P=1, frame# for faulted page; invalidate victim's PTE
  8. Restart faulting instruction

Demand Paging

Demand Paging = load pages into RAM only when they are first referenced (not at process startup).
Why Demand Paging?
  • Process may need only a fraction of its address space
  • Allows more processes in memory simultaneously
  • Faster startup — no need to load entire program
  • Programs can have address spaces larger than physical RAM
Locality of Reference

Programs exhibit temporal locality (recently used pages likely used again soon) and spatial locality (pages near recently used pages likely used). This makes demand paging effective.

Copy-on-Write (COW)
After fork(), parent and child share all pages (read-only). When either writes a page, a private copy is made. Avoids copying GB of memory on fork — critical optimization.
COW: fork() = O(1); actual copy happens lazily on first write to shared page

Page Table Structures

Problem: Page Table Size

64-bit address space with 4KB pages → 2⁵² page table entries → impossibly large. Solutions:

1. Multi-Level Page Tables

Split page number into multiple levels. Two-level paging (x86-32): 10 bits → page directory, 10 bits → page table, 12 bits → offset.

Virtual: [dir(10) | table(10) | offset(12)]
Look up: PD[dir] → PT[table] → frame → frame×4K + offset
  • Only allocated page directory/table entries exist in memory
  • Unused regions: no page table allocated — saves memory
  • x86-64 uses 4-level page tables (PML4 → PDP → PD → PT)
2. Inverted Page Table

One entry per physical frame (not per virtual page). Size ∝ physical RAM, not address space. Used in IBM POWER, some 64-bit systems.

  • Entry: (PID, virtual page number) → physical frame
  • Cons: lookup requires search (use hash table); slower

TLB — Translation Lookaside Buffer

TLB = small, fast hardware cache of recent virtual→physical translations (associative memory).
TLB Operation
  1. CPU generates virtual address, extracts page number p
  2. TLB checked for page p — all entries checked in parallel
  3. TLB Hit: frame number found instantly → fast address translation
  4. TLB Miss: must walk page table in RAM → slow; update TLB with new entry
  5. If TLB full: evict one entry (LRU or random)
Effective Access Time =
h × TLB_time + (1-h) × (TLB_time + Memory_time)
where h = hit ratio (typically 0.98 - 0.999)
TLB Performance Example
/* TLB hit time: 2ns, Memory access: 100ns */
/* h = 0.98 */
/* EAT = 0.98*(2) + 0.02*(2+100) = 1.96 + 2.04 = 4ns */
/* vs no TLB: 100+100 = 200ns for each page walk */
TLB & Context Switch
⚠ TLB Flush on Context Switch
When switching processes, old TLB entries are invalid! Must flush entire TLB (expensive) OR tag each entry with an ASID (Address Space ID) — avoids flush.

Page Table Entry Details

x86-32 PTE Layout (32 bits)
/* Bits 31-12: frame number (20 bits = 2^20 frames × 4KB = 4GB)
   Bit 11-9:  OS/software use (free for OS)
   Bit  8:    Global (don't flush on context switch)
   Bit  7:    Page Size (0=4KB, 1=4MB huge page)
   Bit  6:    Dirty (modified bit, M)
   Bit  5:    Accessed (referenced bit, R)
   Bit  4:    Cache Disable
   Bit  3:    Write-Through
   Bit  2:    User/Supervisor (0=kernel only)
   Bit  1:    Read/Write (0=read-only, 1=r/w)
   Bit  0:    Present (P) */
Huge Pages

Huge pages (2MB or 1GB) reduce TLB pressure: a single TLB entry covers 2MB instead of 4KB. Useful for databases, HPC. Linux: mmap(MAP_HUGETLB).

NX/XD Bit
Modern CPUs add an NX (No-Execute) or XD (Execute Disable) bit to PTEs. Marks stack/data pages as non-executable — prevents code injection attacks.

NRU — Four Page Classes

At each clock interrupt, OS clears R bits. On page fault, classify all pages and evict random from lowest nonempty class:

ClassRMMeaning
000Not referenced, not modified — best to evict
101Not referenced, but dirty — must write to disk
210Referenced, clean — recently used
311Referenced and dirty — most expensive to evict
Clock Algorithm

All frames in circular list. Hand sweeps clockwise on page fault:

  • If R=0: evict page, insert new page, advance hand
  • If R=1: clear R, advance hand (second chance given)
Aging Algorithm (LRU Approximation)
/* Each page has 8-bit counter, initially 0 */
/* On each clock tick, for each page:
   counter >>= 1;        // shift right
   counter |= (R << 7); // add R to MSB
   R = 0;                // clear R bit */
/* On page fault: evict page with lowest counter */
/* Example after 5 ticks:
   Page 0: 11110000 (recently used heavily)
   Page 3: 00100000 (used 3 ticks ago, not since)
   → Evict Page 3 */
Bélády's Anomaly (FIFO only): more physical frames → MORE page faults in some cases. Does NOT affect LRU or OPT.

Segmentation

Segmentation = variable-size logical units matching programmer's view (code, data, stack).
Segmented Address
Virtual Address = (segment number s, offset d)
Physical = segment_table[s].base + d
Check: d < segment_table[s].limit
Segmentation vs Paging

Internal fragmentationYes (last page partial)None
Programmer visibleNo (transparent)Yes (segment:offset)

Paged Segmentation (Best of Both)
Multics: each segment is paged internally. Gets variable-size logical units WITHOUT external fragmentation. Complex but powerful.

Memory Allocation — Heap & Free Lists

Dynamic Memory Allocation

The OS maintains a free list (linked list of free memory holes). When a process requests memory, OS searches free list and splits a hole.

First Fit vs Best Fit vs Worst Fit
/* Free list: [10KB] [4KB] [20KB] [18KB] */
/* Request: 12KB */

/* First Fit: scan from start → use 20KB hole */
/*   → hole becomes [8KB] (20-12=8) */

/* Best Fit: find smallest hole ≥ 12KB → 20KB hole */
/*   → leaves 8KB (same here, but generally wastes less) */

/* Worst Fit: largest hole → 20KB */
/*   → leaves 8KB (bad: destroys large holes) */
Buddy System

Memory divided into blocks of size 2^k. When splitting, always create two equal-size buddies. When freeing, if buddy is also free, merge back up.

Buddy of block at addr x of size s: x XOR s
  • Fast allocation and deallocation (O(log n))
  • Internal fragmentation: request 65 → allocate 128
  • Used in Linux kernel: __get_free_pages()

Working Set Model & Thrashing

Working Set Definition
Working Set W(k,t) = set of pages referenced by process in the k most recent memory references at time t.

The working set is the set of pages a process is actively using. If entire working set is in RAM → few page faults. If not enough frames → thrashing.

Thrashing

Thrashing occurs when a process does not have enough frames to hold its working set. It page faults constantly, spending more time doing page I/O than actual computation.

⚠ Thrashing Cascade
OS sees low CPU utilization → adds more processes → each gets fewer frames → all thrash → CPU utilization drops further → vicious cycle. Solution: reduce degree of multiprogramming (fewer active processes).
If Σ(working sets) > Physical RAM → thrashing!
Detection: page fault rate spikes, CPU utilization drops
Working Set Algorithm

Track pages used in the last τ seconds of virtual time. On page fault, evict any page NOT in working set.

/* Working Set algorithm (simplified) */
/* For each page table entry: */
if (R == 1) {
    /* page referenced this tick */
    last_use = current_virtual_time;
    R = 0;
} else {
    age = current_virtual_time - last_use;
    if (age > tau)
        /* evict: not in working set */
        evict_page();
    else
        /* spare: still in working set */
        remember_oldest();
}
WSClock (Working Set + Clock)
  • Circular list of frames with Time-of-Last-Use field
  • On page fault: inspect page at hand
  • If R=1: clear R, update time, advance hand
  • If R=0 and age > τ: evict (not in working set)
  • If R=0 and age ≤ τ: spare (still in working set)
  • Widely used — good performance, efficient

Global vs Local Allocation

Frame Allocation Policies
PolicyDescriptionRisk
LocalEach process has fixed number of frames; replaces only its own pagesUnderuse if process doesn't need all frames
GlobalProcess can take frame from any other processGreedy process can starve others
Global Often Better in Practice
Global replacement results in better overall throughput because frames flow to processes that need them. But one badly-behaving process can degrade all others.
Page Fault Frequency (PFF) Control

OS monitors each process's page fault rate:

  • Fault rate too high → process needs more frames → give it one
  • Fault rate too low → process has excess frames → take one back
  • If total frames insufficient → suspend a process (swap out)
Target: keep page fault rate in [low_threshold, high_threshold]

Memory-Mapped Files & mmap()

Memory-Mapped Files

mmap() maps a file (or anonymous memory) into the process's address space. Reads/writes to memory automatically read/write the file. Used for:

  • Shared libraries: .so/.dll files mapped into each process (shared read-only pages)
  • Shared memory IPC: two processes map same file; communicate via memory
  • Large file I/O: OS handles paging automatically
  • malloc() internally uses mmap() for large allocations
/* Map file into address space */
int fd = open("data.bin", O_RDWR);
char *mem = mmap(NULL, file_size,
                 PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);
mem[100] = 'X';   /* writes directly to file */
munmap(mem, file_size);
close(fd);
Shared Library Efficiency
libc.so is mapped into every process. Since pages are read-only, they are shared — only one copy in RAM for all processes using it.

Memory Management Key Formulas & Quick Reference

Address Translation Formula
Page Number (p) = virtual_addr / page_size
Offset (d) = virtual_addr mod page_size
Frame (f) = page_table[p]
Physical Addr = f * page_size + d
TLB Effective Access Time
EAT = h*(t_TLB) + (1-h)*(t_TLB + t_mem)
h = hit ratio, t_TLB ≈ 2-5ns, t_mem ≈ 100ns
