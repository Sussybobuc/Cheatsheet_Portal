CIA Triad & Threats

The CIA Triad
Confidentiality · Integrity · Availability
  • Confidentiality — only authorized principals may read data; prevents unauthorized disclosure
  • Integrity — data has not been tampered with or altered by unauthorized parties; ensures trustworthiness
  • Availability — system and data remain accessible to authorized users when needed; prevents denial of service
Authenticity (4th goal)
Some models add Authenticity — verifying that data originates from the claimed source (not fabricated). Addressed by digital signatures.
Key Definitions
  • Security policy — a formal statement of what is and is not allowed in a system
  • Security mechanism — the technical means by which a policy is enforced (encryption, ACLs, firewalls…)
  • Threat — a potential cause of harm to a system or organization
  • Attack — a deliberate attempt to exploit a vulnerability
  • Vulnerability — a weakness that can be exploited by a threat

Security Models Overview
Policy vs. Mechanism
A policy says "employees may only read files in their department." A mechanism (e.g., ACL) enforces that constraint technically. Weak mechanisms break even strong policies.
Attack Surfaces
  • Software — bugs, buffer overflows, logic errors in OS/apps
  • Network — open ports, unencrypted protocols, misconfigured firewalls
  • Human — phishing, social engineering, weak passwords, insider threats
  • Physical — stolen hardware, shoulder surfing, dumpster diving
Threat vs Risk
Risk = Threat × Vulnerability × Impact
Principle of Least Privilege
Every process, user, or system component should operate using the minimum set of privileges necessary to complete its function. Limits blast radius of a compromise.

Authentication Methods

Three Authentication Factors
Know · Have · Are
  • Something you Know — password, PIN, passphrase, security question answer
  • Something you Have — smart card, OTP hardware token, FIDO2 key, phone (TOTP app)
  • Something you Are — biometrics: fingerprint, iris scan, face recognition, voice print
MFA / 2FA
Multi-Factor Authentication requires 2 or more independent factors. Compromising one factor alone is insufficient. Even if a password is stolen, the attacker still lacks the hardware token.
Challenge-Response Protocol
  1. Server generates a random nonce (challenge) R and sends to client
  2. Client computes f(R, secret) where f is a keyed-hash function (e.g., HMAC-SHA256)
  3. Client sends response f(R, secret) to server
  4. Server independently computes f(R, secret) using stored secret and compares
  5. If equal → authenticated. Secret never travels over the wire.
Why Challenge-Response?
Prevents replay attacks — a recorded response cannot be reused because the next challenge will be a different nonce R'.
Biometric Metrics
MetricMeaningIdeal Value
FARFalse Accept Rate — impostor accepted as legitimate userAs low as possible
FRRFalse Reject Rate — legitimate user rejectedAs low as possible
EEREqual Error Rate — point where FAR = FRRLower EER = better system
FAR vs FRR Tradeoff
Lowering the acceptance threshold reduces FAR but increases FRR. Raising it does the opposite. Security systems prefer low FAR; convenience-focused prefer low FRR.
Lamport One-Time Password Hash Chain
P_0 = seed, P_i = H(P_{i-1})
Server stores P_n. Client sends P_{n-1}.
Server verifies: H(P_{n-1}) == P_n → OK
  • Server stores P_n (the last hash in chain). Client sends next-to-last value.
  • Server applies H() once and compares — if match, authenticated
  • Each password used only once; chain consumed in reverse
  • Even if attacker sees P_{n-1}, they cannot derive P_{n-2} (one-way hash)

Password Security

Never Store Plaintext Passwords
If the password file is read, all accounts are immediately compromised. No legitimate system stores plaintext passwords.
UNIX /etc/shadow Scheme
stored = H(password || salt)
salt = random n-bit value, stored alongside hash
  • /etc/shadow stores $algorithm$salt$hash per user
  • Classic UNIX used 12-bit salt → 4096 possible salts
  • Verification: compute H(input || stored_salt), compare to stored hash
  • Password never stored — only the hash is kept
Why Salt Defeats Precomputation
  • Dictionary attack (no salt): precompute H(word) for all common words, compare entire table to password file in one pass
  • With 12-bit salt: attacker must compute H(word || salt_i) for each of 4096 salts — 4096× more work
  • Rainbow tables: time-memory tradeoff chains of hashes — also defeated by salt (table must be rebuilt per salt)
salt forces attacker to crack each password individually
Modern Password Hashing
AlgorithmKey PropertyStatus
MD5 / SHA-1Too fast: 10B+ hashes/sec on GPUDo NOT use
bcryptAdaptive work factor (cost param), blowfish-basedRecommended
scryptMemory-hard: resists ASIC/GPU attacksRecommended
Argon2idPHC winner; memory+time+parallelism paramsBest practice
MD5 / SHA-1 Are Too Fast
A GPU can compute billions of SHA-1 hashes per second. bcrypt with cost=12 takes ~250ms per hash — intentionally slow to frustrate brute force.
Pepper
A pepper is a secret value (like salt) stored outside the database (e.g., in an HSM or config). Even if the DB is stolen, attacker can't crack without the pepper.

Symmetric Cryptography

Core Concept
The same key is used for both encryption and decryption. Both parties must share the key securely before communication — the key distribution problem.
C = E_K(P) (encrypt plaintext P with key K) P = D_K(C) (decrypt ciphertext C with same key K)

AES — Rijndael (NIST Standard 2001)
  • 128-bit block size; supports 128, 192, 256-bit keys
  • 10/12/14 rounds of SubBytes → ShiftRows → MixColumns → AddRoundKey
  • Hardware acceleration: AES-NI instructions on x86/ARM
Block Cipher Modes of Operation
ModePropertiesUse Case
ECBEach block encrypted independently — patterns visible in ciphertext!Never use

ECB is Insecure
Identical plaintext blocks produce identical ciphertext blocks. The infamous "ECB penguin" demonstrates structure leaking through encryption.
Conceptual AES Usage
// Conceptual AES encrypt/decrypt (OpenSSL API)
AES_KEY enc_key, dec_key;
uint8_t key[16], iv[16], pt[16], ct[16], out[16];

AES_set_encrypt_key(key, 128, &enc_key);
AES_cbc_encrypt(pt, ct, 16, &enc_key, iv, AES_ENCRYPT);

AES_set_decrypt_key(key, 128, &dec_key);
AES_cbc_encrypt(ct, out, 16, &dec_key, iv, AES_DECRYPT);
// out == pt (original plaintext recovered)
Symmetric crypto is fast; problem is securely distributing the shared key

Asymmetric (Public-Key) Cryptography

Key Pair Concept
Two mathematically linked keys: public key (share freely) and private key (never share). What one key encrypts, only the other can decrypt.
RSA — Rivest–Shamir–Adleman
// Key Generation p, q = large distinct primes (~2048 bits each) n = p × q (modulus) φ(n) = (p-1)(q-1) (Euler totient) choose e: 1 < e < φ(n), gcd(e, φ(n)) = 1 d = e⁻¹ mod φ(n) (mod inverse) Public key: (e, n) Private key: (d, n)
// Encryption & Decryption C = M^e mod n (encrypt with public key) M = C^d mod n (decrypt with private key)
// Digital Signature (reversed roles) S = H(M)^d mod n (sign with private key) verify: S^e mod n == H(M) (verify with public key)
Security relies on hardness of factoring n = p × q into p and q
  • Current recommendation: RSA-2048 minimum, RSA-4096 for long-term secrets
  • PKCS#1 v1.5 padding was broken — use OAEP padding for encryption
  • PSS padding for signatures
  • RSA-2048 provides ~112 bits of security
Diffie-Hellman Key Exchange
// Public parameters: prime p, generator g Alice: picks secret a, sends A = g^a mod p Bob: picks secret b, sends B = g^b mod p Shared secret: Alice: s = B^a mod p = g^(ab) mod p Bob: s = A^b mod p = g^(ab) mod p → Both derive same s without transmitting it!
DH Never Transmits the Key
An eavesdropper sees g, p, A, B but computing a from g^a mod p is the Discrete Logarithm Problem — computationally infeasible for large primes.
ECDH — Elliptic Curve Diffie-Hellman
  • Same concept, using elliptic curve math instead of integer modular exponentiation
  • 256-bit ECDH ≈ 3072-bit DH in security — much smaller keys!
  • Curve25519 and P-256 are common curves used in TLS 1.3
Comparison: Symmetric vs Asymmetric
PropertySymmetricAsymmetric
Key count1 shared keyKey pair (pub + priv)
SpeedFast (hardware accel)Slow (1000× slower)
Key distributionProblem — must be pre-sharedSolved — public key shared freely
Key scalingO(n²) keys for n partiesO(n) key pairs
Use in TLSBulk data encryptionHandshake / key exchange
Hybrid Encryption
In practice: use asymmetric (RSA or ECDH) to securely exchange a random session key, then use AES-GCM for all bulk data. This is exactly how TLS works.

Hash Functions & Digital Signatures

Cryptographic Hash Properties
  • Deterministic — same input always produces same digest
  • Preimage resistant — cannot reverse H(x) to find x
  • Second preimage resistant — given x, cannot find x' where H(x)=H(x')
  • Collision resistant — cannot find any x, x' where H(x)=H(x')
  • Avalanche effect — 1-bit input change → ~50% output bits change
Hash Algorithm Status
HashDigest SizeStatusNotes
MD5128-bitBrokenCollisions found 2004; do not use for security
SHA-1160-bitBrokenSHAttered attack 2017; deprecated by browsers/CAs
SHA-256256-bitSecurePart of SHA-2 family; current standard
SHA-512512-bitSecureSHA-2; faster on 64-bit CPUs
SHA-3224–512-bitSecureKeccak sponge construction; NIST 2015
Digital Signature Verification Steps
  1. Sender computes digest = H(document) using agreed hash function (e.g., SHA-256)
  2. Sender encrypts digest with their private key: sig = D_priv(digest)
  3. Sender transmits (document, sig) to receiver
  4. Receiver independently computes digest' = H(document)
  5. Receiver decrypts signature with sender's public key: digest'' = E_pub(sig)
  6. If digest' == digest'' → signature valid; document unmodified and from correct sender
// RSA property required for signatures: E_pub( D_priv(x) ) = x D_priv( E_pub(x) ) = x // RSA is commutative — both operations work in either order
PKI — Public Key Infrastructure
  • Certificate Authority (CA) — trusted third party that signs public key certificates
  • X.509 certificate — binds a public key to an identity (domain name, organization)
  • Browsers ship with ~40–140 trusted root CAs pre-installed
  • Certificate chain — root CA → intermediate CA → end-entity cert
  • OCSP / CRL — mechanisms to check if a certificate has been revoked
CA Compromise is Catastrophic
If a root CA is compromised, attackers can issue fraudulent certificates for any domain. Certificate Transparency logs help detect mis-issued certs.

Access Control Models
  • DAC (Discretionary) — resource owner decides access. UNIX permissions, Windows NTFS ACLs. Flexible but owner can make mistakes.
  • MAC (Mandatory) — system policy forces access rules based on security labels. Users cannot override. Used in military/classified systems.
  • RBAC (Role-Based) — permissions assigned to roles (admin, editor, viewer); users assigned to roles. Simplifies management at scale.
  • ABAC (Attribute-Based) — policies based on attributes of user, resource, environment (time, location). Most expressive.
Bell-LaPadula Model (MAC for Confidentiality)
Simple Security Property (no read up): Subject s at level L can read object o only if level(o) ≤ L *-Property (no write down): Subject s at level L can write object o only if level(o) ≥ L
BLP: "Read down, Write up" — prevents leaking secrets to lower levels
Linux Mandatory Access Control
  • SELinux — every process and file has a security context (type enforcement). Used in Android, RHEL/Fedora.
  • AppArmor — profile-based; simpler than SELinux. Used in Ubuntu, Debian.
  • Both enforce that even root cannot exceed what policy allows
RBAC in Practice
Rather than granting Alice individual file permissions, assign her the editor role. All editors share the same permissions. Revoking access means removing the role assignment.
setuid / setgid Bits
  • setuid — execute with file owner's privileges (e.g., passwd runs as root)
  • Allows temporary privilege elevation for specific operations
  • Dangerous if the setuid binary has vulnerabilities — attacker gains root

Rootkits Are Especially Dangerous
Rootkits that modify the kernel intercept system calls — readdir(), ps, netstat all lie. You cannot trust the OS to detect them. External boot media or remote attestation required.
Rootkit Techniques
  • Hook sys_getdents to hide directories/files from ls
  • Patch /proc filesystem to hide processes from ps
  • Modify kernel module list to hide from lsmod
  • Userland rootkits: replace system binaries (ls, ps, netstat) with trojaned versions
Worm vs Virus
A virus needs a host file and human action (running an infected program) to propagate. A worm spreads automatically over networks by exploiting software vulnerabilities — no user interaction needed (e.g., Morris Worm exploited sendmail and fingerd bugs).
Ransomware Kill Chain
  1. Initial access: phishing email with malicious macro or exploit
  2. Establish persistence; exfiltrate valuable data (double extortion)
  3. Enumerate network shares and backup systems
  4. Delete shadow copies: vssadmin delete shadows
  5. Encrypt all files with attacker-controlled key; drop ransom note

OS Defense Mechanisms

ASLR — Address Space Layout Randomization
What It Does
Randomizes the base addresses of the stack, heap, libraries, and executable at each run. Attacker cannot hardcode memory addresses in their exploit payload.
  • Enabled by default on modern Linux, Windows, macOS
  • Defeated by information leaks (format string bugs that reveal addresses)
  • 32-bit: only ~16 bits of entropy — brute-forceable! 64-bit is much stronger.
Stack Canaries
stack frame layout: [ local_vars ] [ canary ] [ saved_rbp ] [ ret addr ] before function return: check canary == original → if not, abort()
  • GCC: -fstack-protector flag inserts canary automatically
  • Canary value chosen randomly at program start, stored in %gs segment
  • Sequential stack overflows must overwrite canary to reach return address — detected
  • Defeated by: arbitrary write primitives, canary leak + overwrite
DEP / NX Bit (No-Execute)
  • CPU marks stack and heap pages as non-executable (NX bit in page table entry)
  • Injected shellcode on the stack cannot be executed — causes fault
  • Defeated by Return-Oriented Programming (ROP) — doesn't inject new code, reuses existing code
Sandboxing
  • seccomp (Linux) — whitelist of allowed syscalls; others → SIGKILL
  • Chrome renderer uses seccomp-BPF to restrict to ~few dozen syscalls
  • namespaces + cgroups → Docker container isolation
  • Syscall filtering limits damage even if attacker achieves code execution
Attack → Defense Mapping

Common Attack Techniques

Buffer Overflow — Stack Smashing
Core Concept
Write more data into a fixed-size stack buffer than it can hold. Overflow overwrites adjacent stack memory including the saved return address. When the function returns, execution jumps to attacker-controlled address. 
// Vulnerable function — never use gets()!
void vuln(void) {
    char buf[64];
    gets(buf);      // reads unlimited input!
    // if input > 64 bytes:
    // buf overflows → overwrites return address
}

// Stack layout after overflow:
// [buf: 64 bytes]
// [saved rbp: 8 bytes]  ← overwritten
// [ret addr: 8 bytes]   ← redirected to shellcode!

// Safe alternative:
void safe(void) {
    char buf[64];
    fgets(buf, sizeof(buf), stdin);  // bounded
}
gets() is Banned
gets() has been removed from C11 standard. Also: strcpy(), sprintf() without bounds. Use strncpy(), snprintf(), strlcpy().
Heap Spray
  • Fill the heap with many copies of NOP sled + shellcode
  • Jump to any approximate address in the heap — high probability of landing in sled
  • Used when exact address cannot be predicted (browser exploits with controlled allocation)
  • Mitigated by ASLR and heap metadata protections
SQL Injection
// Vulnerable query construction: "SELECT * FROM users WHERE user='" + input + "'" // Attacker inputs: ' OR '1'='1 // Resulting query: SELECT * FROM users WHERE user='' OR '1'='1' // '1'='1' is always true → returns ALL users! // Worse — input: '; DROP TABLE users; -- SELECT * FROM users WHERE user=''; DROP TABLE users; --'
  • Prevention: use parameterized queries / prepared statements
  • db.query("SELECT * FROM users WHERE id = ?", [userId])
  • Input is never interpreted as SQL — always treated as data
  • Also: ORMs, stored procedures, input validation/whitelist
TOCTOU — Time-of-Check-to-Time-of-Use
Race Condition Vulnerability
A resource is checked (e.g., access() for permissions) and then used (e.g., open()) — but the resource can change between check and use. Attacker swaps the resource in the gap. 
// TOCTOU vulnerability:
if (access("file.txt", W_OK) == 0) {  // check
    // attacker replaces file.txt with symlink here!
    fd = open("file.txt", O_WRONLY);  // use
    // now writing to attacker's target file
}
// Fix: use O_NOFOLLOW, atomic operations, capabilities
Return-Oriented Programming (ROP)
Bypassing NX/DEP
Instead of injecting shellcode, attacker chains together small existing code sequences ending in ret (gadgets). The return address is overwritten to point to a gadget; each gadget's ret pops the next gadget address off the stack.
// ROP chain concept: stack: [ gadget1_addr | gadget2_addr | gadget3_addr | ... ] // gadget1: pop rdi; ret → sets argument // gadget2: pop rsi; ret → sets argument // gadget3: call puts@plt → calls library fn // No new code injected — only reuse existing code!
  • Requires: stack control + knowledge of gadget addresses
  • Defeated by: ASLR (unknown addresses) + CFI (Control Flow Integrity)
  • Tools: ROPgadget, pwntools for building chains
Social Engineering
  • Phishing — fraudulent email/website impersonates trusted entity to steal credentials
  • Spear phishing — targeted phishing with personalized details (from OSINT)
  • Pretexting — fabricated scenario to manipulate victim (e.g., "IT support" calling for password)
  • Vishing — voice phishing via phone call
  • Baiting — leaving malicious USB drives in parking lots
80%+ of breaches involve phishing or stolen credentials — the human is the weakest link
Format String Vulnerability
// Vulnerable:
printf(user_input);          // user controls format string!
// Attacker inputs: "%x %x %x %n"
// %x reads stack values (information leak)
// %n writes to arbitrary address!

// Safe:

TLS Handshake (TLS 1.3)
  1. ClientHello — client sends supported cipher suites, TLS version, key_share (ECDH public value), random nonce
  2. ServerHello — server selects cipher suite, sends its key_share (ECDH public value)
  3. Shared Secret Derived — both sides independently compute ECDH shared secret and derive session keys via HKDF
  4. Certificate + CertificateVerify — server sends X.509 cert; signs handshake transcript with private key to prove possession
  5. Client verifies — checks cert chain against trusted CAs, verifies signature
  6. Finished — both sides send MAC over full handshake transcript; session is established. All data now encrypted with symmetric keys.
TLS 1.3 provides: Confidentiality → AES-GCM / ChaCha20-Poly1305 Integrity → AEAD authentication tag Authentication → X.509 certificate + signature Forward Secrecy → ephemeral ECDHE (new key per session)
TLS 1.3 Removes Weak Ciphers
TLS 1.3 removed: RC4, 3DES, MD5/SHA-1 in HMACs, RSA static key exchange (no forward secrecy), and export-grade ciphers. Only AEAD ciphers allowed.
Forward Secrecy
Why ECDHE?
Ephemeral DH generates a new key pair per session. Even if the server's long-term private key is later stolen, past session recordings cannot be decrypted — each session had its own ephemeral key.

Trusted Computing & TPM

What is a TPM?
A Trusted Platform Module is a dedicated security hardware chip (or firmware equivalent) on the motherboard. It provides: secure key storage, hardware random number generation, cryptographic operations, and platform state measurement.
Platform Configuration Registers (PCRs)
  • TPM contains 24 (or more) PCR registers (256-bit each)
  • PCRs can only be extended — never directly overwritten: PCR_new = H(PCR_old || new_value)
  • This creates a tamper-evident hash chain of measurements
  • PCR values represent a cryptographic fingerprint of the software boot chain
PCR_extend(i, value): PCR[i] = SHA256( PCR[i] || value ) // initial value: PCR[i] = 0x000...000
Secure Boot
  • UEFI firmware checks digital signature of bootloader before executing it
  • Bootloader must be signed by a trusted CA (Microsoft CA for Windows)
  • Prevents loading of unauthorized OS or malicious bootloaders (bootkits)
  • Linux: signed by distro key or user's enrolled MOK (Machine Owner Key)
Remote Attestation Flow
  1. Boot: BIOS/UEFI measures itself → extends PCR[0]
  2. BIOS measures bootloader → extends PCR[4]
  3. Bootloader measures kernel → extends PCR[8]
  4. Kernel measures initrd, modules → extends further PCRs
  5. Attestation: challenger sends random nonce N to system
  6. TPM signs (PCR_values || N) with AIK (Attestation Identity Key) — private key sealed inside TPM
  7. Challenger receives quote, verifies signature with AIK public key
  8. Challenger compares PCR values to expected "golden" values — if match, platform is trusted
Why the Nonce?
The nonce (fresh random value from challenger) prevents replay attacks — an attacker cannot present a previously recorded valid quote as if it were fresh.

BitLocker & TPM
Windows BitLocker seals the disk encryption key to TPM PCR values. If the boot chain is modified (e.g., malicious bootloader), PCRs change → TPM refuses to unseal → disk remains encrypted even if stolen.
