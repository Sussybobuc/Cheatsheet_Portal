Key Distinction
UMA and NUMA both expose a single shared address space (any CPU can load/store any address). Multicomputers do not — each node has a private address space; data moves only via explicit messages.

Cache Coherence Problem

Each CPU has its own private L1/L2 cache. When multiple CPUs cache the same memory location and one writes to it, the others may read stale (old) data.

The Problem Scenario
// Shared variable X = 5 in RAM
// CPU0 reads X → cache0: X=5  (cache hit next time)
// CPU1 writes X = 10 → RAM: X=10
// CPU0 reads X → cache0: X=5  ← STALE! Wrong value
Solution 1 — Write-Invalidate
  • When a CPU writes to a cache line, broadcast an invalidation message to all other caches
  • Other caches mark their copy as INVALID
  • On next read, those CPUs get a cache miss → fetch updated value from RAM or the writing cache
  • Only one invalidation message per write event (efficient for write-once, read-many)
Solution 2 — Write-Update (Broadcast)
  • When a CPU writes, broadcast the new value to all caches that hold a copy
  • All caches immediately update to the new value — no cache miss on next read
  • Generates much more bus traffic (value sent on every write, not just invalidation signal)
  • Worse for write-many workloads
⚠ Dominant Approach
Write-invalidate dominates all modern hardware. Protocols: MSI, MESI, MOESI, MESIF. Write-update is largely abandoned due to excessive bus traffic.
Snooping vs Directory-Based
MethodMechanismScale
SnoopingAll caches watch (snoop) shared bus for coherence messagesSmall N (≤ ~16 CPUs). Bus is bottleneck.
DirectoryCentral or distributed directory tracks which caches hold each lineLarge N (many sockets, NUMA). Scales to 1000s of nodes.

Extended Protocols
  • MSI — Simpler 3-state: no Exclusive state. Every clean line is Shared, even if only one cache holds it. More bus traffic (S→M requires broadcast even with sole owner).
  • MOESI — Adds O (Owned): a modified line can be shared in dirty state. Saves write-back to RAM before sharing — the Owned cache supplies data to others directly. Used in AMD processors.
  • MESIF — Adds F (Forward): one Shared copy is designated as the responder to peer requests. Used in Intel processors.
💡 E State Optimization
The Exclusive state is key: if CPU has sole ownership (E), a write upgrades to Modified (M) silently — no bus broadcast needed. This avoids unnecessary bus traffic for loads followed by stores.
False Sharing Problem
Two CPUs write to different variables that share the same cache line. The coherence protocol sees it as a conflict and continuously invalidates the line → severe performance degradation. Fix: pad structs so hot variables are on separate cache lines (alignas(64)).

Memory Consistency Models

Defines what value a read is allowed to return — determines visible ordering of memory ops across CPUs.

Sequential Consistency (SC)
  • All memory operations appear to execute in some global total order
  • All CPUs observe the same order of operations
  • Each CPU's operations appear in program order within that global order
  • Simple to reason about; very expensive to implement in hardware
Total Store Order (TSO) — x86
  • Stores can be buffered in a store buffer before becoming visible to other CPUs
  • A CPU sees its own stores immediately (local forwarding)
  • Other CPUs may see stores delayed — load can overtake an earlier store
  • Write-to-Read reordering allowed; all other orders preserved
Weak / Release Consistency
  • Weak Consistency: only synchronization ops must be sequentially consistent; regular reads/writes freely reordered by hardware
  • Release Consistency: distinguish acquire (before critical section) and release (after). Release ensures all prior writes are visible before unlock. Used in ARM, RISC-V.
  • Acquire: no load/store after acquire can move before it
  • Release: no load/store before release can move after it
Memory Barriers Required
Modern CPUs use relaxed consistency for performance. Programmers must insert explicit barriers: mfence (full fence, x86), sfence (store fence), lfence (load fence), DMB/DSB (ARM).

Interconnect Topologies

The network connecting CPUs, memory modules, and I/O determines scalability, bandwidth, and latency of the system.

Bus (Shared Medium)
  • All CPUs share a single electrical bus. Simple and cheap.
  • Only one transfer at a time → bottleneck at high CPU counts
  • Snooping-based coherence works naturally (all CPUs hear every transaction)
  • Practical limit: ~4–16 CPUs before saturation
Crossbar Switch
  • Dedicated wire between every CPU–memory pair: O(N²) switches
  • Non-blocking: any CPU can access any memory simultaneously
  • Expensive for large N; excellent for small N (e.g., 4×4, 8×8)
Omega / Butterfly Network
  • O(N log N) switching elements; multiple paths available
  • Blocking network: two messages may conflict on same switch
  • Used in older mainframes and research parallel computers
Fat Tree (Hierarchical)
  • Tree topology with increasing bandwidth toward the root
  • Eliminates root bottleneck: higher-level switches have more ports/bandwidth
  • Used in InfiniBand-based HPC clusters and modern datacenter fabrics
  • Bisection bandwidth scales well with N
Hypercube
  • n-dimensional binary cube: N = 2ⁿ nodes, diameter = n = log₂N
  • Each node has exactly n neighbors (degree = n)
  • Good for nearest-neighbor algorithms; used in 1980s–1990s supercomputers (nCUBE, Intel iPSC)
Topology Comparison

Multiprocessor Scheduling

Unlike uniprocessors, the OS must assign processes/threads to which CPU in addition to when to run them.

Time Sharing (Multiprogramming on SMP)
  • OS maintains a global run queue; any idle CPU picks the next ready thread
  • Simple extension of uniprocessor scheduling — threads migrate across CPUs freely
  • Problem: migration causes cache warming cost (cold cache on new CPU)
  • Linux CFS uses per-CPU run queues with load balancing to reduce migration
Space Sharing (Partition Scheduling)
  • Divide CPUs into partitions — each job gets dedicated CPUs for its lifetime
  • No preemption within a partition; job runs until completion
  • Eliminates context-switch overhead; poor CPU utilization if jobs are uneven
  • Used in batch HPC environments (PBS, SLURM job scheduling)
Gang Scheduling (Co-scheduling)
  • All threads of a job are scheduled simultaneously across N CPUs
  • Each time slice: a job's threads run together; then another job's threads run together
  • Critical for tightly coupled parallel programs where threads communicate via shared memory barriers or spin-locks
  • A blocked thread on one CPU wastes the spin-wait of all peer threads if not co-scheduled
💡 Why Gang Scheduling Matters for MPI
MPI processes communicate via message passing. If Process 0 sends to Process 1 but Process 1 is de-scheduled, Process 0 blocks waiting for an ACK. Gang scheduling ensures all N processes run simultaneously, preventing convoy delays.
Scheduling Metric: Spin vs Block
  • If a lock wait is expected to be short → spin (busy-wait, avoid context switch)
  • If wait is expected to be long → block (yield CPU, allow other work)
  • Adaptive mutexes: spin briefly, then block (Linux futex, pthreads)
Gang scheduling requires time-slice synchronization across all CPUs — all CPUs switch jobs at the same instant. Overhead: synchronization interrupt at each quantum boundary.

NUMA-Aware Scheduling & Memory Placement

On NUMA systems, where memory is allocated and which CPU runs the thread dramatically affects performance.

Process Affinity
  • CPU affinity: pin a process/thread to a specific CPU or set of CPUs to prevent migration
  • Keeps warm cache data valid — avoids cold-cache penalty after migration
  • Linux: sched_setaffinity() syscall, taskset command
  • Windows: SetProcessAffinityMask(), SetThreadAffinityMask()
Memory Locality (NUMA Placement Policy)
  • First-touch policy (Linux default): physical page allocated on the NUMA node of the CPU that first touches it
  • If allocation happens in main thread (node 0), all pages land on node 0 → remote access for threads on other nodes
  • Fix: initialize data in the thread that will use it, or use numactl --localalloc
NUMA Latency Ratio
Typical ratio: local DRAM access ≈ 4 ns. Remote (cross-socket) ≈ 10–14 ns. Ratio ≈ 2–3× slower for remote. At scale, this difference drives 20–40% throughput gap for memory-bound workloads.
Linux NUMA Tools
# Bind process to NUMA node 0, allocate memory locally
numactl --cpunodebind=0 --membind=0 ./myprogram

# Show NUMA topology
numactl --hardware

# Pin process to specific CPUs (CPU 0-3)
taskset -c 0-3 ./myprogram

# Check NUMA stats
numastat -p myprogram

# View NUMA node distances
cat /sys/devices/system/node/node0/distance
NUMA-Aware Allocation in C
#include <numa.h>

// Allocate 1MB on NUMA node 1
void *buf = numa_alloc_onnode(1024*1024, 1);

// Allocate locally (on current CPU's node)
void *buf2 = numa_alloc_local(size);

// Move pages to preferred node
numa_migrate_pages(pid, from_nodemask, to_nodemask);
💡 NUMA + Thread Pool
Best practice: create one thread pool per NUMA node, each pool serves requests using memory allocated on its own node. Used in databases (PostgreSQL, MongoDB) and JVMs (G1GC NUMA-aware).

Multicomputer Architecture — Message Passing Systems

Multicomputers are physically separate machines connected by a high-speed interconnect (InfiniBand, Ethernet). No shared memory — all communication is explicit message passing.

Network Switching Modes
ModeHow it worksLatency
Store-and-ForwardEach switch receives full packet, buffers it, then forwardsProportional to #hops × packet size
Circuit SwitchingPre-establish path before transmission; no buffering en routeSetup latency + transfer time
Wormhole RoutingForward head flit as soon as routing decision made; body follows in pipelineNear-constant (independent of distance)
Network Interface Board (NIC)
  • Dedicated hardware with its own RAM for packet buffering
  • DMA engine: transfers data between host RAM and NIC RAM without CPU involvement
  • RDMA (Remote Direct Memory Access): NIC can read/write remote node's memory directly, bypassing the remote OS kernel
  • Latency with RDMA: ~1–2 µs vs ~50–100 µs with kernel-mediated send/recv
Basic Send/Receive Primitives
// Blocking send: CPU waits until message delivered
send(dest_addr, &msg_ptr, msg_len);

// Blocking receive: CPU waits until message arrives
receive(src_addr, &buf_ptr, &msg_len);

// Non-blocking send: returns immediately
// buffer must NOT be modified until confirmed sent!
nb_send(dest_addr, &msg_ptr, msg_len, &handle);
wait_for_completion(&handle);
⚠ Non-Blocking Send Hazard
After a non-blocking send, modifying the send buffer before the OS copies it leads to corrupted messages. Solutions: (1) OS copies to kernel buffer immediately, (2) use interrupt/callback when safe to reuse buffer.
RDMA Operation Flow
Node A (sender) Node B (target) ────────────────── ────────────────── app posts RDMA Write ──────────► NIC writes to B's RAM NIC handles transfer (no CPU interrupt on B) app gets completion B's app reads data later (no kernel involvement) Latency: ~1 µs (InfiniBand)
Blocking vs Non-Blocking
TypeCPU during sendBuffer safe after call?
Blocking sendWaits (spins or blocks)Yes — message delivered
Non-blocking send (copy)Returns immediatelyYes — OS copied buffer
Non-blocking send (no copy)Returns immediatelyNO — wait for completion

Message Passing Challenges
  • Buffering: if receiver isn't ready, where does the message go? Sender-side buffer or receiver-side buffer?
  • Flow control: prevent fast sender from overwhelming slow receiver
  • Partial failures: node crash mid-transfer — message lost? delivered? duplicated?
  • Ordering: FIFO ordering per pair vs global ordering vs no ordering
InfiniBand vs Ethernet
InfiniBand: ~1 µs latency, 200–400 Gbps, native RDMA, lossless fabric. Used in HPC.
Ethernet (RoCE): RDMA over Converged Ethernet — brings RDMA to commodity networks. More common in datacenter AI/ML clusters.

Remote Procedure Call (RPC)

RPC makes a remote call look like a local procedure call — the programmer writes result = add(a, b) and the infrastructure transparently handles the network communication.

Core Concept
  • Client stub: local proxy that looks like the real function. Marshals parameters and sends message over network.
  • Server stub: receives message, unmarshals parameters, calls the actual function, marshals the return value, sends reply.
  • Marshaling (serialization): converting parameters (int, structs, pointers) into a flat byte stream suitable for network transmission.
  • Unmarshaling (deserialization): reconstructing the original data types from the byte stream on the remote side.
RPC Call Flow
  1. Client calls stub_add(a, b) — looks like a normal function call
  2. Client stub marshals a and b into a byte buffer (handles endianness, alignment)
  3. Client stub sends the message to the server via the transport layer (TCP/UDP/IB)
  4. Server OS receives message and wakes up the server stub
  5. Server stub unmarshals the parameters from the byte buffer
  6. Server stub calls the actual add(a, b) function on the server
  7. Server stub marshals the return value and sends reply back to client
  8. Client stub receives reply, unmarshals return value, returns it to caller
Marshaling Challenges
ProblemCauseSolution
Endiannessx86 is little-endian; network is big-endianAlways convert to network byte order (htonl/ntohl)
Pointer semanticsPointers are address-space specific — meaningless remotelyMust deep-copy pointed-to data; no raw pointer transmission
Struct alignment/paddingCompiler adds padding bytes for alignmentUse packed structs or IDL-generated serializers
Variable-length dataStrings, arrays have unknown size at compile timePrefix with length field
Failure Semantics
  • Exactly-once: ideal but hard — requires idempotency + dedup log
  • At-most-once: server records call ID; re-sends cached reply if duplicate detected. Used by most RPC frameworks.
  • At-least-once: client retries on timeout. Safe only for idempotent operations (e.g., read, SET same value).
⚠ Partial Failure Problem
If client times out, it cannot tell: (a) request was lost before server received it, (b) server crashed during execution, (c) reply was lost after server finished. RPC fundamentally cannot distinguish these cases without application-level acknowledgment.
Modern RPC Frameworks
  • gRPC (Google): Protocol Buffers + HTTP/2. Language-agnostic IDL. Used widely in microservices.
  • Thrift (Facebook/Apache): IDL-based, supports many languages and transports.
  • XML-RPC / SOAP: older, verbose XML-based. Still in enterprise systems.
  • JSON-RPC: lightweight JSON format, common in web APIs.

MPI Basics — Message Passing Interface

MPI is the de-facto standard API for writing distributed-memory parallel programs. Programs run as N independent processes, each with its own address space, communicating via MPI_Send/MPI_Recv and collective operations.

Core Concepts
  • Rank: unique ID of each process (0 to N-1) within a communicator
  • Communicator: group of processes that can communicate (MPI_COMM_WORLD = all processes)
  • Tag: integer label on a message for selective receive
  • Collective: operation involving all processes in a communicator simultaneously
Point-to-Point Communication
// Send 100 ints to process 1, tag=0
MPI_Send(buf, 100, MPI_INT, 1, 0, MPI_COMM_WORLD);

// Receive from process 0, tag=0
MPI_Recv(buf, 100, MPI_INT, 0, 0,
         MPI_COMM_WORLD, &status);

// Non-blocking versions
MPI_Isend(buf, n, MPI_INT, dest, tag, comm, &req);
MPI_Irecv(buf, n, MPI_INT, src, tag, comm, &req);
MPI_Wait(&req, &status); // wait for completion
Collective Operations
// Broadcast: root sends to all
MPI_Bcast(buf, 100, MPI_INT, 0, MPI_COMM_WORLD);

// Reduce: all send, root collects with op
MPI_Reduce(sendbuf, recvbuf, 100, MPI_INT,
           MPI_SUM, 0, MPI_COMM_WORLD);

// All-Reduce: all get the result
MPI_Allreduce(sendbuf, recvbuf, 100, MPI_INT,
              MPI_SUM, MPI_COMM_WORLD);

// Scatter: root distributes chunks to all
MPI_Scatter(sendbuf, n, MPI_INT,
            recvbuf, n, MPI_INT, 0, comm);

// Gather: all send, root collects
MPI_Gather(sendbuf, n, MPI_INT,
           recvbuf, n, MPI_INT, 0, comm);

// Barrier: all processes synchronize here
MPI_Barrier(MPI_COMM_WORLD);
Complete MPI Hello World
#include <mpi.h>
#include <stdio.h>

int main(int argc, char **argv) {
    int rank, size;
    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    if (rank == 0) {
        // Master: send work to workers
        for (int i = 1; i < size; i++) {
            int work = i * 10;
            MPI_Send(&work, 1, MPI_INT, i,
                     0, MPI_COMM_WORLD);
        }
    } else {
        // Worker: receive and process
        int work;
        MPI_Recv(&work, 1, MPI_INT, 0, 0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        printf("Rank %d got work=%d\n", rank, work);
    }

    MPI_Finalize();
    return 0;
}
Running MPI Programs
# Compile
mpicc -o myprogram myprogram.c

# Run on 4 processes locally
mpirun -np 4 ./myprogram

# Run on cluster (SLURM)
srun --ntasks=64 --nodes=8 ./myprogram

⚠ Deadlock in MPI
If all processes call blocking MPI_Send at the same time (circular send), and no one is receiving, all block forever. Fix: use MPI_Sendrecv, non-blocking MPI_Isend/MPI_Irecv, or alternate send/recv order between even/odd ranks.

Quick Reference — Key Facts & Exam Traps

Architecture Quick Facts
UMA = SMP = Symmetric Multiprocessor. All CPUs equal. Single shared bus or crossbar.
NUMA: "Non-Uniform" → local memory faster. Must manage thread-memory affinity explicitly.
Multicomputer: NO shared address space. Every byte of communication is explicit send/recv.
MESI E→M transition: NO bus traffic needed (only this cache has the line).
MESI S→M transition: Must broadcast invalidate to all caches sharing the line.
False sharing: different variables, same cache line → unnecessary coherence traffic.
Sequential Consistency: strongest model; most expensive; modern HW doesn't implement it.
x86 uses TSO: stores can be delayed; loads can pass pending stores from other CPUs.
Scheduling Quick Facts
Time sharing: threads migrate between CPUs freely. Simple but cold-cache cost.
Space sharing: dedicated CPUs per job. No migration. Used in HPC batch systems.
Gang scheduling: all threads of a job run simultaneously. Required for spin-based synchronization.
NUMA first-touch: page allocated on node of CPU that first touches it. Init in the right thread!
numactl --localalloc: all future allocations go to local NUMA node of running CPU.
RPC / Message Passing Quick Facts
RPC stub hides network: client stub marshals; server stub unmarshals. Transparent to programmer.
Marshaling must handle: endianness, pointer dereferencing, struct padding, variable-length data.
Wormhole routing: latency nearly independent of number of hops. Used in modern interconnects.
MPI Patterns
Master-Worker: rank 0 → distributes tasks via MPI_Send rank 1..N → receive task, compute, MPI_Send result back rank 0 → collect results SPMD (Single Program, Multiple Data): all ranks run same code; branch on rank collective ops (Bcast, Reduce) for coordination Pipeline: rank i → processes data, sends to rank i+1 MPI_Send(to i+1) + MPI_Recv(from i-1)

💡 Exam Tip — Coherence vs Consistency
Cache coherence: all CPUs eventually see the same value for a single memory location.
Memory consistency: defines the ordering of operations across different memory locations. These are independent concerns!
