What is Deadlock?

A deadlock is a situation where a set of processes is permanently blocked: each process waits for an event (usually resource release) that only another process in the same set can cause.

⚠ Key Property
No process in the deadlocked set can run, release resources, or be awakened. The system is stuck indefinitely with no way out unless the OS intervenes.
Two Types of Deadlock
  • Resource Deadlock — processes waiting for hardware/software resources (mutex, disk, printer) held by other processes in the set
  • Communication Deadlock — processes waiting for messages from each other while the channel is empty; no resource involved
Commonly Deadlocked Resources
  • Sync primitives: mutex, semaphore, lock, condition_variable
  • Hardware: printers, tape drives, disk controllers, network cards
  • Software objects: database records, file locks, memory regions
📌 Historical Note
Deadlock was first formally described by Dijkstra in 1965 in the context of THE operating system. The term "deadly embrace" was also used early on.

Four Coffman Conditions — ALL Must Hold Simultaneously

⚡ Break ANY ONE condition → Deadlock is structurally impossible. All four together = deadlock can occur.
  1. Mutual Exclusion — A resource can be held by at most one process at a time. Other processes requesting it must wait until it is released.
    Example: mutex held by thread A; thread B blocks on pthread_mutex_lock(). A printer belongs to exactly one print job.
  2. Hold and Wait — A process is holding at least one resource while simultaneously waiting for additional resources currently held by other processes.
    P1 holds R1 and wants R2; P2 holds R2 and wants R1. Neither releases what it has.
  3. No Preemption — Resources cannot be forcibly taken away from a holding process. A resource is released only voluntarily by the process after it has finished using it.
    The OS cannot simply revoke a mutex or pull a file lock from a running thread — doing so would corrupt program state.
  4. Circular Wait — A circular chain exists: P1 waits for R1 held by P2, P2 waits for R2 held by P3, …, Pn waits for Rn held by P1. A closed loop with no exit.
    Formally: ∃ {P0,P1,...,Pn} such that Pi waits for resource held by P(i+1 mod n+1).
💡 Prevention Strategy
Design systems that structurally eliminate at least one condition at design time. This guarantees no deadlock — unlike avoidance or detection which react at runtime.
ℹ Coffman, Elphick & Shoshani, 1971
These four conditions were formalized in their 1971 ACM Computing Surveys paper. They remain the canonical framework for deadlock analysis in every OS textbook.

RAG Components

The Resource Allocation Graph (RAG) is a directed bipartite graph that precisely captures the resource allocation state of the system.

Node Types
  • Process nodes (circles ○) — one circle per process Pi in the system
  • Resource nodes (rectangles □) — one rectangle per resource type Rj; dots inside the rectangle represent individual instances of that resource type
Edge Types
  • Request edge P→R — process P is requesting (waiting for) resource type R; dashed or directed arrow from process to resource
  • Assignment edge R→P — one instance of resource R is currently assigned to process P; directed arrow from resource to process
Degree of resource node Rj: in-degree = # processes waiting for Rj out-degree = # instances currently allocated
📌 Reading the Graph
Count dots in resource rectangle = total instances. Count outgoing edges from R = allocated instances. Available = total − allocated.

Deadlock Detection via RAG

Single Instance Resources (One Dot per Rectangle)
Cycle in RAG ⟺ DEADLOCK (necessary AND sufficient)
  • Use DFS to search for back edges (cycles) in the RAG
  • If a cycle is found: every process in the cycle is deadlocked
  • Algorithm complexity: O(n²) where n = total number of nodes
Multiple Instance Resources (Multiple Dots)
⚠ Important Distinction
A cycle is necessary but NOT sufficient for deadlock. A cycle may exist yet no deadlock if other non-cycle processes can release instances needed by cycle processes.
  • No cycle → No deadlock (guaranteed; cycle is necessary)
  • Cycle exists → Deadlock might exist; must run matrix-based detection algorithm to confirm
💡 Reduction Technique
Iteratively remove edges for processes that can be fully satisfied from available resources. If graph reduces completely → no deadlock.

R1 has 2 instances:

  R1 → P2  (one instance to P2)
  P3 → R1  (P3 waiting for R1)

R1(1 inst)→P2, P1→R1, P2→R2
R2(1 inst)→P3, P3→R2
→ P1,P2,P3 all blocked. No escape.
→ DEADLOCK confirmed.

Attack Mutual Exclusion

Eliminate the requirement that resources be held exclusively. Make resources sharable wherever technically possible.

Techniques
  • Read-only files — multiple processes can read simultaneously; no exclusion needed; only write access needs protection
  • Spooling — instead of processes accessing a printer directly, all print requests go to a spooler queue. Only the spooler daemon holds the printer; user processes never hold it directly
  • Virtualization — abstract physical devices (via hypervisors or device drivers) so multiple users share them safely through multiplexing
  • Readers-Writer Lock — pthread_rwlock_t allows concurrent reads, exclusive writes — reduces mutual exclusion scope
⚠ Hard Limit
Some resources are inherently non-sharable — a mutex by definition, a physical tape drive mid-write, a network socket in connected state. This approach cannot always be applied.
📌 Practical Impact
This is the hardest Coffman condition to eliminate in practice. Hardware and correctness constraints often mandate mutual exclusion.

Attack Hold and Wait

Ensure that a process never holds one resource while requesting another.

Strategy 1 — Request Everything Upfront
Before execution begins, process must request ALL resources it will ever need at once. Granted only when ALL are simultaneously free. If any unavailable → process waits holding NONE.
Strategy 2 — Release Before Requesting
When process needs new resources: 1. Release ALL currently held resources 2. Request the new full set atomically 3. Restart or continue only when all granted
Drawbacks
  • Resource underutilization — holding tape drive for hours while only using it for 5 minutes at end
  • Starvation risk — process needing many popular resources may wait forever while others keep acquiring them piecemeal
  • Impractical declaration — many programs cannot know all needed resources in advance (interactive programs, database queries)
ℹ Related: Two-Phase Locking
2PL is a practical variant of this strategy used in databases and OS thread libraries. Details in Section 8.

Attack No Preemption & Circular Wait

Attack No Preemption

Allow the OS to forcibly remove resources from processes under specific conditions:

  1. Process Pi holds some resources and requests Rj (not immediately available)
  2. OS preempts: Pi must release ALL currently held resources
  3. Pi is added to the wait queue for all resources it needs (old + new)
  4. Pi restarts only when ALL requested resources are simultaneously available
⚠ Feasibility Constraint
Only works for resources whose state can be saved/restored (CPU registers, memory pages). Not applicable to mid-operation I/O (printer, tape drive), network connections, or non-idempotent operations.
Attack Circular Wait — Global Resource Ordering
Assign unique integer to each resource type. Processes MUST request resources in strictly increasing numerical order ONLY.
  • tape=1, disk=2, printer=3, plotter=4 — must always acquire tape before disk before printer
  • If process holds R5 and needs R3: must release R5, acquire R3, then re-acquire R5
  • Prevents circular chain because you can never have Pi waiting for lower-numbered resource than it holds
💡 Why No Cycle is Possible
In a circular chain P1→R1→P2→R2→P1, P1 holds R(high) waiting for R(low) and P2 holds R(low) waiting for R(high). But ordering rule requires P1 to always request R(low) before R(high) — contradiction → no cycle possible.

Safe State vs. Unsafe State — Deadlock Avoidance Foundation

Safe State

A state is SAFE if there exists at least one safe sequence of process execution ⟨P1, P2, …, Pn⟩ such that every Pi can obtain all its maximum needed resources using currently available resources plus resources released by all Pj where j < i (processes that finished before Pi).

Safe State: ∃ safe sequence ⟨P₁, P₂, ..., Pₙ⟩ where ∀ i: Need[i] ≤ Available + Σ Allocation[j] (j=1..i-1) → Pi can finish; its resources free up for Pi+1 → All processes eventually complete
💡 Key Guarantee
In a safe state, the OS can always find an execution ordering that lets every process complete — even if some must temporarily wait for resources.
Unsafe State

A state is UNSAFE if no safe sequence exists. The OS cannot guarantee that all processes will eventually complete.

⚠ Critical Distinction
Unsafe ≠ Deadlock. An unsafe state does NOT mean deadlock has already occurred. It means deadlock is possible depending on future resource requests. Processes might still finish if they use fewer resources than their declared maximum.
State Transition Model
SAFE ──(bad grant)──→ UNSAFE UNSAFE ──(bad requests)──→ DEADLOCK Avoidance Rule: Never grant any resource request that would transition system into unsafe state. Make request WAIT instead.
📌 Avoidance Algorithm
On each resource request: tentatively grant it, run safety algorithm on resulting state. If safe → commit grant. If unsafe → deny, process waits. This is the Banker's Algorithm.

Prevention is cheapest at runtime but most restrictive (reduces parallelism). Detection allows maximum concurrency but requires costly recovery. Avoidance is a middle ground but requires advance max declarations.

Avoidance Requirements & Request Flow

Prerequisites for Avoidance
  • Each process must declare maximum resource needs upfront before execution (Max[i] matrix)
  • OS maintains Allocation, Need, and Available at all times — updated on every grant/release
  • Number of processes and resource types must be fixed (static system) — dynamic creation complicates this
  • Processes must not lie about their maximum needs (trusted declarations)
Per-Request Decision Flow
  1. Receive request Request[i] from Pi
  2. Verify Request[i] ≤ Need[i] — sanity check (error if exceeded)
  3. Verify Request[i] ≤ Available — resources physically exist
  4. Tentatively update: Available−=Req, Alloc[i]+=Req, Need[i]−=Req
  5. Run safety algorithm on tentative state
  6. SAFE → commit changes, Pi gets resources. UNSAFE → rollback, Pi waits.

Banker's Algorithm — Data Structures

Dijkstra's Banker's Algorithm (1965) is the classic deadlock avoidance algorithm. The name refers to a bank that only grants loans (resources) if it can still service all customers (processes).

System Parameters
  • n = number of processes in the system
  • m = number of resource types (distinct resource classes)
Matrices — size n × m
  • Allocation[n][m] — resources currently held by each process (known, tracked by OS)
  • Max[n][m] — maximum resources each process may ever simultaneously need (declared by process)
  • Need[n][m] = Max[n][m] − Allocation[n][m] — remaining potential need of each process
Vectors — size 1 × m
  • Available[m] — currently free instances of each resource type (can be granted immediately)
  • E[m] (Existing) — total instances of each resource type in the system
Key invariant: Available[j] = E[j] − Σ Allocation[i][j] for all i
Need[i][j] = Max[i][j] − Allocation[i][j] ∀ i,j: Need[i][j] ≥ 0 (can't need negative resources) ∀ j: Σ Allocation[i][j] ≤ E[j] (can't allocate more than exist)

Safety Algorithm

Determines whether the current allocation state is safe by simulating process completion in some order.

Initialization
  • Work[m] = Available[m] — copy of available resources (simulated free pool)
  • Finish[n] = {false, false, ..., false} — all processes unfinished initially
Algorithm (repeat until no progress)
  1. Find any i such that: Finish[i] == false AND Need[i] ≤ Work (Pi can obtain all remaining needed resources right now)
  2. Simulate Pi finishing: Work += Allocation[i] (Pi releases its resources); set Finish[i] = true
  3. Go back to step 1. Repeat until no unfinished process satisfies Need[i] ≤ Work
  4. If all Finish[i] == true → state is SAFE; the safe sequence is the order Pi were completed. Otherwise → UNSAFE
Time complexity: O(n² × m) Outer loop: at most n iterations (one per process finish) Inner search: O(n × m) to find candidate process
⚠ Conservative Estimate
Safety algorithm assumes each Pi will eventually request up to its full Need[i]. In reality processes may request less — so a safe state may be more conservative than necessary.

Resource-Request Algorithm

Called when process Pi issues Request_i[j] — requesting j instances of resource type j.

Steps
  1. Sanity check: If Request[i] > Need[i] → raise error (process claimed more than its declared maximum)
  2. Availability check: If Request[i] > Available → Pi must wait (insufficient free resources right now)
  3. Tentative allocation (pretend-grant):
    Available -= Request[i] /* reduce free pool */ Allocation[i] += Request[i] /* add to Pi's held */ Need[i] -= Request[i] /* reduce Pi's need */
  4. Safety check: Run full safety algorithm on the tentative state
  5. If SAFE: commit the tentative changes — Pi actually gets the resources. If UNSAFE: rollback all three changes from step 3, Pi waits.
💡 Rollback is Trivial
Since we only tentatively modified three values, rollback is: undo the three subtractions/additions. No complex state restoration needed.
📌 Practical Limitation
Requires processes to accurately declare Max upfront. Impractical for interactive programs, shells, compilers — anything with data-dependent resource usage.

Single Resource Type Detection

When each resource type has exactly one instance, deadlock detection reduces to finding cycles in a simplified graph.

Wait-For Graph Construction

Simplify the RAG by removing all resource nodes. Add a direct edge Pi→Pj if Pi is waiting for a resource currently held by Pj.

RAG edge: Pi→Rk and Rk→Pj ⟹ Wait-For edge: Pi→Pj (Pi is waiting for Pj to release Rk)
Cycle in Wait-For Graph ⟺ DEADLOCK (single-instance resources)
DFS Cycle Detection Algorithm
  1. Maintain visited[n] (globally visited) and recStack[n] (current DFS path stack)
  2. For each unvisited process Pi, start DFS from Pi
  3. For each neighbor Pj of Pi: if Pj is in recStack → cycle found → deadlock!
  4. On DFS return from Pi: remove Pi from recStack
Time complexity: O(n²) n = number of processes Each node visited at most once; each edge examined once

Multiple Resource Types Detection

Similar to Banker's safety algorithm but uses actual current requests (not maximum need declarations). No advance declarations required.

Data Structures
  • E[m] — Existing total resources vector (fixed)
  • A[m] — Available (free) resources vector (changes as allocations change)
  • C[n][m] — Current Allocation matrix (what each process holds right now)
  • R[n][m] — Request matrix (what each process is currently blocked waiting for)
Detection Algorithm
  1. Initialize: Work = A. Mark Finish[i] = true for processes with zero allocation (can't be in deadlock if holding nothing)
  2. Find i with Finish[i] == false AND R[i] ≤ Work (Pi's current requests can be satisfied)
  3. If found: Work += C[i] (Pi finishes, releases resources); Finish[i] = true; goto step 2
  4. If no such process found: all processes with Finish[i] == false are in deadlock
💡 vs. Banker's Safety Algorithm
Detection uses R[i] (actual current requests) instead of Need[i] (max remaining need). This makes detection more optimistic — a process is only assumed to need what it's currently blocked on, not its theoretical maximum.

Detection Overhead & Timing Policy

Running detection too frequently wastes CPU; running too infrequently delays recovery and allows deadlocked processes to consume resources unnecessarily.

📌 Practical Heuristic
Most production systems trigger detection when CPU utilization falls below ~40% for a sustained period — suggesting processes might be blocked. Combined with monitoring of blocked-process count for better accuracy.
⚠ Attribution Problem
When deadlock involves many processes, it's hard to determine which to kill. The longer detection waits, the more processes may have joined the deadlock, making recovery harder.

Recovery via Process Termination

Break the deadlock by forcibly terminating one or more deadlocked processes, freeing their held resources.

Strategy 1 — Kill All Deadlocked Processes
⚠ Nuclear Option
Abort ALL deadlocked processes simultaneously. Guaranteed to break deadlock immediately. Extremely expensive — all computation since last checkpoint is lost for every terminated process.
Strategy 2 — Incremental Kill (Preferred)

Kill processes one at a time, re-run detection after each kill, stop when deadlock resolves.

Victim Selection Factors (Minimize Recovery Cost)
  • Process priority — prefer killing lowest-priority (least important) processes
  • CPU time consumed — prefer killing processes with least CPU time (minimize wasted work)
  • Resources held — prefer killing processes holding most resources (maximizes freed resources per kill)
  • Remaining time — prefer killing processes with most remaining execution time (least regret)
  • Interactive vs. batch — strongly prefer killing background batch processes over interactive sessions
  • Rollback cost — prefer processes with recent checkpoints (cheaper to restart)
📌 No Perfect Metric
Victim selection typically uses a weighted scoring function combining the above factors. Different OSes and DBMSes use different weight schemes tuned for their workloads.

Recovery via Resource Preemption

Break deadlock by forcibly removing (preempting) resources from deadlocked processes and granting them to others that need them.

Preemption Recovery Steps
  1. Select victim resource and holder — decide which resource to preempt and from which process (minimize total recovery cost: CPU wasted, rollback cost, preemption frequency)
  2. Roll back victim to checkpoint — restore the preempted process to its most recent state snapshot taken before it acquired the contested resource
  3. Grant resource to waiter — allocate the preempted resource to the process that was blocked waiting for it
  4. Restart victim from checkpoint — resume the rolled-back process; it will re-request the resource when it reaches that point again
  5. Monitor for starvation — if same process is always preempted, it will never complete; track preemption count and set upper limit
⚠ Starvation Risk
If the same process is repeatedly chosen as victim (e.g., lowest priority process), it may be rolled back indefinitely and never complete — this is starvation caused by the recovery mechanism itself.
ℹ Feasibility Constraint
Preemption is only feasible for resources whose state is fully capturable: memory pages (swap out), CPU state (context switch), open file positions. Not feasible for printer mid-job, network connection, or non-idempotent I/O.

Recovery via Rollback & Checkpointing

Checkpointing: periodically save the complete state of every process to stable storage so any process can be restarted from a known-good earlier point.

What a Checkpoint Captures
  • CPU register state: program counter PC, stack pointer SP, all general-purpose registers
  • Full memory state: heap, stack, global data segment snapshot
  • Open file descriptors with their current seek positions
  • Pending I/O operations and their states
  • Current resource allocation: which resources held, which requested
  • Network socket states and pending data buffers
Rollback Procedure
  1. Detect deadlock involving process Pi
  2. Identify the most recent checkpoint of Pi taken before it acquired the resource causing the deadlock
  3. Restore Pi's complete state from that checkpoint (memory, registers, file positions)
  4. The released resource(s) are now granted to the waiting process(es)
  5. Pi restarts from checkpoint; may encounter the deadlock again — handled by detection and rollback again
Checkpoint frequency trade-off: Too frequent → high I/O and memory overhead per checkpoint Too infrequent → large amounts of recomputation on rollback Optimal: checkpoint at natural quiescent points (idle, sync)
💡 Database Analogy
Directly analogous to database ROLLBACK TO SAVEPOINT sp1. Many DBMSes (PostgreSQL, Oracle) implement this automatically for transaction deadlock recovery.

Two-Phase Locking (2PL)

Two-Phase Locking is a practical deadlock-prevention protocol widely used in database systems and concurrent OS code. It attacks the Hold-and-Wait condition.

ℹ The Two Phases
Phase 1 — Growing: acquire locks one by one; NEVER release any lock in this phase.
Phase 2 — Shrinking: release locks one by one; NEVER acquire new locks in this phase.
Phase 1 — Growing Phase (Lock Acquisition)
  • Thread attempts to acquire each needed mutex/semaphore one at a time in defined order
  • If any lock acquisition fails (resource unavailable or try-lock timeout): release ALL currently held locks immediately and restart Phase 1 from scratch
  • Phase 1 ends successfully only when ALL needed locks are held simultaneously
Phase 2 — Shrinking Phase (Critical Section + Release)
  • Execute the critical section with full set of locks held
  • Release locks in reverse acquisition order (LIFO)
  • No new locks may be acquired once Phase 2 begins
2PL prevents Hold-and-Wait: process either holds ALL needed locks or NONE. No partial holding while waiting.
⚠ Livelock Risk
If two threads repeatedly acquire some locks, fail, release everything, and retry at the same time, they may livelock. Solution: add random exponential backoff before Phase 1 retry.

Ostrich Algorithm

The Ostrich Algorithm: pretend deadlock cannot happen — ignore the problem entirely (like an ostrich burying its head in the sand when danger approaches).

📌 Rationale
Economically valid when deadlocks are extremely rare and the cumulative cost of prevention/avoidance/detection mechanisms outweighs the occasional cost of a manual recovery (reboot, kill).
When the Ostrich Approach is Justified
  • Deadlock probability is statistically negligible (once per month on average)
  • System can tolerate occasional manual restarts without data integrity issues
  • Prevention overhead would severely degrade normal performance (e.g., require single-threading)
  • Human recovery cost (rebooting, restarting app) is low and acceptable
Real-World Usage
  • Traditional UNIX/Linux — process and inode tables can deadlock; historically handled by rebooting or killing stuck processes
  • Desktop Windows historically — most subsystems use ostrich for rare resource deadlocks
  • Most general-purpose OSes — user kills the "Not Responding" application
⚠ Unacceptable for Critical Systems
Real-time systems, medical devices, aviation control, nuclear plants, financial trading systems — zero-downtime requirements mandate formal deadlock prevention, not ostrich.

Communication Deadlock

Communication Deadlock: processes blocked waiting for messages from each other, with no physical resource involved. The communication channel is empty — messages were lost or never sent.

Classic Scenario
Process A: send(B, request_msg) wait for reply from B (blocks) Process B: send(A, request_msg) wait for reply from A (blocks) → If A's message lost: B waits forever for A's msg → B's message lost: A waits forever for B's msg → Both blocked, channel empty → communication deadlock
Root Causes
  • Message loss — unreliable network or IPC channel drops messages
  • Circular dependency — each process waits for the other to send first (protocol design error)
  • Sequencing errors — processes expect messages in wrong order (e.g., reply before request)
  • Buffer overflow — receive buffer full; sender blocks; receiver also blocks waiting for sender
Solutions
  • Timeouts — if reply not received within T seconds, assume lost; resend or abort with error
  • Explicit ACK protocol — sender retransmits until ACK received; detects lost messages
  • Sequence numbers — detect lost, duplicate, out-of-order messages; enable recovery
  • Heartbeat signals — periodic alive ping; prolonged silence triggers reconnect/recovery
  • Protocol redesign — eliminate circular "you go first" dependencies by designating one party as initiator
ℹ Not Resource Deadlock
Standard RAG-based methods don't apply. Must use protocol-level solutions: timeouts, retransmission, acknowledgments. TCP/IP does this at the transport layer automatically.

Livelock vs. Deadlock

Livelock: processes are not blocked — they are actively executing — but they keep changing state in response to each other's actions without making any real forward progress.

⚠ Key Distinction from Deadlock
Deadlock: processes are blocked — sleeping, not using CPU, waiting for event.
Livelock: processes are running — consuming CPU — but stuck in a futile loop of state changes.
Classic Hallway Analogy
📌 The Hallway Dance
Two people approach each other in a corridor. Both step right to let the other pass — but they step in the same direction, blocking each other again. Both step left — same problem. They keep moving but make no progress. They are ACTIVE (not frozen) but permanently stuck.
Code-Level Example
/* Thread A and Thread B both run this: */
while (true) {
    lock(mutex_A);             /* A and B both grab A */
    if (!trylock(mutex_B)) {   /* both fail to grab B */
        unlock(mutex_A);       /* both release A */
        sleep(backoff);        /* both sleep same time */
        continue;              /* both retry → livelock! */
    }
    /* critical section */
    unlock(mutex_B); unlock(mutex_A);
    break;
}
Solutions
  • Random exponential backoff — each thread waits a random time before retry; probability of collision drops exponentially (Ethernet CSMA/CD)
  • Priority/role asymmetry — designate one thread as "deferor" and one as "proceeder" when conflict occurs
  • Central lock arbiter — a coordinator process decides who gets to proceed
  • Token-based approach — only the thread holding a token may proceed in conflict situations

Starvation

Starvation: a process is indefinitely denied resources or CPU time it needs to proceed — not because of a deadlock cycle, but because the scheduling or allocation policy consistently favors other processes.

Causes of Starvation
  • Pure priority scheduling — if high-priority work is always present, low-priority processes never run
  • SJF without aging — Shortest Job First always prefers shorter jobs; long jobs may wait indefinitely
  • Biased resource allocation — allocation policy consistently chooses same processes (e.g., always grants to requesters with fewer current resources)
  • Banker's avoidance side effect — a process needing many resources may never find a safe state to receive them while others keep requesting single resources
Starvation vs. Deadlock
  • Starved process could run if others release resources/CPU — just never gets its turn
  • Other processes are making progress; system as a whole is not stuck
  • Starvation is a fairness/liveness problem; deadlock is a safety problem
Solutions
  • FCFS (First-Come, First-Served) — strictly fair ordering; earliest request always served first; prevents starvation by definition
  • Priority aging — gradually increase waiting process priority over time; eventually it preempts even high-priority work
  • Fair-share scheduling — guarantee each process/user a minimum CPU percentage
  • Lottery scheduling — processes hold tickets; random draw for CPU; all have nonzero chance
💡 Priority Aging Example
Linux's Completely Fair Scheduler (CFS) uses a virtual runtime that grows slower for high-priority processes — effectively aging lower-priority processes into competitiveness over time.

Numbered Resources Approach

Assign a global monotonically increasing integer to every resource type in the system. All processes must obey a strict acquisition ordering rule.

Rule: Process may only request resource Rj if Rj's assigned number is strictly GREATER than the number of ANY resource it currently holds.
Example Numbering Scheme
Resource TypeAssigned #Must acquire before...
Magnetic tape drive1Disk, printer, plotter
Disk / SSD drive2Printer, plotter
Plotter3Printer only
Line printer4(nothing; highest number)
mutex_db10mutex_cache (#20)
mutex_cache20mutex_log (#30)
VALID: acquire tape(1) → disk(2) → printer(4) INVALID: hold printer(4) → request disk(2) → must release printer first, then: → acquire disk(2) → re-acquire printer(4)
💡 Why No Circular Wait is Possible
In any cycle P1→R1→P2→R2→P1, process P1 holds R(high) while waiting for R(low) — but the rule requires always requesting R(low) before R(high). Contradiction → cycle impossible → no deadlock.
⚠ Numbering Challenge
Choosing a good ordering that matches natural usage patterns is difficult in large systems with hundreds of resource types (database locks on arbitrary table rows, for instance).

Semaphore Coding Discipline

The most practical prevention technique: always acquire semaphores/mutexes in the same global order across all threads in the program. Enforce via code review and static analysis.

✅ Correct — Consistent Global Order
/* Thread 1 and Thread 2: ALWAYS same order */
sem_wait(&mutex_A);  /* lower-numbered first  */
sem_wait(&mutex_B);  /* higher-numbered second */
/* --- critical section --- */
sem_post(&mutex_B);  /* release in reverse    */
sem_post(&mutex_A);
❌ Incorrect — Inconsistent Order Causes Deadlock
/* Thread 1: acquires A then B */
sem_wait(&mutex_A);
sem_wait(&mutex_B);  /* blocks if T2 holds B */

/* Thread 2: acquires B then A — WRONG ORDER! */
sem_wait(&mutex_B);
sem_wait(&mutex_A);  /* blocks if T1 holds A */
/* → Both blocked → Circular wait → DEADLOCK! */
Try-Lock with Timeout & Backoff
struct timespec ts = {0, 100000000L}; /* 100ms */
while (true) {
    if (sem_timedwait(&mutex_A, &ts) == 0) {
        if (sem_timedwait(&mutex_B, &ts) == 0) {
            /* both acquired — do work */
            sem_post(&mutex_B);
            sem_post(&mutex_A);
            break;
        }
        sem_post(&mutex_A); /* release A, retry */
    }
    /* exponential backoff before retry */
    nanosleep(&ts, NULL);
    ts.tv_nsec = (long)(ts.tv_nsec * 1.5);
}
Enforce global lock ordering via documentation, code review, and tools: Helgrind (Valgrind), ThreadSanitizer (-fsanitize=thread), or custom lock-order checkers.

Deadlock-Free Design Checklist

A practical engineering checklist for writing deadlock-free concurrent systems:

✓ 1. Assign a unique ordering number to every lock/mutex/semaphore/resource in the system. Write it down in a document.
✓ 2. Always acquire locks in strictly increasing order of their assigned numbers. Never acquire a lower-numbered lock while holding a higher-numbered one.
✓ 3. Use pthread_mutex_trylock() or sem_timedwait() with timeouts instead of blocking lock() wherever possible.
✓ 4. On trylock failure: release ALL currently held locks, apply random exponential backoff, then retry from scratch.
✓ 5. Minimize lock scope — compute everything possible BEFORE acquiring locks. Hold locks for the shortest possible duration.
✓ 6. Avoid nested lock acquisition unless the ordering is documented and enforced. Flat locking (one lock at a time) is safest.
✓ 7. Never call external code (callbacks, plugins, virtual functions) while holding a lock — you lose control of what locks that code may acquire.
✓ 8. Use lock-free data structures (std::atomic<T>, compare-and-swap) where performance requirements allow — eliminates mutex deadlock risk entirely.
✓ 9. Use RAII wrappers: std::lock_guard, std::unique_lock, std::scoped_lock — ensures locks are always released even on exceptions or early returns.
✓ 10. Verify with runtime tools: Helgrind, ThreadSanitizer (-fsanitize=thread in GCC/Clang), or formal model checkers (SPIN, TLA+).
💡 C++17 std::scoped_lock
std::scoped_lock lk(mutex_A, mutex_B); acquires multiple mutexes simultaneously using an internal deadlock-avoidance algorithm. It handles ordering for you when you must lock multiple mutexes at once — the safest approach in modern C++.
ℹ POSIX pthread_mutex_lock Ordering
POSIX provides PTHREAD_MUTEX_ERRORCHECK type that returns EDEADLK if a thread tries to re-lock its own mutex — catches one class of self-deadlocks automatically during development.
