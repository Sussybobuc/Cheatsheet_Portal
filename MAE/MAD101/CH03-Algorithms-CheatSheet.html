⟨⟩ Algorithm Basics

Algorithm: Finite sequence of precise steps to solve problem

Properties

  • Input: values provided to algorithm
  • Output: solution(s) produced
  • Definiteness: steps unambiguous
  • Finiteness: always terminates
  • Effectiveness: can be done in finite time

Specification

Pseudocode: high-level, language-independent description

Implementation: concrete code in specific language

O(n) Complexity Analysis

Big-O Notation

f(n) = O(g(n)) if ∃ constants c, k: |f(n)| ≤ c|g(n)| for n > k

Describes upper bound on growth rate

Common Complexities (fastest → slowest)

  • O(1) – Constant
  • O(log n) – Logarithmic
  • O(n) – Linear
  • O(n log n) – Linearithmic
  • O(n²) – Quadratic
  • O(n³) – Cubic
  • O(2ⁿ) – Exponential
  • O(n!) – Factorial

Ω(n) Θ(n)

Big-Omega Ω(n)

f(n) = Ω(g(n)): lower bound

∃ c, k: f(n) ≥ c·g(n) for n > k

Big-Theta Θ(n)

f(n) = Θ(g(n)): tight bound (both upper & lower)

f(n) = O(g(n)) AND f(n) = Ω(g(n))

Growth rates same (same order)

O describes "at most"
Ω describes "at least"
Θ describes "exactly (asymptotically)"

🔄 Algorithm Paradigms

Brute Force

Try all possible solutions; check each for validity

Simple, often inefficient; exhaustive search

Divide and Conquer

Divide problem into subproblems, solve recursively, combine

Examples: Merge sort, Quick sort, Binary search

Greedy Algorithm

Make locally optimal choice at each step

Hope: leads to global optimum; not always correct

Dynamic Programming

Solve overlapping subproblems, store results (memoization)

Trade memory for speed; optimal substructure required

Backtracking

Explore search tree; abandon paths that violate constraints

Depth-first search with pruning

⎕ Searching

Linear Search

Check each element sequentially | O(n)

Binary Search

Requires sorted array; divide search space in half | O(log n)

Compare middle element; recurse left/right half

Pseudocode: Binary Search

function binarySearch(a, x, left, right)
  if left > right return -1
  mid := ⌊(left+right)/2⌋
  if x = a[mid] return mid
  else if x < a[mid] 
    return binarySearch(a, x, left, mid-1)
  else return binarySearch(a, x, mid+1, right)

🔀 Bubble Sort

Idea

Repeatedly swap adjacent elements if in wrong order

Largest elements "bubble" to end each pass

procedure bubbleSort(a: array)
  n := length(a)
  for i := 1 to n-1
    for j := 1 to n-i
      if a[j] > a[j+1]
        swap a[j] and a[j+1]

Complexity: O(n²)

Simple, stable, in-place

Very slow for large arrays

⤓ Merge Sort

Idea

Divide array in half, recursively sort, merge sorted halves

Stable, optimal O(n log n) guaranteed

Merge Procedure

Two sorted arrays → one sorted array in O(n)

Compare first elements, add smaller to result

Complexity: O(n log n)

Stable, predictable

Requires O(n) extra space (not in-place)

Tip
Excellent for linked lists; no random access needed

↥ Quick Sort

Idea

Choose pivot, partition into smaller/larger, recursively sort

In-place, cache-friendly, widely used

Partition

Rearrange so all < pivot on left, all > pivot on right

Pivot position may vary from initial choice

Complexity

Average: O(n log n) | Worst: O(n²) (bad pivot choice)

Space: O(log n) recursion stack

Warning
Not stable; pivot selection critical for performance

📊 Algorithm Comparison

When n = 1,000,000

O(n²): ~10¹² operations (1 sec ≈ 10⁸ ops) → ~3 hours

O(n log n): ~2×10⁷ operations → 0.2 seconds

O(n): ~10⁶ operations → 0.01 seconds

Rule of Thumb

  • n < 50: Any algorithm fine
  • n = 100-1000: O(n²) acceptable
  • n = 1000-1,000,000: Need O(n log n) or better
  • n > 1,000,000: O(n) required; constants matter
Exponential (2ⁿ) is NEVER acceptable for n > 30

⟲ Recurrence Relations

Definition

Relation between function terms: T(n) = expression + T(smaller)

Example: Merge Sort

T(n) = 2·T(n/2) + O(n)

T(1) = O(1) (base case)

Master Theorem

For T(n) = a·T(n/b) + f(n):

  • If f(n) = O(n^(log_b(a)-ε)): T(n) = Θ(n^(log_b(a)))
  • If f(n) = Θ(n^(log_b(a))): T(n) = Θ(n^(log_b(a))·log n)
  • If f(n) = Ω(n^(log_b(a)+ε)): T(n) = Θ(f(n))

💾 Space Complexity

Definition

Maximum memory needed as function of input size

Often traded for time: more space → faster

In-Place Algorithm

Uses O(1) or O(log n) extra space (not counting input)

Examples: Bubble, Quick sort (if pivot chosen well)

Auxiliary Space

Extra space beyond input; includes recursion stack

Space-Time Tradeoff

Hash tables: O(n) space for O(1) lookup (vs O(n²) time)

Memoization: store results to avoid recomputation

🎯 Optimization Tips

  • Profile first: measure actual bottlenecks
  • Algorithm choice: 10x better than code optimization
  • Data structures: right structure for operations
  • Cache locality: sequential memory access
  • Precomputation: trade space for time
  • Parallelization: multi-core speedup possible
  • Avoid redundant work: detect/skip identical operations
