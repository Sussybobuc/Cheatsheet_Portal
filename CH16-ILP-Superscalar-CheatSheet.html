Superscalar — Definition & Overview

A superscalar processor (term coined 1987 by Agerwala & Cocke) executes multiple scalar instructions simultaneously in independent parallel pipelines. The essential capability: issue & execute ≥2 instructions per clock cycle to separate functional units.

Degree-N superscalar: can issue/execute up to N instructions per clock cycle

Steady-state comparison
In steady state, degree-2 superpipelined and degree-2 superscalar have the same number of instructions executing simultaneously. Difference emerges at branches and start of program.

ILP & Machine Parallelism

Instruction-Level Parallelism (ILP): degree to which instructions in a sequence are independent and can execute in parallel. A program property.

Machine Parallelism: the processor's ability to find and exploit ILP. A hardware property.

ILP Example (Jouppi & Wall 1989)
HIGH ILP — all 3 independent: LOW ILP — chain of deps: Load R1 <-- R2 Add R3 <-- R3 + "1" Add R3 <-- R3 + "1" Add R4 <-- R3 + R2 <- uses R3 Add R4 <-- R4 + R2 Store [R4] <-- R0 <- uses R4 ↑ Execute all 3 in parallel ↑ Must be strictly sequential
Key Factors Determining ILP
  • True data dependency frequency — how often does instr N need result of instr N-k?
  • Procedural dependency frequency — branch density (1 branch per ~5–10 instrs)
  • Operation latency — longer latency = more dependent instructions stall
  • ISA design — fixed-length (RISC) aids parallel fetch; variable-length (x86) hinders

Data, Control & Resource Hazards — Details

RAW — True Data Dependency (cannot be eliminated)
ADD EAX, ECX ; EAX <-- EAX + ECX (writes EAX) MOV EBX, EAX ; EBX <-- EAX (reads EAX — must wait!) Superscalar degree-2 timing: Cycle: 1 2 3 4 [ADD] [ADD] [ADD] [ADD] ← I1 (2-cycle execute) [MOV.IF][MOV.DE][---stall][MOV.EX] ^issued with I1 ^cannot exec until I1 done
WAR — Antidependency (eliminable)
I2: R4 <-- R3 + 1 (reads R3) I3: R3 <-- R5 + 1 (writes R3 — if I3 finishes before I2 reads, WRONG!) Fix: rename R3 in I3 to R3' (new physical register)
WAW — Output Dependency (eliminable)
I1: R3 <-- R3 op R5 (writes R3 first) I3: R3 <-- R5 + 1 (writes R3 later — if I3 finishes BEFORE I1, WRONG!) I4 then reads R3 — needs I3's value, not I1's Fix: rename R3 in I3 to R3' so I1 and I3 don't conflict
Resource Conflict (structural hazard)
I0 and I1 both need the integer ALU at the same cycle: Cycle: 1 2 3 [I0.EX] ← occupies ALU [I1.EX] ← must wait 1 cycle (stall) Fix: duplicate the ALU (add second integer FU)
Procedural / Control Dependency

Branch instruction makes successor instructions' execution uncertain. Variable-length instructions (x86) add another procedural dep: next instruction boundary unknown until current decoded.

Loop-Level Parallelism

ILP can be exploited across loop iterations when iterations are independent (no loop-carried dependencies). Primary compiler technique: loop unrolling.

Loop Unrolling Example
Original (1 iteration/cycle, 1 branch per iteration): for(i = 0; i < N; i++) A[i] = B[i] + C[i]; After unrolling ×4 (assuming N divisible by 4): A[0] = B[0] + C[0]; // independent A[1] = B[1] + C[1]; // independent A[2] = B[2] + C[2]; // independent A[3] = B[3] + C[3]; // independent // One branch every 4 iterations // 4 add ops can be issued in parallel on superscalar Benefits: + Reduces branch overhead by 4× + Exposes 4 independent operations for parallel issue + Enables better register allocation - Increases code size - Only works when iterations are independent
Loop-Carried Dependence (prevents unrolling)
Recurrence — Cannot parallelize
S[i] = S[i-1] + A[i] — iteration i+1 depends on result of iteration i. No unrolling benefit.
Other Compiler ILP Techniques
  • Software pipelining — interleave instructions from different iterations
  • Tree-height reduction — rebalance expression trees to shorten critical path
  • Trace scheduling — optimize most-likely execution trace globally
  • Instruction scheduling — move independent instructions near each other
  • Induction variable elimination — simplify loop counter computations

Instruction Window (for OOO Issue)

Buffer between decode and execute. Processor fills it as fast as instructions are decoded, then issues any instruction from the window whose operands are ready and whose required FU is free.

Commit constraint
Even with OOO execution, results must be committed (retired) to the architectural register file in program order to maintain precise exceptions and correct program behavior.

Register Renaming

Dynamically maps logical (architectural) registers to a larger pool of physical registers, eliminating WAR and WAW (artificial) dependencies while preserving RAW (true) dependencies.

Code Before Renaming (all deps present)
I1: R3 <-- R3 op R5 ← writes R3 (original) I2: R4 <-- R3 + 1 ← reads R3; RAW dep on I1 I3: R3 <-- R5 + 1 ← writes R3; WAW w/I1, WAR w/I2 I4: R7 <-- R3 op R4 ← must read I3's R3; RAW dep on I3 Problems: I3 cannot issue until I1 completes (WAW) AND I3 cannot complete before I2 reads R3 (WAR)
After Register Renaming (artificial deps removed)
I1: R3b <-- R3a op R5a ← allocate new physical reg R3b I2: R4b <-- R3b + 1 ← reads R3b (RAW on I1, correct) I3: R3c <-- R5a + 1 ← allocate new physical reg R3c (NO WAW, NO WAR!) I4: R7b <-- R3c op R4b ← reads R3c (RAW on I3, correct) Result: I3 can now issue immediately — independent of I1 and I2!
Key Rule
When an instruction writes a register, allocate a new physical register. Rename all subsequent reads of that logical register to use the newly allocated physical register.

Scoreboarding (Alternative to Renaming)

Bookkeeping technique (no physical renaming). Tracks which FU will produce each register. Instructions execute when operands available and no structural hazards. Used in ARM Cortex-A8 decode stage (D3).

Tomasulo's Algorithm & Reorder Buffer

Hardware algorithm (IBM 360/91, 1967) for dynamic OOO scheduling. Uses reservation stations (RS) as distributed instruction buffers ahead of each FU.

Core Data Structures
  • Reservation Stations (RS) — one set per FU; each slot holds: opcode, operand values (if ready), or RS-tag of producing instruction (if not ready)
  • Common Data Bus (CDB) — broadcast bus; every completing instruction broadcasts (result, RS-tag); all RS and register file listen and capture matching tags
  • Register Result Status Table — maps each register to the RS that will produce its next value (or "available" if ready)
Three Phases of Tomasulo
  1. Issue (Dispatch): Dequeue instruction from FIFO queue. If RS slot available: copy instruction + read available operands from reg file; replace unavailable operand values with RS tags (this IS implicit register renaming). Stall issue if no RS free (structural hazard).
  2. Execute: When ALL operands available (values, not tags), send to FU for execution. For loads/stores: compute address first, then access memory. Monitor CDB each cycle — if broadcast tag matches a pending operand, capture the value.
  3. Write Result (Broadcast): Completed FU writes (result, own RS-tag) onto CDB. All RS slots with matching tag capture the value. Register file entry updated if no newer instruction targets same register. RS slot freed.
Reorder Buffer (ROB) — adds precise exceptions

Circular FIFO buffer holding in-flight instructions in program order. Results written to ROB first; committed to architectural state only when instruction reaches head of ROB and has completed. Enables both OOO execution and in-order commit.

ROB Entry fields (Intel Core: 126-entry ROB): [State: Sched|Dispatched|Ready-Retire] [Memory Address: PC of x86 instruction] [Micro-op: the actual operation] [Alias Register: arch-reg → physical-reg mapping] Flow: Enter ROB in-order → Execute OOO → Commit in-order

Dynamic Branch Prediction — 1-bit & 2-bit

Dynamic prediction uses runtime history of each branch. History stored in a Branch History Table (BHT) / Pattern History Table, indexed by low-order PC bits.

1-bit Predictor
State machine: T ←──────────── T ───────→ NT (Taken) (Not Taken) If in state T: predict Taken; if wrong → flip to NT If in state NT: predict NT; if wrong → flip to T Problem with loops (e.g. taken 9 times, then not taken): Last iteration: state=T → predict T → WRONG (flip to NT) First next time: state=NT → predict NT → WRONG (flip to T) → 2 mispredictions per loop execution, every time!
2-bit Saturating Counter Predictor
State Machine (4 states): Taken outcome Taken outcome ┌──────────────── ST ←──── WT ──────────────────┐ │ (must be wrong TWICE to change prediction) │ │ │ └──→ SNT ──→ WNT ──→ WT ──→ ST ─────────────────┘ NT outcome NT outcome ST = Strongly Taken → predict Taken WT = Weakly Taken → predict Taken WNT= Weakly NT → predict Not Taken SNT= Strongly NT → predict Not Taken Loop of 9 iterations (taken 9×, not taken 1×): → State eventually reaches ST → After last NT: ST→WT (still predicts Taken on next loop entry) ✓ → Only 1 misprediction per loop execution!

BTB & Two-Level Adaptive Branch Prediction

Branch Target Buffer (BTB)

A cache associating branch instruction addresses to their predicted targets and history. Provides both the prediction direction and the target address in the fetch stage, before decode.

Each BTB entry contains: [Branch PC] → [Target Address | History Bits] Every fetch cycle: PC → lookup BTB ├─ HIT: use history bits → predict T or NT │ provide target addr for prefetch └─ MISS: use static prediction rules allocate new BTB entry on first encounter (evict oldest entry if full) After execution: Update history field of BTB entry with actual outcome
Two-Level Adaptive (Global History) Predictor

Combines a Branch History Register (BHR) — a k-bit shift register of last k branch outcomes — with a Pattern History Table (PHT) of 2-bit counters.

Level 1 — Branch History Register (BHR): k-bit shift register: [T,NT,T,T,NT,...] (T=1, NT=0) Captures the pattern of the last k branches globally Level 2 — Pattern History Table (PHT): 2^k entries × 2-bit saturating counters Indexed by BHR (captures branch correlations!) Prediction: PHT[BHR] → 2-bit counter → predict T or NT After execution: update PHT[BHR]; shift BHR left, insert outcome Variant — gshare: index = BHR XOR (lower bits of PC) → reduces aliasing between unrelated branches
ARM Cortex-A8 Two-Level Predictor

BTB512 entriesBranch PC

Why two-level?
Captures correlated branch behavior: e.g., if branch A is taken, branch B (later) is usually taken too. A single 2-bit counter per branch misses this pattern.

Speculative Execution

Execute instructions before it is certain they will be needed (along the predicted path of a branch). Critical for keeping deep pipelines full.

Why It Is Necessary
  • Branches occur every ~5–10 instructions in typical programs
  • Deep pipelines: Intel Core = 14 stages → branch resolves late
  • Without speculation: pipeline stalls 10+ cycles per branch = huge CPI penalty
  • Superscalar: penalty multiplied by degree (wastes N slots per stall cycle)
Speculative Execution Flow
Branch encountered at stage k of pipeline: ↓ Predict: Taken → fetch from target addr T Not Taken → fetch from PC+4 ↓ Execute instructions on predicted path (speculatively) Results held in ROB — NOT committed to arch. registers ↓ Branch resolves at execute stage: ┌──────────────────────────────────────┐ │ CORRECT prediction │ │ → Commit ROB entries in order │ │ → Architectural state updated │ └──────────────────────────────────────┘ ┌──────────────────────────────────────┐ │ WRONG prediction (misprediction) │ │ → Flush all speculative instructions│ │ from pipeline and ROB │ │ → Restore register state (rename │ │ table rolled back to checkpoint) │ │ → Fetch from correct path │ │ → Penalty: 14 cycles (Intel Core) │ │ 8+ cycles (Cortex-A8) │ └──────────────────────────────────────┘
Hardware Requirements for Speculative Execution
  • ROB — holds results speculatively; enables squash
  • Rename table checkpoint — saved at each branch for rollback
  • Pipeline flush — cancels all ROB entries after mispredicted branch
  • In-order retirement — speculative results only committed when safe

Superscalar Implementation — 5 Key Elements

Smith (1995) identifies the essential hardware components for superscalar implementation:

  1. Multi-instruction Fetch Strategies — simultaneously fetch ≥2 instructions per cycle; predict & fetch beyond branches; multiple pipeline fetch/decode stages; branch prediction unit
  2. True Dependency Logic — identify all RAW dependencies; communicate produced values to consuming instructions (forwarding/bypassing network across all FUs)
  3. Multi-Issue Mechanisms — hardware to initiate multiple instructions in parallel per cycle; instruction window/RS; issue logic to select ready instructions
  4. Parallel Execution Resources — multiple pipelined functional units (int ALU, FP, load, store, branch); multi-ported register file; memory hierarchy supporting simultaneous multiple references (separate I/D caches)
  5. In-Order Commit Mechanisms — reorder buffer; retire oldest completed instruction first; handle interrupts precisely; squash speculative instructions on misprediction

ARM Cortex-A8 — 13-Stage Dual In-Order Pipeline

Superscalar RISC design — dual in-order issue, 13-stage integer pipeline + separate 10-stage NEON/SIMD pipeline. In-order chosen for power efficiency (mobile/embedded target).

Complete Pipeline Stages
FETCH PIPELINE (3 stages: F0, F1, F2): F0: AGU generates virtual address (normally PC+N or BTB target) F1: Fetch from L1 I-cache; access BTB + GHB in parallel F2: Place up to 12 instr into fetch queue; issue 2/cycle to decode DECODE PIPELINE (5 stages: D0–D4): D0: Thumb→ARM32 decompress; preliminary decode D1: Full instruction decode complete D2: Write to / read from pending+replay queue D3: Scheduling logic; scoreboard checks register availability (static) Hazard detection (RAW, load-use, structural) D4: Final decode → control signals for execute units EXECUTE PIPELINE (6 stages: E0–E5): E0: Register file read (up to 6 registers for 2 instructions) E1: Barrel shifter; load/store: address generation from base+offset E2: ALU operation; load/store: cache array access E3: Saturation arithmetic; load: data return and format for forwarding E4: Branch misprediction / exception / replay processing E5: Writeback results to architectural register file
Fetch Rate & Issue

Fetch up to 4 instrs/cycle; queue up to 12; issue 2 per cycle (dual-issue). pipe0 = older instruction, pipe1 = newer. If pipe0 stalls → pipe1 stalls too (in-order guarantee).

ARM Cortex-M3 — 3-Stage Scalar Pipeline

Microcontroller-class ARM processor. 3-stage in-order scalar pipeline: Fetch → Decode → Execute. No superscalar, no OOO, no branch prediction. Designed for determinism, low power, and small silicon area.

Pipeline Stages
Stage 1: FETCH — Fetch 16/32-bit Thumb-2 instruction from I-cache/flash Stage 2: DECODE — Decode instruction; read register file Stage 3: EXECUTE — ALU operation + memory access + flag update + writeback Normal timing (no stalls): Cycle: 1 2 3 4 5 I1: [F] [D] [E] I2: [F] [D] [E] I3: [F] [D] [E] → 1 instruction/cycle in steady state (scalar)
Branch Handling
Branch TypePenaltyNotes
Unconditional (B)1–3 cyclesTarget computed in decode
Conditional (BEQ etc.)1–3 cyclesCondition evaluated in execute
Indirect (BX, BLX)3 cyclesTarget not known until execute
IT block (if-then)0 extraPredicated instructions; no branch
Key Features
  • Thumb-2 ISA — mix of 16-bit and 32-bit instructions (code density)
  • Hardware divider — SDIV/UDIV: 2–12 cycles
  • NVIC — Nested Vectored Interrupt Controller (12-cycle interrupt latency)
  • No branch predictor — deterministic timing for RTOS & safety-critical
  • Optional DSP extensions — multiply-accumulate (MAC) operations
Design philosophy
Trades throughput for determinism and minimal gate count. A Cortex-A8 has ~7× the transistors of a Cortex-M3 but targets 100–1000× higher performance workloads.
