Why Generic Programming

Generics = type-safe, reusable code. Write once, work with any type. Errors caught at compile time.
Without generics (pre-Java 5)
// Old way — Object-based (unsafe)
ArrayList list = new ArrayList();
list.add("Hello");
list.add(42);           // accidentally mixed types!
String s = (String) list.get(0);  // cast needed
String s2 = (String) list.get(1); // ClassCastException at runtime!
With generics
// Type-safe
ArrayList<String> list = new ArrayList<>();
list.add("Hello");
// list.add(42);  // COMPILE ERROR — caught early!
String s = list.get(0);  // no cast needed
Tip
Generics shift type errors from runtime to compile time — far better!

Generic Classes

// Generic class definition
public class Pair<T, U> {
    private T first;
    private U second;

    public Pair(T first, U second) {
        this.first = first;
        this.second = second;
    }

    public T getFirst() { return first; }
    public U getSecond() { return second; }

    @Override
    public String toString() {
        return "(" + first + ", " + second + ")";
    }
}

// Usage
Pair<String, Integer> p = new Pair<>("Alice", 30);
String name = p.getFirst();  // no cast!
int age = p.getSecond();

// Common type parameter names (convention):
// T = Type, E = Element, K = Key, V = Value, R = Result
// N = Number, S,U,V = extra types

Generic Methods

public class ArrayUtil {
    // Generic method — type param before return type
    public static <T> T getMiddle(T[] array) {
        return array[array.length / 2];
    }

    // Multiple type bounds
    public static <T extends Comparable<T>> T min(T[] array) {
        T min = array[0];
        for (T item : array) {
            if (item.compareTo(min) < 0) min = item;
        }
        return min;
    }

    // Generic method returning generic type
    public static <T> List<T> listOf(T... elements) {
        return Arrays.asList(elements);
    }
}

// Usage
String mid = ArrayUtil.getMiddle(new String[]{"a","b","c"});
int minimum = ArrayUtil.min(new Integer[]{3,1,4,1,5});

Bounded Type Parameters

Upper bound (extends)
// T must be Comparable or subclass
public <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) >= 0 ? a : b;
}

// T must implement both interfaces
public <T extends Serializable & Comparable<T>> void sort(T[] arr) {
    Arrays.sort(arr);
}

// Multiple bounds — class must come first, interfaces after
// <T extends ClassName & Interface1 & Interface2>

// Restricting to Number and subclasses
public <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue();
    return total;
}
Warning
In Java, generic type erasure means <T extends Foo> is upper-bounded — no lower bound for type parameters (use wildcards for that)

Wildcards

Wildcards = ? = unknown type. More flexible than concrete type params.
Unbounded wildcard ?
// Accept any List regardless of element type
public void printList(List<?> list) {
    for (Object elem : list) System.out.println(elem);
}
printList(new ArrayList<String>());
printList(new ArrayList<Integer>());
Upper bounded ? extends T (read-only / producer)
// Can read Numbers (Integer, Double, etc.)
public double sumList(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) sum += n.doubleValue();
    return sum;  // can read but NOT add items
}
sumList(new ArrayList<Integer>());  // works!
sumList(new ArrayList<Double>());   // works!
Lower bounded ? super T (write-only / consumer)
// Can add Integers (or supertypes accept them)
public void addIntegers(List<? super Integer> list) {
    list.add(1); list.add(2);  // can add
}
addIntegers(new ArrayList<Integer>());  // works
addIntegers(new ArrayList<Number>());   // works
addIntegers(new ArrayList<Object>());   // works
Tip
PECS — Producer Extends, Consumer Super: if list PRODUCES items you read: use extends. If list CONSUMES items you add: use super.

Type Erasure

At runtime, all generic types are ERASED. List<String> becomes List. The compiler adds casts.
// What you write:
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// What the compiler generates (approximately):
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);  // cast inserted
Consequences of erasure
  • Cannot do new T() — T is erased at runtime
  • Cannot do instanceof List<String> — only instanceof List
  • Cannot create generic arrays: new T[10] is illegal
  • Static fields cannot be type parameters
  • Overloading: void foo(List<String>) and void foo(List<Integer>) — COMPILE ERROR, both erase to same

Generics in Practice

Generic Interface Implementation
public interface Comparable<T> {
    int compareTo(T other);
}

public class Student implements Comparable<Student> {
    private double gpa;

    @Override
    public int compareTo(Student other) {
        return Double.compare(this.gpa, other.gpa);
    }
}
Generic Stack example
public class Stack<E> {
    private List<E> elements = new ArrayList<>();

    public void push(E item) { elements.add(item); }

    public E pop() {
        if (isEmpty()) throw new EmptyStackException();
        return elements.remove(elements.size() - 1);
    }

    public E peek() {
        if (isEmpty()) throw new EmptyStackException();
        return elements.get(elements.size() - 1);
    }

    public boolean isEmpty() { return elements.isEmpty(); }
    public int size() { return elements.size(); }
}

Stack<String> stack = new Stack<>();
stack.push("Hello");
String top = stack.pop(); // "Hello"

Raw Types (avoid!)
// Raw type — avoid in new code
List rawList = new ArrayList();  // raw
rawList.add("anything");
// No compile warning suppression — get unchecked warnings

// Always prefer parameterized:
List<String> typedList = new ArrayList<>();
Warning
Raw types exist for backward compatibility only. Always use parameterized types in new code.
Diamond Operator
// Java 7+: diamond infers type from left side
Map<String, List<Integer>> map = new HashMap<>();
// Instead of: new HashMap<String, List<Integer>>()
