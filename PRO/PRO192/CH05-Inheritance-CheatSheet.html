Inheritance Basics

Subclass extends Superclass → inherits all non-private fields/methods
// Superclass
public class Animal {
    private String name;
    protected int age;  // protected = subclass can access

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() { return name; }

    public String describe() {
        return name + " (age " + age + ")";
    }
}

// Subclass
public class Dog extends Animal {
    private String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);  // MUST call super constructor first
        this.breed = breed;
    }

    @Override
    public String describe() {
        return super.describe() + " [" + breed + "]";
    }

    public void bark() { System.out.println("Woof!"); }
}
  • extends = inheritance keyword, Java supports single inheritance only
  • super = reference to superclass
  • Subclass constructor MUST call super(...) as first statement
  • If omitted → implicit super() called (requires no-arg constructor in parent)

Method Overriding

Rules for Override
  • Same method name, same parameter types
  • Same or covariant return type
  • Access modifier same or wider (can't reduce visibility)
  • Must NOT be final or static in parent
public class Shape {
    public double area() { return 0; }
    public String toString() { return "Shape"; }
}

public class Circle extends Shape {
    private double radius;

    public Circle(double radius) { this.radius = radius; }

    @Override  // annotation catches mistakes at compile time
    public double area() {
        return Math.PI * radius * radius;
    }
}
Override vs Overload

Same method sig✓✗ (different params)

Warning
@Override annotation is optional but highly recommended — catches typos and wrong signatures at compile time

super Keyword

Uses of super
public class Manager extends Employee {
    private double bonus;

    public Manager(String name, double salary, double bonus) {
        super(name, salary);  // 1. Call parent constructor (must be 1st)
        this.bonus = bonus;
    }

    @Override
    public double getSalary() {
        return super.getSalary() + bonus;  // 2. Call parent method
    }

    // Access parent field (if protected):
    // super.fieldName
}
super(args) → calls parent constructor (only in constructor, must be 1st line)
super.method() → calls parent's version of overridden method
Warning
Cannot use super.super.method() — Java only allows one level up

Abstract Classes

Abstract class = cannot be instantiated; may have abstract methods (no body)
public abstract class Shape {
    private String color;

    public Shape(String color) { this.color = color; }

    // Abstract method - subclass MUST implement
    public abstract double area();
    public abstract double perimeter();

    // Concrete method - subclass can use/override
    public String getColor() { return color; }

    public void printInfo() {
        System.out.printf("%s: area=%.2f%n", color, area());
    }
}

public class Rectangle extends Shape {
    private double width, height;

    public Rectangle(String color, double w, double h) {
        super(color);
        this.width = w; this.height = h;
    }

    @Override public double area() { return width * height; }
    @Override public double perimeter() { return 2 * (width + height); }
}

Polymorphism

Polymorphism = same interface, different implementations; method called depends on ACTUAL runtime type
Shape[] shapes = {
    new Circle("red", 5),
    new Rectangle("blue", 3, 4),
    new Triangle("green", 3, 4, 5)
};

for (Shape s : shapes) {
    System.out.println(s.area());  // Calls correct subclass area()!
    // At runtime: Circle.area(), Rectangle.area(), Triangle.area()
}
Dynamic Dispatch
  • JVM looks up actual object type at runtime
  • Calls the most specific method available
  • This is the core of polymorphism in Java
  • Makes code extensible without modifying existing code (Open/Closed Principle)
Tip
Key insight: You can add new Shape subclasses without changing the loop above — that's the power of polymorphism

Type Casting & instanceof

Upcasting (always safe)
Dog dog = new Dog("Rex", 3, "Husky");
Animal animal = dog;  // implicit upcast (always safe)
// animal.bark();  // ERROR: Animal doesn't know bark()
Downcasting (can fail — use instanceof check)
Animal a = new Dog("Rex", 3, "Husky");

// Old way (pre-Java 16)
if (a instanceof Dog) {
    Dog d = (Dog) a;  // explicit downcast
    d.bark();
}

// Pattern matching instanceof (Java 16+) — PREFERRED
if (a instanceof Dog d) {
    d.bark();  // d is already cast and in scope
}

// ClassCastException if wrong:
Animal cat = new Cat("Whiskers");
Dog d = (Dog) cat;  // Throws ClassCastException at runtime!
Warning
Always check instanceof before downcasting, or use pattern matching instanceof (Java 16+)

Sealed Classes, Final & Inheritance Hierarchy

final Keyword (prevents extension/override)
public final class String { ... }
// Cannot extend String!

public class Base {
    public final void cannotOverride() { ... }
    // Subclass cannot @Override this method
}
Sealed Classes (Java 17+)
// Only permitted classes can extend Shape
public sealed class Shape
    permits Circle, Rectangle, Triangle { }

public final class Circle extends Shape { }
public final class Rectangle extends Shape { }
public non-sealed class Triangle extends Shape { }
// non-sealed allows further subclassing
Tip
Sealed classes + pattern matching = exhaustive type checking at compile time

Inheritance Design Principles

Liskov Substitution Principle (LSP)
If S extends P, then objects of type P can be replaced with objects of type S without breaking behavior
  • Subclass should NOT weaken preconditions (caller promises less)
  • Subclass should NOT strengthen postconditions (caller gets less)
  • Override methods should maintain same contract as parent
Warning
Violating LSP: Square extends Rectangle but setWidth() breaks rectangle invariant (w≠h)
Composition over Inheritance
// Prefer composition when "has-a" relationship
public class Car {
    private Engine engine;      // HAS-A engine
    private Transmission trans; // HAS-A transmission

    public Car() {
        engine = new Engine();
        trans = new Transmission();
    }
}
// vs. fragile: class Car extends Engine (wrong: car IS-NOT an engine)

Common Inheritance Patterns

Template Method Pattern
abstract class DataProcessor {
    // Template method (defines skeleton)
    public final void process() {
        readData();    // implemented here
        processData(); // abstract - subclass implements
        writeData();   // implemented here
    }
    protected abstract void processData();
    private void readData() { /* ... */ }
    private void writeData() { /* ... */ }
}

class CSVProcessor extends DataProcessor {
    @Override
    protected void processData() {
        // Specific CSV processing logic
    }
}
Tip
Template Method uses inheritance to share algorithm structure while allowing subclass to customize steps
