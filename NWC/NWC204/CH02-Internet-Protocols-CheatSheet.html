DoD (TCP/IP) Model vs OSI Model

DoD Layer DoD Name Corresponding OSI Layers Key Protocols PDU Function
4 Process / Application Application (7) + Presentation (6) + Session (5) Telnet, SSH, FTP, TFTP, SMTP, POP3, IMAP, HTTP, HTTPS, DNS, SNMP, NTP, DHCP Data User-facing services; data formatting, session management
3 Host-to-Host Transport (4) TCP, UDP Segment / Datagram End-to-end delivery; reliability (TCP), speed (UDP); port numbers; flow control
2 Internet Network (3) IP, ICMP, ARP, RARP, BootP, DHCP Packet Logical addressing; routing; path selection; best-effort connectionless delivery

Model Comparison Note
The DoD model was developed by the US Department of Defense and predates OSI. TCP/IP is based on the DoD model. OSI is a theoretical reference model; DoD/TCP-IP is the practical implementation. Both map to each other as shown above.

FTP — Active vs Passive Mode

Active Mode (PORT command)
  • Client connects from ephemeral port to server port 21 (control)
  • Client sends PORT command specifying its data IP:Port
  • Server initiates data connection from port 20 to client's specified port
  • Problem: firewalls/NAT often block inbound connections from server
Client:1025 ──────→ Server:21  (control)
Client:1026 ←────── Server:20  (data — SERVER initiates)
Passive Mode (PASV command)
  • Client connects from ephemeral port to server port 21 (control)
  • Client sends PASV command
  • Server replies with its data IP:Port (ephemeral >1023)
  • Client initiates the data connection to server's ephemeral port
  • Firewall-friendly — client always initiates both connections
Client:1025 ──────→ Server:21    (control)
Client:1026 ──────→ Server:5001  (data — CLIENT initiates)
Key Difference
Active: Server initiates data connection (port 20 → client). Problems with firewalls.
Passive: Client initiates data connection (client → server high port). Firewall-friendly.
DNS Resolution Process
  1. Client queries local DNS cache first
  2. If not found, query Recursive DNS server (ISP/configured)
  3. Recursive DNS queries Root DNS server (knows TLD servers)
  4. Root refers to TLD server (.com, .org, etc.)
  5. TLD refers to Authoritative DNS server for that domain
  6. Authoritative server returns IP address to client
DNS Record Types
A = hostname → IPv4  |  AAAA = hostname → IPv6
CNAME = alias  |  MX = mail server  |  PTR = reverse lookup

Connection Termination (4-Way)
Client ──── FIN ────→ Server  (initiates close)
Client ←─── ACK ──── Server  (half-closed)
Client ←─── FIN ──── Server  (server closes)
Client ──── ACK ────→ Server  (TIME_WAIT then CLOSED)
Windowing (Flow Control)
Sender Window = number of segments sent before ACK required Small window → slow, reliable Large window → fast, risk of overflow Dynamic: expands on success, shrinks on loss
Window=1:  [1]→ ACK → [2]→ ACK → [3]→ ACK  (slow)
Window=3:  [1,2,3]→ ACK → [4,5,6]→ ACK      (faster)

SYN — synchronize sequence numbers (connection start)
ACK — acknowledgement number is valid
FIN — no more data from sender (graceful close)
RST — reset/abort connection immediately
PSH — push data to application immediately

ICMP is used by IP for error reporting and diagnostic functions. Not used for data delivery. Encapsulated directly in IP (Protocol 1).

Ping — How It Works
Host A ──── ICMP Echo Request (Type 8) ────→ Host B
Host A ←─── ICMP Echo Reply   (Type 0) ───── Host B
Traceroute — How It Works
  1. Send packet with TTL=1 → first router decrements to 0, sends "TTL Exceeded" (Type 11)
  2. Send packet with TTL=2 → second router responds with TTL Exceeded
  3. Continue incrementing TTL until destination reached
  4. Destination sends Echo Reply (or ICMP Port Unreachable for UDP traceroute)
  5. Each hop's IP address and RTT are recorded
ICMP Is Not for Data
ICMP does not use TCP or UDP. It is a Layer 3 protocol. Firewalls often block ICMP, making ping/traceroute unreliable for troubleshooting through firewalls.

ARP, RARP, BootP & DHCP

ARP — Address Resolution Protocol

Resolves IP address → MAC address. Required before Layer 2 frame can be built.

  1. Host A wants to send to 192.168.1.10; checks ARP cache
  2. ARP Request: broadcast (FF:FF:FF:FF:FF:FF) "Who has 192.168.1.10?"
  3. Host B (192.168.1.10) receives broadcast, sends ARP Reply (unicast)
  4. ARP Reply: "192.168.1.10 is at MAC AA:BB:CC:DD:EE:FF"
  5. Host A caches the IP→MAC mapping; builds frame
ARP Request:  SRC=A's MAC, DST=FF:FF:FF:FF:FF:FF  [broadcast]
ARP Reply:    SRC=B's MAC, DST=A's MAC              [unicast]
ARP Cache
  • OS maintains ARP cache table (IP → MAC mappings)
  • Entries expire after timeout (typically 4 hours Windows, 20 min Cisco)
  • View with arp -a (Windows/Linux) or show arp (Cisco)
  • Gratuitous ARP: device ARPs its own IP (announce MAC change)
  • ARP spoofing: attacker responds to ARP requests with attacker's MAC (MITM)
RARP — Reverse ARP
  • Resolves MAC address → IP address (opposite of ARP)
  • Used by diskless workstations to discover their IP at boot
  • Obsolete — replaced by BootP then DHCP
  • Requires RARP server on same broadcast domain
DHCP DORA Process
  1. Discover: Client broadcasts DHCPDISCOVER (src=0.0.0.0, dst=255.255.255.255)
  2. Offer: DHCP server unicasts/broadcasts DHCPOFFER (IP, mask, gateway, DNS, lease)
  3. Request: Client broadcasts DHCPREQUEST (accepts offer, notifies other servers)
  4. Acknowledge: Server sends DHCPACK confirming lease; client configures interface
D-O-R-A: Discover → Offer → Request → Acknowledge
DHCP Lease Info Provided
  • IP address (from configured pool)
  • Subnet mask
  • Default gateway (router)
  • DNS server addresses
  • Lease duration (typically 8 days)
DHCP Relay Agent
DHCP uses broadcasts. Routers don't forward broadcasts. A DHCP relay agent (ip helper-address on Cisco router) forwards DHCP broadcast as unicast to DHCP server on another network. 
Router config for DHCP relay:
interface FastEthernet0/0
 ip helper-address 10.0.0.10

Class A Breakdown
Network.Host.Host.Host (N.H.H.H) First bit always 0 → first octet 0–127 (but 0 and 127 reserved) Valid Class A networks: 1.0.0.0 – 126.0.0.0 (126 networks) Hosts per network: 2^24 - 2 = 16,777,214
Class B Breakdown
Network.Network.Host.Host (N.N.H.H) First 2 bits always 10 → first octet 128–191 Valid Class B networks: 128.0.0.0 – 191.255.0.0 (16,382 networks) Hosts per network: 2^16 - 2 = 65,534
Class C Breakdown
Network.Network.Network.Host (N.N.N.H) First 3 bits always 110 → first octet 192–223 Valid Class C networks: 192.0.0.0 – 223.255.255.0 (2,097,152 networks) Hosts per network: 2^8 - 2 = 254

IP is Connectionless & Best-Effort
IP does NOT guarantee delivery, ordering, or error recovery. It just forwards packets based on destination IP. Reliability is TCP's job (Layer 4).
127.0.0.0 Reserved — Never Usable
The entire 127.0.0.0/8 range is reserved for loopback. That's why Class A only has 126 usable networks (1–126), not 128. Network 0 is also reserved.
NAT Overview
  • NAT = Network Address Translation
  • Translates private IPs to public IP(s) for internet access
  • Static NAT: 1 private ↔ 1 public (one-to-one)
  • Dynamic NAT: pool of public IPs, assigned from pool
  • PAT (NAT Overload): many private → 1 public (using port numbers)
